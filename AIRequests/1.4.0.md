# 実装仕様書

## Phase 1: 削除系 UI / API 整理、管理対象除外、ごみ箱移動化

## 1. 目的

AZViewer における画像削除系操作の意味を整理し、誤操作・誤解を減らす。

従来は UI 上の「削除」が、以下のどれを意味するか曖昧になりやすかった。

```text
- AZViewer の DB 管理対象から外す
- サムネイルキャッシュを削除する
- 実画像ファイルを即時削除する
- 実画像ファイルをごみ箱へ移動する
```

本 Phase では、削除系操作を次の 2 種類へ明確に分離する。

```text
管理対象から除外
- AZViewer の DB 管理対象から外す
- 実画像ファイルは残す

ごみ箱へ移動
- 実画像ファイルを OS ごみ箱へ移動する
- 成功した画像は AZViewer の DB 管理対象からも外す
```

また、通常 UI から即時完全削除を行う操作は提供しない。

---

## 2. 用語定義

## 2.1 管理対象から除外

AZViewer の管理対象から画像を外す操作。

### 実行すること

```text
- image_file_data の対象レコードを削除する
- tag_image_link の関連レコードを削除する
- image_model_to_file_data の関連レコードを削除する
- tag_hash の関連レコードを削除する
- 対象画像のサムネイルキャッシュを削除する
```

### 実行しないこと

```text
- 実画像ファイルの削除
- 実画像ファイルの移動
- tag マスタ削除
- image_model マスタ削除
```

未使用になったタグ・モデルは、既存のマスタメンテナンス機能で整理する。

## 2.2 ごみ箱へ移動

実画像ファイルを OS のごみ箱へ移動し、成功した画像を AZViewer の管理対象から外す操作。

### 実行すること

```text
- 実画像ファイルを OS ごみ箱へ移動する
- ごみ箱移動に成功した画像を DB 管理対象から除外する
- サムネイルキャッシュを削除する
```

### 実行しないこと

```text
- Path.unlink() による即時完全削除
- ごみ箱移動に失敗した画像の DB レコード削除
- tag / image_model マスタ削除
```

## 2.3 完全削除

実画像ファイルを OS ごみ箱に入れず、即時削除する操作。

本仕様では通常 UI に提供しない。

```text
Path.unlink() による実画像ファイルの即時削除は行わない。
既存実装がある場合は撤廃し、ごみ箱移動へ置き換える。
```

これは重要である。完全削除は便利に見えるが、ローカル画像管理ツールでは刃が鋭すぎる。包丁をキーボードの Enter キーに割り当てるようなものです。


### 7.2.1 removeSelectedImagesFromCatalog

```js
async function removeSelectedImagesFromCatalog({ ids, refresh, clearSelection }) {
  ...
```

#### 処理フロー

```text
1. ids が空なら何もしない
2. confirm を表示する
3. キャンセルなら何もしない
4. LoadingOverlay を表示する
```

```text
実行しますか？
```

```text

メッセージ: 選択画像の登録情報を AZViewer から除外しています...
```

成功。

```text

```

API 失敗。


## 10. Repository 仕様

## 10.1 ImageFileRepository

既存の `find_by_ids(record_ids)` を利用する。

管理対象除外用に以下を追加、または既存 `delete_by_ids` を利用する。

```python
def remove_from_catalog_by_ids(self, record_ids: list[int]) -> int:
    ...
```

### 処理内容

```text
- tag_image_link から対象 image_file_id の関連を削除
- image_model_to_file_data から対象 image_file_data_id の関連を削除
- tag_hash から対象 image_file_data_id の関連を削除
- image_file_data から対象 id を削除
```

SQLite の外部キー cascade が設定済みなら、明示削除を省略できる可能性はある。
ただし仕様としては、関連情報が残らないことを受け入れ条件に含める。

### トランザクション

管理対象除外は DB 操作で完結するため、対象 ID 全体を 1 トランザクションで処理してよい。

```text
成功:
全対象を除外

失敗:
DB 操作はロールバック
```

## 11. ごみ箱移動処理仕様

## 11.1 処理順序

```text
1. ids を正規化する
2. DB から対象画像を再取得する
3. 1 件ずつ send2trash でごみ箱へ移動する
4. ごみ箱移動に成功した ID を removed_ids に追加する
5. ごみ箱移動に失敗した ID は failedFiles に追加する
6. removed_ids のサムネイルを削除する
7. removed_ids を DB 管理対象から除外する
8. 結果 DTO を返す
```

## 11.2 ファイル存在なしの場合

対象ファイルが存在しない場合は、ごみ箱移動失敗ではなく、管理対象除外候補に含めてよい。

```text
実ファイル不存在
→ ごみ箱移動は行わない
→ DB 管理対象から除外
→ failedFiles には含めない
```

理由は、起動時整合性チェックでも実ファイル不存在レコードは整理対象だからである。

件数名を厳密に分けたい場合、`missingFileCount` を追加してもよい。
初回実装では省略可とする。

## 11.3 ごみ箱移動失敗時

```text
ごみ箱移動失敗
→ DB レコードは残す
→ サムネイルも残す
→ failedFiles に追加
→ 他画像の処理は継続
```

## 11.4 DB 除外失敗時

```text
ごみ箱移動済み
→ DB 除外失敗
```

この状態は発生し得る。
ファイルシステム操作と DB 操作は同一トランザクションではないため、完全には防げない。

この場合は API 全体を `success=false` とし、ログ出力する。
再起動時の整合性チェックにより、実ファイル不存在レコードとして整理される想定とする。

## 13. 既存 API の廃止方針

## 13.1 delete_image_file

タイル右上 × ボタン廃止により、UI からの呼び出しがなくなる。

今後の扱いは以下のいずれかとする。

```text
A. 完全に削除する
B. 互換用に残すが UI からは呼ばない

```


互換用に残す場合も、内部では `move_image_files_to_trash` へ委譲する。
ただし、名前と挙動がずれるため、新規コードからは呼ばない。

## 17. 注意事項

## 17.1 「削除」という文言を不用意に使わない

UI 上では、主要操作名として「削除」を避ける。

採用する。

```text
ごみ箱へ移動
管理対象から除外
```

避ける。

```text
削除
完全削除
物理削除
```

## 17.2 完全削除を残さない

通常 UI から即時完全削除できる状態を残さない。

```text
Path.unlink() による実画像ファイル削除は使用しない。
```

サムネイルキャッシュ削除には `unlink()` を使ってよい。
対象は派生キャッシュであり、実画像ファイルではないためである。

## 17.3 ごみ箱移動は完全な安全策ではない

ごみ箱移動は即時完全削除より安全だが、復元を保証するものではない。

```text
環境によっては、ごみ箱移動に失敗する
環境によっては、復元できない可能性がある
```

UI 文言では、この点を断定しない。

## Phase 2: ImageDetailModal 前後移動・キーボード操作・ごみ箱移動・リネーム・属性編集ショートカット

## 1. 目的

`ImageDetailModal` を、単なる画像拡大表示ダイアログから、現在表示ページ内の画像を連続確認しながら整理できるダイアログへ拡張する。

主な追加機能は以下。

```text
- 現在ページ内の表示順に従った前後移動
- キーボード左右キーによる前後移動
- ダイアログ下部の前へ / 次へボタン
- 表示中画像のごみ箱移動
- 表示中画像のリネーム
- 属性編集ショートカット
```

ただし、`ImageDetailModal.vue` 自体には API 呼び出しや検索更新ロジックを持たせない。
詳細ダイアログは表示と操作イベント通知に限定し、実処理は `App.vue` / composable / backend API に委譲する。

モーダルに全部やらせると、すぐ「表示担当」ではなく「なんでも課」になる。なんでも課はだいたい燃えます。

## 2. 前提

Phase 1 により、削除系の用語と API は以下に整理されている前提とする。

```text
ごみ箱へ移動

- 成功分は AZViewer の管理対象から除外する

管理対象から除外
- DB 管理対象から外す
- 実画像ファイルは残す

完全削除
- 通常 UI には提供しない
```

本 Phase の `ImageDetailModal` に配置するファイル除去系操作は、`ごみ箱へ移動` のみとする。

`管理対象から除外` は、ヘッダーの選択画像一括操作として扱い、詳細ダイアログには配置しない。

## 8. useImageDetail.js 仕様

## 8.1 役割

画像詳細モーダルの状態を管理する。

既存の詳細表示状態に加えて、現在ページ内の移動対象リストと現在 index を扱う。

## 8.2 管理状態

例。

```js
const detailModal = reactive({
  show: false,
  item: null,
  imageSrc: '',
  items: [],
  currentIndex: -1,
  isLoading: false,
})
```

## 8.3 openDetail

```js
async function openDetail({ item, items }) {
  detailModal.show = true
  detailModal.items = [...(items ?? [])]
  detailModal.currentIndex = detailModal.items.findIndex((x) => x.id === item.id)
  detailModal.item = item
  await loadDetailImage(item)
}
```

`items` は現在ページの `searchResult.items` を渡す。

## 8.4 hasPrevious / hasNext

```js
const hasPrevious = computed(() => detailModal.currentIndex > 0)

const hasNext = computed(() =>
  detailModal.currentIndex >= 0 &&
  detailModal.currentIndex < detailModal.items.length - 1
)
```

## 8.5 movePrevious / moveNext

```js
async function movePrevious() {
  if (!hasPrevious.value) {
    return
  }

  const nextIndex = detailModal.currentIndex - 1
  const nextItem = detailModal.items[nextIndex]
  detailModal.currentIndex = nextIndex
  detailModal.item = nextItem
  await loadDetailImage(nextItem)
}

async function moveNext() {
  if (!hasNext.value) {
    return
  }

  const nextIndex = detailModal.currentIndex + 1
  const nextItem = detailModal.items[nextIndex]
  detailModal.currentIndex = nextIndex
  detailModal.item = nextItem
  await loadDetailImage(nextItem)
}
```

## 8.6 refreshAfterSearch

ごみ箱移動やリネーム後、再検索結果をもとに詳細モーダル状態を更新する。

```js
function refreshDetailItemsAfterSearch(items, preferredId = null, fallbackId = null) {
  const nextItems = [...(items ?? [])]
  detailModal.items = nextItems

  let nextIndex = -1

  if (preferredId != null) {
    nextIndex = nextItems.findIndex((item) => item.id === preferredId)
  }

  if (nextIndex < 0 && fallbackId != null) {
    nextIndex = nextItems.findIndex((item) => item.id === fallbackId)
  }

  if (nextIndex < 0) {
    closeDetail()
    return null
  }

  detailModal.currentIndex = nextIndex
  detailModal.item = nextItems[nextIndex]
  return nextItems[nextIndex]
}
```

## 13. リネーム仕様

## 13.1 基本方針

リネームは、同一フォルダ内のファイル名変更のみとする。

```text
対象:
filename の変更

対象外:
別フォルダへの移動
拡張子変更
上書きリネーム
ファイルコピー
```

## 13.2 拡張子変更禁止

現在の拡張子は維持する。

例。

```text
sample.png
→ new_name.png は可
→ new_name.jpg は不可
```

理由は、画像実体と拡張子が不一致になると、Pillow 読み込みや外部ツール連携で混乱するためである。
ファイルも嘘をつかされると困ります。たぶん黙っていますが。

## 13.3 同名ファイル存在時

同一フォルダ内に変更後ファイル名が既に存在する場合、失敗とする。

```text
上書きしない
```

## 13.4 入力制約

禁止する。

```text
- 空文字
- 前後空白のみ
- パス区切り文字を含む名前
- Windows 禁止文字を含む名前
- 拡張子なし
- 拡張子変更
- 同一ファイル名
```

Windows 禁止文字。

```text
< > : " / \ | ? *
```

## 13.5 リネーム API

### AppApi

```python
def rename_image_file(self, payload=None):
    return self._image_catalog_api.rename_image_file(payload)
```

### ImageCatalogApi

```python
def rename_image_file(self, payload=None):
    ...
```

### payload

```json
{
  "id": 123,
  "filename": "new_name.png"
}
```

フロントから `path` は渡さない。
バックエンド側で ID から現在の path を再取得する。

## 13.6 レスポンス

### 正常時

```json
{
  "success": true,
  "message": "ファイル名を変更しました。",
  "data": {
    "id": 123,
    "filename": "new_name.png",
    "path": "C:/images/new_name.png",
    "folder": "images"
  }
}
```

### 異常時

```json
{
  "success": false,
  "message": "ファイル名を変更できませんでした。",
  "data": null
}
```

## 13.7 Repository

`ImageFileRepository` に path / filename 更新用メソッドを追加する。

```python
def update_file_identity(self, record_id: int, filename: str, path: str, folder: str) -> int:
    ...
```

更新対象。

```text
filename
path
folder
```

`tag_hash` は変更不要。
タグ構成は変わらないためである。

## 13.8 処理順序

```text
1. id / filename を検証する
2. DB から対象画像を取得する
3. 現在 path を確認する
4. 新 filename の拡張子が現在拡張子と一致するか検証する
5. 新 path を作成する
6. 新 path が既に存在しないことを確認する
7. Path.rename() で実ファイルをリネームする
8. DB の filename / path / folder を更新する
9. 結果を返す
```

## 13.9 DB 更新失敗時

```text
実ファイル rename 成功
→ DB 更新失敗
```

この状態は発生し得る。
ファイルシステムと DB は同一トランザクションではないためである。

可能であれば、DB 更新失敗時にファイル名を元に戻す復元処理を試みる。

```text
復元成功:
API success=false
元状態を維持

復元失敗:
API success=false
ログ出力
再起動時整合性チェックで整理される可能性あり
```

## 13.10 リネーム後の詳細表示

リネーム成功後。

```text
1. 現在条件で再検索する
2. リネーム対象 ID が現在ページに存在すれば詳細表示を更新する
3. 存在しなければ詳細ダイアログを閉じる
```

将来的に filename 検索が入った場合、リネームにより現在検索条件から外れる可能性があるためである。

## 17.3 ごみ箱移動

```js
async function handleMoveDetailImageToTrash(item) {
  if (!item?.id) {
    return
  }

  const confirmed = window.confirm(
    [
      'この画像を OS のごみ箱へ移動します。',
      '',
      'ごみ箱への移動に成功した場合、AZViewer の管理対象からも除外されます。',
      '環境によっては、ごみ箱から復元できない場合があります。',
      '',
      '実行しますか？',
    ].join('\n')
  )

  if (!confirmed) {
    return
  }

  const currentIndex = imageDetail.detailModal.currentIndex
  const items = imageDetail.detailModal.items
  const nextCandidateId = items[currentIndex + 1]?.id ?? null
  const previousCandidateId = items[currentIndex - 1]?.id ?? null

  const result = await imageMutations.moveSingleImageToTrash({ id: item.id })

  if (!result?.success) {
    pushToast({
      type: 'danger',
      message: result?.message || '画像をごみ箱へ移動できませんでした。',
    })
    return
  }

  pushToast({
    type: 'success',
    message: '画像をごみ箱へ移動しました。',
  })

  await imageSearch.executeSearch()

  const nextItem = imageDetail.refreshDetailItemsAfterSearch(
    imageSearch.searchResult.items,
    nextCandidateId,
    previousCandidateId
  )

  if (nextItem) {
    await imageDetail.loadDetailImage(nextItem)
  }
}
```

## 19. 受け入れ条件

## 19.1 前後移動

```text
- ImageDetailModal 表示中に前へ / 次へボタンが表示されること
- 現在ページ内の表示順で前後移動できること
- 先頭では前へが disabled になること
- 末尾では次へが disabled になること
- 別ページの画像へは移動しないこと
- ループ移動しないこと
```

## 19.2 キーボード操作

```text
- ArrowLeft で前の画像へ移動できること
- ArrowRight で次の画像へ移動できること
- Escape で閉じられること
- input / textarea / select フォーカス中は左右キー移動しないこと
- リネームモーダル表示中は左右キー移動しないこと
```

## 19.3 ごみ箱へ移動

```text
- ImageDetailModal に「ごみ箱へ移動」ボタンが表示されること
- 押下時に確認 confirm が表示されること
- move_image_files_to_trash に ID 1 件が渡されること
- Path.unlink() による即時完全削除が行われないこと
- 成功後に現在条件で再検索されること
- 次候補が存在すれば次候補が表示されること
- 次候補がなければ前候補が表示されること
- 候補がなければ詳細ダイアログが閉じること
```

## 19.4 リネーム

```text
- ImageDetailModal に「リネーム」ボタンが表示されること
- 現在 filename を初期値としてリネーム入力できること
- 拡張子変更が拒否されること
- 同一フォルダ内の同名ファイルがある場合は失敗すること
- 成功時に実ファイル名が変更されること
- 成功時に image_file_data.filename / path / folder が更新されること
- 成功後に現在条件で再検索されること
- 対象 ID が現在ページに残っていれば詳細表示が更新されること
- 対象 ID が現在ページに存在しなければ詳細ダイアログが閉じること
```

## 19.5 属性編集ショートカット

```text
- ImageDetailModal に「属性編集」ボタンが表示されること
- 押下時に詳細ダイアログが閉じること
- 対象タイルまでスクロールすること
- 対象タイルが edit モードになること
- 保存またはキャンセル後、編集対象状態が解除されること
```

## 19.6 ImageDetailModal 責務

```text
- ImageDetailModal.vue が API 呼び出しを直接行わないこと
- ImageDetailModal.vue が再検索処理を直接行わないこと
- ImageDetailModal.vue がタイルスクロールを直接行わないこと
- 操作は emit で親へ通知すること
```

## 21. 注意事項

## 21.1 詳細ダイアログに管理対象除外は置かない

詳細ダイアログは、画像を見ながらファイルを整理する場として扱う。

そのため、置く操作は `ごみ箱へ移動` とする。

`管理対象から除外` は選択画像一括操作としてヘッダー側に置く。

## 21.2 完全削除は提供しない

本 Phase でも、実画像ファイルを即時完全削除する操作は提供しない。

```text
Path.unlink() による実画像ファイル削除は禁止
```

ただし、サムネイルキャッシュ削除は対象外。

## 21.3 前後移動は現在ページ内のみ

詳細ダイアログから次ページをまたいだ移動は行わない。

将来的に必要になった場合は、ページング込みの詳細ビューとして別途設計する。
今回の仕様に混ぜると、詳細ダイアログがページャーと検索状態まで抱え始める。重すぎるリュックです。

## 21.4 リネームと検索条件の関係

将来的に filename 検索が追加された場合、リネーム後に現在検索条件から外れる可能性がある。

そのため、リネーム成功後は必ず再検索し、対象 ID が現在ページに存在するか確認する。
