# 実装設計書

## AZViewer DB接続管理の SQLAlchemy Core 移行

## 1. 目的

AZViewer の SQLite 接続管理方式を見直し、従来の `sqlite3.Connection` 共有方式から、SQLAlchemy Core の `Engine` 共有方式へ移行する。

これにより、以下を実現する。

```text
- 単一 sqlite3.Connection 共有によるスレッド干渉を避ける
- Repository 層から Connection の寿命管理を分離する
- SQL 実行単位ごとに短い Connection / Transaction を使う
- SELECT と更新処理の接続利用方針を明確化する
- SQLite の WAL / busy_timeout / foreign_keys を明示設定する
- 長時間 transaction を避け、DB ロック問題を起こしにくくする
```

本変更では ORM は導入しない。

使用するのは SQLAlchemy Core の以下に限定する。

```text
- create_engine
- Engine
- Connection
- text()
- engine.connect()
- engine.begin()
```

## 2. 背景

従来は、アプリケーション全体で `sqlite3.Connection` を保持し、Repository へ渡す構成だった。

しかし、複数スレッドや複数処理経路から DB へアクセスする場面が増えたことで、以下の問題が起きやすくなっている。

```text
- 同じ sqlite3.Connection を複数スレッドで共有する危険
- check_same_thread=False による制限解除後の排他制御不足
- 書き込み transaction 中の SELECT 停止
- commit / rollback / cursor ライフサイクルの干渉
- database is locked 系エラー
- 長時間 transaction による UI 操作待ち
```

AZViewer は現在、画像検索、詳細更新、タグ更新、モデル更新、caption 読み込み、ワイルドカード出力、マスタメンテナンス、ファイル移動、ごみ箱移動、起動時整合性確認など、DB 利用箇所が増えている。

そのため、単一 Connection を大事に抱える方式はやめる。

接続を抱きしめすぎると、愛ではなくロックになります。

## 3. 基本方針

## 3.1 採用方針

```text
- SQLAlchemy Core を採用する
- ORM は使わない
- Repository は sqlite3.Connection ではなく Engine を受け取る
- Engine は DatabaseLifecycleManager が生成・保持する
- SELECT は原則 with engine.connect()
- INSERT / UPDATE / DELETE は原則 with engine.begin()
- 複数テーブルを整合更新する処理は、上位で transaction を1つだけ開く
- Repository 内部メソッドは必要に応じて Connection を受け取れるようにする
- poolclass は NullPool
- PRAGMA foreign_keys = ON
- PRAGMA busy_timeout = 30000
- PRAGMA journal_mode = WAL
- journal_mode=WAL は初期化時に一度設定し、結果を確認する
- 長時間 transaction を持たない
- ファイル操作・画像処理・メタデータ読み取り中は transaction を保持しない
- 重い一括処理中は既存方針どおり UI ロックする
```

## 3.2 避ける方針

```text
- 単一 sqlite3.Connection をアプリ全体で共有する
- check_same_thread=False にして共有 Connection を使い続ける
- SQLAlchemy StaticPool で単一 Connection 共有へ寄せる
- ファイル操作中に DB transaction を保持する
- 画像処理中に DB transaction を保持する
- メタデータ読み取り中に DB transaction を保持する
- 長い一括処理全体を1つの transaction にする
```

## 6. DatabaseLifecycleManager 仕様変更

## 6.1 役割

`DatabaseLifecycleManager` は、SQLite Connection ではなく SQLAlchemy Engine のライフサイクルを管理する。

## 6.2 変更前の責務

```text
- sqlite3.Connection を生成する
- sqlite3.Connection を保持する
- get_connection() で Connection を返す
- close() で Connection を閉じる
```

## 6.3 変更後の責務

```text
- DB ファイルパスを解決する
- SQLAlchemy Engine を生成する
- Engine を保持する
- Engine を返す
- SQLite 初期 PRAGMA を設定する
- アプリ終了時に engine.dispose() する
```

## 6.4 持たせない責務

```text
- Repository インスタンス生成
- Service インスタンス生成
- API インスタンス生成
- 起動時 cleanup の業務処理本体
- tag_hash 補完処理本体
- thumbnail 補完処理本体
```

これらは既存通り `AppLifecycleApi` や各 Service に置く。

`DatabaseLifecycleManager` に何でも持たせ始めると、名前は Lifecycle なのに中身は町内会長になります。

## 6.5 公開インターフェース

```python
from sqlalchemy.engine import Engine


class DatabaseLifecycleManager:
    def initialize(self) -> None:
        ...

    def get_engine(self) -> Engine:
        ...

    def dispose(self) -> None:
        ...
```

互換用に一時的に `close()` を残してもよい。

```python
def close(self) -> None:
    self.dispose()
```

ただし、新規実装では `dispose()` を使用する。

## 6.6 get_connection() の扱い

`get_connection()` は廃止する。

移行途中の互換用として残す場合でも、最終的には全参照を削除する。

```text
非推奨:
database_lifecycle_manager.get_connection()

推奨:
database_lifecycle_manager.get_engine()
```

## 7. Engine 作成仕様

## 7.1 新規関数

`database_lifecycle_manager.py` 内、または `backend/db/database_engine.py` を新設して、Engine 作成処理を分離する。

推奨は新規ファイル追加。

```text
backend/db/database_engine.py
```

## 7.2 create_database_engine

```python
from __future__ import annotations

from pathlib import Path

from sqlalchemy import create_engine, event, text
from sqlalchemy.engine import Engine
from sqlalchemy.pool import NullPool


def create_database_engine(db_path: str | Path) -> Engine:
    db_file = Path(db_path)
    db_file.parent.mkdir(parents=True, exist_ok=True)

    engine = create_engine(
        f"sqlite:///{db_file.as_posix()}",
        connect_args={
            "timeout": 30,
        },
        poolclass=NullPool,
        future=True,
    )

    @event.listens_for(engine, "connect")
    def set_sqlite_pragma(dbapi_connection, connection_record):
        cursor = dbapi_connection.cursor()
        cursor.execute("PRAGMA foreign_keys = ON")
        cursor.execute("PRAGMA busy_timeout = 30000")
        cursor.close()

    with engine.begin() as conn:
        journal_mode = conn.execute(
            text("PRAGMA journal_mode = WAL")
        ).scalar()

        if str(journal_mode).lower() != "wal":
            raise RuntimeError(
                f"SQLite WAL mode を有効化できませんでした: {journal_mode}"
            )

    return engine
```

## 7.3 PRAGMA 方針

## 7.3.1 connect event で毎回設定するもの

```sql
PRAGMA foreign_keys = ON;
PRAGMA busy_timeout = 30000;
```

理由。

```text
- foreign_keys は Connection ごとの設定であるため
- busy_timeout も Connection ごとの設定として明示するため
```

## 7.3.2 初期化時に一度設定するもの

```sql
PRAGMA journal_mode = WAL;
```

理由。

```text
- journal_mode は DB ファイル側に残る設定であるため
- 毎 Connection で実行する必要は薄いため
- 設定結果が wal になったことを起動時に確認したいため
```

## 8. DatabaseLifecycleManager 実装例

```python
from __future__ import annotations

from pathlib import Path

from sqlalchemy.engine import Engine

from backend.db.database_engine import create_database_engine


class DatabaseLifecycleManager:
    """AZViewer の SQLite Engine ライフサイクルを管理する。

    Repository / Service / API インスタンスは保持しない。
    DB 接続方式の詳細をアプリケーション各所から隠蔽し、
    SQLAlchemy Engine の生成、取得、破棄のみを担当する。
    """

    def __init__(self, db_path: str | Path | None = None):
        self._db_path = Path(db_path) if db_path else self._resolve_default_db_path()
        self._engine: Engine | None = None

    def initialize(self) -> None:
        if self._engine is not None:
            return

        self._engine = create_database_engine(self._db_path)

    def get_engine(self) -> Engine:
        if self._engine is None:
            self.initialize()

        if self._engine is None:
            raise RuntimeError("Database engine が初期化されていません。")

        return self._engine

    def dispose(self) -> None:
        if self._engine is None:
            return

        self._engine.dispose()
        self._engine = None

    def close(self) -> None:
        self.dispose()

    def _resolve_default_db_path(self) -> Path:
        return Path("data") / "azviewer.db"
```

## 10. Repository 実装方針

## 10.1 コンストラクタ

`ImageFileRepository` は `sqlite3.Connection` ではなく `Engine` を受け取る。

```python
from sqlalchemy.engine import Engine


class ImageFileRepository:
    def __init__(self, engine: Engine):
        self.engine = engine
```

## 10.2 単発 SELECT

単発 SELECT は Repository メソッド内で `engine.connect()` を使う。

```python
from sqlalchemy import text


def find_by_id(self, image_id: int):
    with self.engine.connect() as conn:
        return conn.execute(
            text("""
                SELECT *
                FROM image_file_data
                WHERE id = :image_id
            """),
            {"image_id": image_id},
        ).mappings().first()
```

## 10.3 単発更新

単一 SQL または単一意味単位で完結する更新は `engine.begin()` を使う。

```python
def update_comment(self, image_id: int, comment: str | None) -> None:
    with self.engine.begin() as conn:
        conn.execute(
            text("""
                UPDATE image_file_data
                SET comment = :comment
                WHERE id = :image_id
            """),
            {
                "image_id": image_id,
                "comment": comment,
            },
        )
```

## 10.4 複数テーブル整合更新

複数テーブルをまとめて更新する処理は、上位メソッドで `engine.begin()` を1回だけ開く。

内部メソッドは `conn` を受け取る。

```python
def update_detail_with_tags_and_model(
    self,
    detail,
    tags: list[str],
    model_name: str | None,
) -> None:
    with self.engine.begin() as conn:
        self._update_detail(conn, detail)
        self._replace_tags(conn, detail.id, tags)
        self._replace_model(conn, detail.id, model_name)
        self._replace_tag_hash(conn, detail.id)
```

内部メソッド例。

```python
def _update_detail(self, conn, detail) -> None:
    conn.execute(
        text("""
            UPDATE image_file_data
            SET
                rating = :rating,
                is_checked = :is_checked,
                is_favorite = :is_favorite,
                comment = :comment
            WHERE id = :id
        """),
        {
            "id": detail.id,
            "rating": detail.rating,
            "is_checked": detail.is_checked,
            "is_favorite": detail.is_favorite,
            "comment": detail.comment,
        },
    )
```

## 10.5 実装ルール

```text
- public メソッドは原則 engine.connect() / engine.begin() を使ってよい
- ただし、複合更新に使われる処理は private メソッド化し、conn を受け取る
- private メソッド内では commit / rollback を行わない
- private メソッド内では engine.begin() を開かない
- transaction の境界は上位 public メソッドで決める
```

変更前。

```python
cursor.execute(

)
```

変更後。

```python
conn.execute(

        FROM image_file_data
```


## 12. IN 句の扱い

SQLAlchemy Core の `text()` で可変長 `IN` 句を扱う場合は、`bindparam(expanding=True)` を使う。

```python
from sqlalchemy import bindparam, text


stmt = text("""
    SELECT *
    FROM image_file_data
    WHERE id IN :ids
""").bindparams(bindparam("ids", expanding=True))

rows = conn.execute(stmt, {"ids": image_ids}).mappings().all()
```

空配列の場合は SQL 実行前に戻す。

```python
if not image_ids:
    return []
```

## 14. Service 層変更方針

## 14.1 Service は Engine を直接持たない

原則として、既存通り Service は Repository を使う。

ただし、Service 内で Repository を生成している場合は、`get_connection()` ではなく `get_engine()` を渡す。

変更前。

```python
repository = ImageFileRepository(
    self._database_lifecycle_manager.get_connection()
)
```

変更後。

```python
repository = ImageFileRepository(
    self._database_lifecycle_manager.get_engine()
)
```

## 14.2 一括処理の transaction 粒度

一括処理全体を1 transaction にしない。

対象。

```text
- prompt タグ読み取り
- caption タグ読み込み
- 一括タグ追加
- サムネイル補完
- 起動時 cleanup
- ファイル移動
- ごみ箱移動
```

基本方針。

```text
- 1画像単位または小さなまとまりで DB 更新する
- 1件失敗しても継続可能な処理は、個別に失敗扱いにする
- ファイル操作を DB transaction 内に入れない
```

## 15. ファイル操作を伴う処理の方針

## 15.1 ごみ箱移動

処理順序。

```text
1. DB から対象画像情報を取得する
2. DB Connection を閉じる
3. 実画像をごみ箱へ移動する
4. 成功した ID を集計する
5. 成功分だけ DB 管理対象から除外する
```

避けること。

```text
- DB transaction を開いたまま send2trash を実行する
```

## 15.2 ファイル移動

処理順序。

```text
1. DB から対象画像情報を取得する
2. DB Connection を閉じる
3. 実ファイルを移動する
4. 成功したファイルだけ path / folder 更新する
```

## 15.3 リネーム

処理順序。

```text
1. DB から対象画像情報を取得する
2. DB Connection を閉じる
3. 実ファイルをリネームする
4. DB の filename / path / folder を更新する
5. DB 更新に失敗した場合は、可能な範囲で実ファイル名を戻す
```

## 15.4 サムネイル生成

処理順序。

```text
1. DB から対象画像情報を取得する
2. DB Connection を閉じる
3. Pillow でサムネイル生成する
4. 必要なら生成結果だけ DB / 結果集計へ反映する
```

Pillow 処理中に transaction を持たない。

画像処理中に DB ロックを握るのは、レジで財布を出したまま昼寝するようなものです。

## 16.3 remove_image_files_from_catalog

管理対象除外は DB 内の関連テーブル削除とサムネイルキャッシュ削除が絡む。

推奨順序。

```text

2. DB transaction で関連レコード削除
3. transaction 成功後にサムネイルキャッシュ削除
```

サムネイル削除失敗は、DB 削除 rollback の理由にしない。

キャッシュはキャッシュであり、王族ではない。

## 17. ImageRegisterApi / ImageFileImportService 変更方針

## 17.1 登録処理

画像登録時は、DB 登録とサムネイル生成を分ける。

推奨順序。

```text
1. ファイル走査
2. 既存登録済み path を DB から取得
3. 新規候補を作成
4. DB へ登録
5. 登録成功したレコードを取得
6. DB transaction 外でサムネイル生成
```

## 17.2 insert_many

`insert_many` は `engine.begin()` を使う。

大量登録時に1件ずつ transaction を開くと遅くなるため、1回の登録操作内ではまとめてよい。

ただし、画像読み込みやサムネイル生成は含めない。

## 18. StartupCleanupService 変更方針

## 18.1 処理順序

```text
1. DB から登録済み path 一覧を取得
2. DB Connection を閉じる
3. 実ファイル存在確認を行う
4. 不存在 ID を集計する
5. DB transaction で管理対象から除外する
6. transaction 成功後にサムネイルキャッシュを削除する
```

実ファイル存在確認中に transaction を保持しない。

## 19. エラー処理

## 19.1 database is locked

`database is locked` が出た場合、まず以下を疑う。

```text
- 長時間 transaction を保持していないか
- ファイル操作中に transaction を保持していないか
- 画像処理中に transaction を保持していないか
- 複数 writer を同時に走らせていないか
- UI ロック対象の重処理で多重実行を許していないか
```

安易に busy_timeout を延ばして隠さない。

timeout を伸ばすだけの対策は、火災報知器の音量を下げるようなものです。

## 19.2 ログ出力

DB 更新失敗時は、最低限以下をログへ残す。

```text
- 対象 API / Service
```

## 23. 移行時の注意

## 23.1 全面置換を一気にやりすぎない

最初に `DatabaseLifecycleManager` と `ImageFileRepository` の constructor だけ変えて、全メソッドを一気に書き換えると差分が大きくなりすぎる。

推奨は以下。

```text
Phase 1:
- Engine 作成
- DatabaseLifecycleManager 変更
- Repository constructor 変更

Phase 2:
- create_table / search / find 系 SELECT を移行

Phase 3:
- update_detail / replace_tags / replace_model / replace_tag_hash を移行

Phase 4:
- 削除、移動、リネーム、マスタメンテナンスを移行

Phase 5:
- Service 側の get_connection 参照削除
```

## 23.2 row_factory 前提を捨てる

sqlite3 の `row_factory` 前提コードは SQLAlchemy では使わない。

`mappings()` を使う。

## 23.3 lastrowid

INSERT 後の ID 取得は、SQLAlchemy の result を使う。

```python
result = conn.execute(...)
new_id = result.lastrowid
```

ただし、SQLite で期待通り取得できるかは実装時に確認する。

必要なら INSERT 後に UNIQUE key で再 SELECT する。

## 23.4 executescript は使わない

複数 SQL をまとめた `executescript` は使わない。

SQLAlchemy では `conn.execute(text(...))` を SQL 単位で実行する。

## 23.5 PRAGMA の確認

起動時に以下を確認できる簡易ログを出してよい。

```text
- journal_mode
- foreign_keys
- busy_timeout
```

ただし UI 表示は不要。

## 24. 完了後の削除候補

SQLAlchemy 移行が完了したら、以下を確認する。

```text
- backend/db/connection_manager.py が不要になっていないか
- sqlite3 import が残っていないか
- get_connection() が残っていないか
- check_same_thread 設定が残っていないか
```

`backend/db/connection_manager.py` が完全に不要であれば削除する。

ただし、ファイル構成上の削除は移行完了後に行う。

## 25. 最終方針

本変更後の DB アクセス方針は以下とする。

```text
AZViewer は SQLAlchemy Core の Engine をアプリ全体で共有する。
sqlite3.Connection は共有しない。

Repository は Engine を受け取り、SQL 実行時だけ Connection を開く。
SELECT は connect、更新は begin を使う。

複数テーブルを整合更新する場合は、上位で transaction をまとめる。
ファイル操作、画像処理、メタデータ読み取り中は transaction を保持しない。

SQLite は WAL / foreign_keys / busy_timeout を明示設定する。
重い処理は既存方針どおり UI ロックと組み合わせる。
```

## ページング仕様見直し

## 1. 目的

画像一覧画面のページング仕様を見直し、以下を実現する。

* ページサイズ選択肢の拡張
* 大量表示時の注意喚起
* ページリンク表示の改善
* ページ変更・ページサイズ変更時の選択状態制御の明文化
* 詳細ビューア、一括操作、表示中全選択との関係整理

本変更では、検索・ページングを DB 側で実行する既存方針は維持する。

## 2. 基本方針

### 2.1 DB 側ページングは維持する

検索条件、ページ番号、ページサイズ、ソート条件をフロントエンドからバックエンドへ渡し、バックエンド側で対象範囲を取得する。

フロントエンド側で全件取得してからページングしない。

```text
維持する方針:
- 検索・ページングは DB 側で実行
- フロントエンドは条件のみ送信
- SearchImageFilesResult の DTO 構造は維持
```

### 2.2 一括操作は現在ページ内を対象とする

以下の操作は、現在画面に表示されている `searchResult.items` を基準とする。

```text
- 表示中をすべて選択
- 表示中の選択を解除
- ワイルドカード出力
- タグ出力
- ファイル移動
- ごみ箱移動
- 管理対象から除外
- 一括属性編集
- 一括タグ追加
```

検索結果全件を対象にする仕様は追加しない。

### 2.3 ページ変更時は選択状態を解除する

ページ変更、ページサイズ変更、検索、ソート変更時は、既存方針通り選択状態を解除する。

```text
選択状態を解除するタイミング:
- 検索
- ページ変更
- ページサイズ変更
- ソート変更
- 一括操作完了
```

見えていない画像が選択されたままになる状態を避ける。

### 2.4 詳細ビューアの前後移動は現在ページ内に限定する

詳細ビューアの「前へ」「次へ」は、現在ページの `items` 内だけを対象とする。

ページ末尾で次ページを自動取得しない。
ページ先頭で前ページを自動取得しない。

```text
詳細ビューア:
- 現在ページ内でのみ前後移動
- ページ境界をまたぐ自動読み込みなし
- 詳細ビューア側に検索・ページング制御を持たせない
```

### 6.4.6 ページリスト生成

```js
const displayedPages = computed(() => {
  const current = Number(props.page || 1)
  const total = Number(props.totalPages || 1)

  if (total <= 1) {
    return []
  }

  if (total <= 7) {
    return Array.from({ length: total }, (_, index) => index + 1)
  }

  const pages = new Set()

  pages.add(1)
  pages.add(total)

  for (let page = current - 2; page <= current + 2; page += 1) {
    if (page >= 1 && page <= total) {
      pages.add(page)
    }
  }

  const sorted = [...pages].sort((a, b) => a - b)
  const result = []

  for (let index = 0; index < sorted.length; index += 1) {
    const page = sorted[index]
    const prev = sorted[index - 1]

    if (prev && page - prev > 1) {
      result.push('ellipsis-' + prev + '-' + page)
    }

    result.push(page)
  }

  return result
})
```

### 7.1.1 page_size 正規化

# 実装仕様書

## 大量表示時の Tile 分割描画および現在ページ全選択仕様見直し

## 1. 目的

ページサイズ拡張に伴い、`500 / 1000 / 2500` 件表示時の初回描画負荷を軽減する。

あわせて、Tile 分割描画を導入した場合でも、現在ページ全選択・一括操作・詳細ビューアの対象がぶれないように、以下を明確化する。

```text
描画対象:
- renderedItems

操作対象:
- searchResult.items
```

これにより、Tile の段階的表示と、現在ページ全体に対する一括操作を両立する。

## 2. 基本方針

### 2.1 分割描画はフロントエンドのみで行う

バックエンド API、DB 検索、Repository のページング仕様は変更しない。

`search_image_files` は従来通り、現在ページ分のレコードをまとめて返す。

```text
バックエンド:
- 現在ページの検索結果を返す

フロントエンド:
- 受け取った items を分割して Tile 描画する
```

### 2.2 `searchResult.items` は現在ページ全体を表す

`searchResult.items` は、現在ページに属する全画像の配列として扱う。

これは、Tile がまだ描画されている途中でも変わらない。

```text
page_size = 1000 の場合:
searchResult.items.length = 最大 1000
renderedItems.length = 分割描画の進行状況に応じて 0〜1000
```

### 2.3 `renderedItems` は描画専用

`renderedItems` は `Tile` の `v-for` にだけ使う。

以下には使わない。

```text
- 現在ページ全選択
- selectedCount
- 一括操作対象 ID
- ワイルドカード出力対象
- タグ出力対象
- 詳細ビューアの現在ページ内リスト
```

### 2.4 現在ページ全選択は `searchResult.items` を対象にする

分割描画中でも、現在ページ全選択は `searchResult.items` に含まれる全 ID を対象とする。

描画済みの `renderedItems` のみを対象にしない。

### 2.5 詳細ビューアは現在ページ全体を対象にする

Tile クリックで詳細ビューアを開く際、詳細ビューアに渡す `items` は `renderedItems` ではなく `searchResult.items` とする。

これにより、詳細ビューアの「前へ」「次へ」は、描画済み Tile の範囲ではなく、現在ページの検索結果全体で動作する。

## 3. 変更概要

以下を追加・変更する。

```text
- Content.vue に renderedItems を追加
- page_size >= 500 の場合、Tile を一定件数ずつ分割描画する
- page_size < 500 の場合は即時描画してよい
- 分割描画中は進行状況を表示する
```

## 6.5 分割描画処理

### 6.5.1 基本フロー

```text
1. searchResult.items の変更を watch する
2. renderToken を更新する
3. renderedItems を空にする
4. items.length < 500 の場合は即時代入する
5. items.length >= 500 の場合は 50 件ずつ renderedItems に追加する
6. 各 batch の間で requestAnimationFrame を挟む
7. 新しい検索結果が来たら古い分割描画を中断する
```

### 6.5.2 実装例

```js
watch(
  () => props.searchResult.items,
  (items) => {
    startRenderItems(items ?? [])
  },
  {
    immediate: true,
  }
)

async function startRenderItems(items) {
  const token = ++renderToken

  renderedItems.value = []

  if (!shouldChunkRender(items)) {
    renderedItems.value = [...items]
    isChunkRendering.value = false
    return
  }

  isChunkRendering.value = true

  for (let index = 0; index < items.length; index += CHUNK_RENDER_BATCH_SIZE) {
    if (token !== renderToken) {
      return
    }

    renderedItems.value.push(
      ...items.slice(index, index + CHUNK_RENDER_BATCH_SIZE)
    )

    await new Promise((resolve) => requestAnimationFrame(resolve))
  }

  if (token === renderToken) {
    isChunkRendering.value = false
  }
}
```

### 6.5.3 コンポーネント破棄時

`onBeforeUnmount` で `renderToken` を進め、未完了の分割描画を無効化する。

```js

})
```

## 7.4 isAllCurrentPageSelected

```js
const isAllCurrentPageSelected = computed(() => {
  const ids = currentPageImageIds.value

  if (ids.length === 0) {
    return false
  }

  return ids.every((id) => selectedImageIdSet.value.has(id))
})
```

## 7.5 handleToggleCurrentPageSelection

```js
function handleToggleCurrentPageSelection() {
  const ids = currentPageImageIds.value

  if (ids.length === 0) {
    return
  }

  const next = new Set(selectedImageIdSet.value)

  if (isAllCurrentPageSelected.value) {
    for (const id of ids) {
      next.delete(id)
    }
  } else {
    for (const id of ids) {
      next.add(id)
    }
  }

  selectedImageIdSet.value = next
}
```

## 15. 現在ページ全選択仕様

## 15.1 基本方針

現在ページ全選択は、現在の `searchResult.items` に含まれる全画像を対象とする。

Tile の分割描画を行っている場合でも、描画済みの `renderedItems` ではなく、現在ページ全体の `searchResult.items` を対象とする。

## 15.2 対象

```text
- 現在ページの検索結果
- searchResult.items に含まれる画像 ID
```

## 15.3 対象外

```text
- まだ取得していない別ページの画像
- 検索条件に一致する全件
- 描画済みの Tile のみ
- DOM 上に存在する Tile のみ
```

## 15.4 ボタン文言

未全選択時。

```text
現在ページをすべて選択 ({currentPageCount})
```

全選択済み時。

```text
現在ページの選択を解除
```

## 15.5 判定

現在ページ内の全 ID が `selectedImageIdSet` に含まれている場合、全選択済みとみなす。

## 15.6 切り替え

全選択済みでない場合。

```text
searchResult.items の全 ID を selectedImageIdSet に追加する
```

全選択済みの場合。

```text
searchResult.items の全 ID を selectedImageIdSet から削除する
```

## 15.7 分割描画中の扱い

分割描画中でも、現在ページ全選択ボタンは使用可能とする。

後から描画された Tile は、`selectedImageIdSet` の状態に従ってチェック済み表示になる。

## 16. 一括操作との関係

## 16.1 一括操作対象

一括操作対象は、従来通り `selectedImageIdSet` に含まれる ID とする。

```text
対象:
- selectedImageIdSet

非対象:
- renderedItems
```

## 16.2 現在ページ全選択後の一括操作

`page_size = 2500` の状態で現在ページ全選択を行った場合、最大 2500 件が `selectedImageIdSet` に入る。

この状態で一括操作を行った場合、選択済み 2500 件を対象とする。

## 16.3 確認ダイアログ

既存の一括操作確認ダイアログは維持する。

件数は `selectedCount` を使う。

## 17. キャンセル・競合制御

## 17.1 分割描画中に検索条件が変わった場合

以下の操作が発生した場合、古い分割描画は中断する。

```text
- 検索
- ページ変更
- ページサイズ変更
- ソート変更
- 重複タグ構成検索条件反映
```

実装上は `renderToken` を更新する。

## 17.2 分割描画中に現在ページ全選択した場合

許可する。

```text
- selectedImageIdSet には searchResult.items 全体の ID が入る
- まだ描画されていない Tile は、描画された時点で checked 表示になる
```

## 17.3 分割描画中に詳細ビューアを開いた場合

許可する。

クリック可能な Tile は既に `renderedItems` に含まれているため、対象 item は存在する。

詳細ビューアには `searchResult.items` 全体を渡す。

## 17.4 分割描画中に一括操作を実行した場合

許可する。

ただし、一括操作開始時に LoadingOverlay が表示される場合、背面の分割描画は継続しても中断してもよい。

推奨は、中断しない。

理由。

```text
一括操作後は再検索されるため、検索結果更新時に renderToken により古い描画は自然に無効化される。
```

```text
- 分割描画中にページ変更しても古い Tile が混ざらないこと
```

```text
- 現在ページの全画像が選択済みの場合、押下で全 ID が選択解除されること
```

## 20. 注意点

## 20.1 renderedItems を操作対象にしない

最重要。

```text
NG:
- renderedItems を現在ページ全選択の対象にする
- renderedItems を一括操作の対象にする
- renderedItems を詳細ビューアの items に渡す
```

`renderedItems` はあくまで DOM 描画負荷を分散するための表示専用配列である。

## 20.2 名前の意味を揃える

`visible` という名前は、分割描画導入後に「DOM 上に見えているもの」と誤解される。

可能なら currentPage 系へリネームする。

ただし、大規模差分を避ける場合は、既存名を残しつつ画面文言だけ「現在ページ」へ変えてもよい。

## 20.3 仮想スクロールは今回対象外

本仕様では仮想スクロールを導入しない。

理由。

```text
- Tile 高さがコメント、タグ、モデル、編集状態で変動する
- 実装複雑度が高い
- 今回の目的は大量表示時の初回描画固まりを軽減すること
```

## 20.4 分割描画は総描画量を減らすものではない

分割描画は、DOM 生成を小分けにして UI フリーズ感を抑えるための対応である。

最終的に 2500 件を表示する場合、2500 件分の Tile は DOM 上に存在する。

つまり、これは「負荷を消す」対策ではなく、「一気に来る負荷を分割する」対策である。
2500 件表示は、相変わらず腹筋ローラーです。効きます。
