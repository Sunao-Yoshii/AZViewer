# 実装仕様書

## タグ部分一致検索機能追加

## 1. 目的

画像一覧画面において、登録済みタグ名に含まれる文字列を指定し、その文字列を含むタグが紐づいている画像を検索できるようにする。

既存のタグ検索は、選択済みタグ名を完全一致 AND 条件として扱う。

今回追加する機能は、それとは別に、タグ名の部分一致条件を検索フォームから指定するものである。

```text
既存:
tags = ["blue eyes", "smile"]
→ blue eyes と smile の両方を持つ画像を検索

今回:
tag_keyword = "hair"
→ hair を含むタグを1つ以上持つ画像を検索
```

## 2. 基本方針

## 2.1 検索条件として `tag_keyword` を追加する

既存の `search_image_files` payload に、タグ部分一致検索用の `tag_keyword` を追加する。

```json
{
  "path": "",
  "rating": null,
  "is_checked": null,
  "is_favorite": null,
  "tags": ["blue eyes", "smile"],
  "folder": "set01",
  "model": "waiNSFWIllustrious_v50",
  "tag_hash": null,
  "tag_set": null,
  "tag_keyword": "hair",
  "page": 1,
  "page_size": 25,
  "sort": "id_desc"
}
```

## 2.2 `tags` とは別条件として扱う

`tags` と `tag_keyword` は意味が異なるため、混在させない。

```text
tags:
  タグ完全一致 AND 検索

tag_keyword:
  タグ名部分一致検索
```

両方指定された場合は、既存条件と同様に AND 条件で組み合わせる。

```text
tags = ["blue eyes"]
tag_keyword = "hair"

→ blue eyes を持ち、
  かつ hair を含むタグを1つ以上持つ画像
```

## 2.3 入力が空の場合は条件に含めない

以下の場合、タグ部分一致条件は検索 SQL に追加しない。

```text
- null
- undefined
- 空文字
- trim 後に空文字
```

## 2.4 SQL は `EXISTS` サブクエリを使う

二段階 SQL ではなく、既存の検索 SQL に `EXISTS` 条件を追加する。

理由は、既存の画像検索が複数条件を `ImageFileRepository.search(...)` に集約しており、タグ部分一致だけ別経路にすると検索条件の合成が崩れるためである。

```sql
EXISTS (
    SELECT 1
    FROM tag_image_link til
    INNER JOIN tag t
        ON t.id = til.tag_id
    WHERE til.image_file_id = i.id
      AND t.name LIKE :tag_keyword_like ESCAPE '\'
)
```

ここで二段階 SQL にすると、ページング、ソート、既存条件、`total_count` の整合性が面倒になります。検索処理が二重帳簿になります。だいたい税務調査、じゃなくてバグ調査が来ます。

## 8.3 `_append_tag_keyword_condition(...)` 追加

```python
def _append_tag_keyword_condition(
    self,
    where_clauses: list[str],
    params: dict,
    tag_keyword: str | None,
) -> None:
    keyword = str(tag_keyword or "").strip().lower()

    if not keyword:
        return

    where_clauses.append("""
        EXISTS (
            SELECT 1
            FROM tag_image_link til
            INNER JOIN tag t
                ON t.id = til.tag_id
            WHERE til.image_file_id = i.id
              AND t.name LIKE :tag_keyword_like ESCAPE '\\'
        )
    """)

    params["tag_keyword_like"] = f"%{self._escape_like(keyword)}%"
```

## 8.4 `_escape_like(...)` 追加

`%`, `_`, `\` をリテラルとして検索できるようにする。

```python
def _escape_like(self, value: str) -> str:
    return (
        value
        .replace("\\", "\\\\")
        .replace("%", "\\%")
        .replace("_", "\\_")
    )
```

### 理由

SQLite の `LIKE` では `%` と `_` がワイルドカードとして扱われる。

今回の仕様は「入力文字列を含むタグ名を検索」であり、ユーザーが `%` を入力したときに全件一致のような挙動になるのは不自然である。

## 8.5 SQL 方針

タグ部分一致条件は、以下の SQL を WHERE 句へ追加する。

```sql
EXISTS (
    SELECT 1
    FROM tag_image_link til
    INNER JOIN tag t
        ON t.id = til.tag_id
    WHERE til.image_file_id = i.id
      AND t.name LIKE :tag_keyword_like ESCAPE '\'
)
```

## 8.6 total_count への反映

`items` 取得 SQL と `total_count` 取得 SQL の両方に、同じ WHERE 条件を使う。

既存のタグ検索でも、一覧取得 SQL だけでなく `total_count` 算出 SQL に同じタグ条件を適用する方針です。
今回も同じです。

```text
- items 取得 SQL
- total_count 取得 SQL
```

両方に `tag_keyword` 条件を反映する。

## 8.7 既存タグ AND 条件との併用

既存のタグ AND 条件は変更しない。

既存のタグ AND 条件は、指定されたタグすべてを持つ画像を `GROUP BY` と `HAVING COUNT(DISTINCT t.name)` で絞り込む仕様です。

`tag_keyword` は追加の `EXISTS` 条件として扱う。

```sql
WHERE
    ...
    AND i.id IN (
        SELECT
            til.image_file_id
        FROM tag_image_link til
        INNER JOIN tag t
            ON t.id = til.tag_id
        WHERE t.name IN (:tag_0, :tag_1)
        GROUP BY til.image_file_id
        HAVING COUNT(DISTINCT t.name) = :tag_count
    )
    AND EXISTS (
        SELECT 1
        FROM tag_image_link til_kw
        INNER JOIN tag t_kw
            ON t_kw.id = til_kw.tag_id
        WHERE til_kw.image_file_id = i.id
          AND t_kw.name LIKE :tag_keyword_like ESCAPE '\'
    )
```

実装上は alias の衝突を避けるため、`til_kw`, `t_kw` のように別名にすることを推奨する。

```text
hair を含むタグを持つ画像だけ表示される
```

## 13.2 空条件

```text
期待:

既存検索結果と同じ
```

## 13.3 既存 tags との併用

```text
期待:

```

## 13.4 folder との併用

```text
期待:
folder が set01 で、hair を含むタグを持つ画像だけ表示される
```



## 13.7 LIKE 特殊文字

```text
期待:

ワイルドカード扱いにならない
```

## 14. 注意事項

## 14.1 既存のタグ候補検索とは別物

既存の `fetch_tags_for_search` は、タグ検索ダイアログに表示する候補を取得する API であり、画像一覧そのものを絞り込む API ではない。

既存のタグ候補取得では `keyword` による `tag.name` 部分一致検索が使われています。
今回の `tag_keyword` は、同じ `LIKE` でも **画像一覧検索の WHERE 条件** として使う。

```text
fetch_tags_for_search:
  タグ候補一覧を絞る

search_image_files.tag_keyword:
  画像一覧を絞る
```

## 14.2 既存 tags に混ぜない

`tags` 配列に `"hair"` を入れて部分一致扱いにしてはいけない。

```text
悪い例:
tags = ["hair"]
→ hair 完全一致タグ検索なのか、hair 部分一致検索なのか曖昧

良い例:
tags = ["blue eyes"]
tag_keyword = "hair"
```

曖昧な検索条件は、あとで仕様書を読んだ人間の SAN 値を削ります。たいてい削られるのは未来の自分です。

## 14.3 検索結果全件選択などには影響しない

本機能は検索条件の追加であり、選択状態や一括操作仕様には影響しない。

```text
- 一括選択
- 一括編集
- ワイルドカード出力
- ファイル移動
- 選択画像削除
```

これらは、検索結果として表示された item に対して既存通り動作する。

## 14.4 tag_hash は使わない

`tag_hash` は全く同じタグ構成を持つ画像群を検索するための派生インデックスであり、タグ名部分一致検索には使わない。既存仕様でも `tag_hash` は `hash + tag_set` の両方で完全一致検索に使う設計です。

今回の正本は以下。

```text
tag
tag_image_link
```

`tag_hash` に寄せると、用途が違う道具を無理に使うことになります。ドライバーで味噌汁を混ぜるようなものです。できなくはないですが、やめた方がいいです。

## タグ・モデル マスタメンテナンス機能追加

## 1. 目的

画像管理上不要になったタグ・モデルを整理し、タグ名・モデル名の統廃合を一括で行えるようにする。

本機能では、以下を実現する。

```text
- 使用中タグの削除
- 使用中モデルの削除
- タグ名の置き換え / 統合
- モデル名の置き換え / 統合
- 未使用タグの一括削除
- 未使用モデルの一括削除
```

タグ操作では、`tag_image_link` の更新だけでなく、重複タグ構成検索で利用する `tag_hash` の再同期も行う。

既存構成では `ImageCatalogApi` がタグ・モデル検索候補取得や重複タグ構成取得を担当しており、`ImageFileRepository` が `tag`, `tag_image_link`, `tag_hash`, `image_model`, `image_model_to_file_data` を扱う構成であるため、本機能もこの流れに追加する。

# 5. 基本方針

## 5.1 タグとモデルは UI を共通化し、処理は分ける

UI は `mode` により共通化する。

```text
mode = "tag"
mode = "model"
```

ただし、バックエンド API はタグ用・モデル用で分ける。

```text
タグ:
- fetch_tags_for_maintenance
- delete_tag_master
- replace_tag_master
- delete_unused_tags

モデル:
- fetch_models_for_maintenance
- delete_model_master
- replace_model_master
- delete_unused_models
```

理由は、タグとモデルでは内部処理が異なるためである。

```text
タグ:
- tag_image_link 更新
- tag 削除 / 更新
- tag_hash 再同期が必要

モデル:
- image_model_to_file_data 更新
- image_model 削除 / 更新
- tag_hash 再同期は不要
```

共通 API にして `type=tag/model` で分岐すると、API 名は短くなるが中身が濁る。濁った API は後でだいたい沼になる。

## 5.2 一覧には使用件数を表示する

メンテナンス一覧には、必ず使用件数を表示する。

```text
masterpiece (1240)
best quality (1198)
blue eyes (82)
```

モデルも同様。

```text
Illustrious XL (520)
Pony Diffusion (210)
```

## 5.3 使用中でも削除可能とする

使用件数が 1 件以上あるタグ・モデルでも削除可能とする。

ただし、確認ダイアログでは影響件数を明示する。

タグ削除確認。

```text
タグ「masterpiece」を削除します。
このタグは 1240 件の画像から外されます。

削除しますか？
```

モデル削除確認。

```text
モデル「Illustrious XL」を削除します。
このモデルは 520 件の画像から解除されます。

削除しますか？
```

## 5.4 未使用マスタの一括削除を追加する

削除タブ内に、未使用マスタ一括削除ボタンを配置する。

タグ。

```text
未使用タグを一括削除
```

モデル。

```text
未使用モデルを一括削除
```

使用件数 0 件のマスタのみ削除する。

## 5.5 タグ名置き換えは既存正規化ルールを使う

タグ置き換え入力は、既存の `TagNormalizeService` を利用して正規化する。

ただし、本機能の置き換え先は **1タグのみ** とする。

正規化結果が 0 件の場合はエラー。

```text
タグ名を入力してください。
```

正規化結果が 2 件以上の場合はエラー。

```text
タグ名は1件だけ指定してください。
```

カンマ区切りで複数タグに分裂する入力は許可しない。ここで許すと「置き換え」ではなく「タグの細胞分裂」になる。顕微鏡はこのアプリには付属していない。

## 5.6 モデル名は既存仕様に合わせる

モデル名は以下で正規化する。

```text
- trim する
- 空文字はエラー
- 512文字超過はエラー
```

既存の詳細保存でも、モデル名は trim、空文字は未設定、512文字超過はエラーにする方針があるため、それに合わせる。

## 5.7 各操作は 1 トランザクションで完結する

以下の各操作は、操作単位で 1 トランザクションとする。

```text
- タグ削除
- タグ置き換え
- 未使用タグ一括削除
- モデル削除
- モデル置き換え
- 未使用モデル一括削除
```

成功時。

```text
全件反映して commit
```

失敗時。

```text
rollback
```

タグ置き換え途中で止まると、DB が「整理中の机」ではなく「引っ越し業者が帰った後の部屋」になる。これは避ける。

## 5.8 処理中は画面ロックする

本機能の変更処理中は `LoadingOverlay` で画面全体をロックする。

対象。

```text
- タグ削除
- タグ置き換え
- 未使用タグ一括削除
- モデル削除
- モデル置き換え
- 未使用モデル一括削除
```

理由。

```text
- tag_image_link 更新中に画像編集されることを防ぐ
- tag_hash 再同期中に検索されることを防ぐ
- モデルリンク更新中にモデル検索されることを防ぐ
- UI 表示と DB 状態のズレを防ぐ
```

既存構成には `LoadingOverlay.vue` と `useLoadingOverlay.js` が存在するため、新規ロック機構は作らず既存機構を利用する。

## 8.16 runMasterMaintenanceMutation

共通の更新処理。

```js
async function runMasterMaintenanceMutation({
  title,
  message,
  action,
  successMessage,
}) {
  masterMaintenanceModal.isProcessing = true

  loading.showLoading({
    title,
    message,
  })

  try {
    const result = await action()

    if (!result?.success) {
      pushToast({
        type: 'danger',
        message: result?.message || 'マスタメンテナンス処理に失敗しました。',
      })
      return
    }

    const data = result.data ?? {}

    pushToast({
      type: 'success',
      message: [
        successMessage,
        data.affectedImageCount != null ? `影響画像: ${data.affectedImageCount} 件` : null,
        data.deletedCount != null ? `削除: ${data.deletedCount} 件` : null,
      ].filter(Boolean).join(' / '),
    })

    masterMaintenanceModal.selectedItem = null
    masterMaintenanceModal.replacementName = ''

    await searchMasterMaintenanceItems()
    await refresh?.()
  } catch (error) {
    pushToast({
      type: 'danger',
      message: 'マスタメンテナンス処理に失敗しました。',
    })
  } finally {
    loading.hideLoading()
    masterMaintenanceModal.isProcessing = false
  }
}
```

```python
    data = payload or {}
```

## 15.1 delete_tag_master

処理。

```text
2. 存在しなければ ValueError
```

```python
def delete_tag_master(self, tag_id: int) -> MasterDeleteResult:
    cursor = self._connection.cursor()

    try:
        tag = self._find_tag_by_id(tag_id)
        if tag is None:
            raise ValueError("削除対象のタグが存在しません。")

        image_ids = self._find_image_ids_by_tag_id(tag_id)

        cursor.execute(
            "DELETE FROM tag_image_link WHERE tag_id = ?",
            (tag_id,),
        )

        cursor.execute(

            (tag_id,),
        )

        self._replace_tag_hashes(image_ids)

        self._connection.commit()

        return MasterDeleteResult(
            id=tag_id,
            name=tag["name"],
            affected_image_count=len(image_ids),

        )

    except Exception:
        self._connection.rollback()
        raise
```

## 15.2 tag_hash 再同期

対象画像 ID ごとに `replace_tag_hash(image_file_id)` を呼ぶ。

# 16. タグ置き換え処理

## 16.1 基本仕様

置き換え先タグ名が既存タグと衝突するかどうかで処理を分岐する。

```text
衝突しない:
- tag.name を UPDATE

衝突する:
- source tag_id のリンクを target tag_id へ統合
- source tag を DELETE
```

## 16.2 source と target が同じ場合

置き換え後タグ名が現在と同じ場合はエラーにする。

```text
同じタグ名には置き換えできません。
```

## 16.3 衝突しない場合

処理。

```text
1. source tag を取得
2. new_name の既存 tag を検索
3. target が存在しない
4. source tag.name を new_name へ UPDATE
5. source tag に紐づく image_file_id の tag_hash を再同期
6. commit
```

## 16.4 衝突する場合

例。

```text
source: best quality
target: masterpiece
```

処理。

```text
1. source tag を取得
2. target tag を取得
3. source tag に紐づく image_file_id 一覧を取得
4. source と target の両方を持つ画像について source 側リンクを DELETE
5. target を持たない画像について source_tag_id を target_tag_id へ UPDATE
6. source tag を DELETE
7. 影響画像の tag_hash を再同期
8. commit
```

重複衝突回避 SQL。

```sql
DELETE FROM tag_image_link
WHERE tag_id = :source_tag_id
  AND image_file_id IN (
    SELECT image_file_id
    FROM tag_image_link
    WHERE tag_id = :target_tag_id
  );
```

その後。

```sql
UPDATE tag_image_link
SET tag_id = :target_tag_id
WHERE tag_id = :source_tag_id;
```

## 16.5 replace_tag_master 疑似コード

```python
def replace_tag_master(self, tag_id: int, new_name: str) -> MasterReplaceResult:
    cursor = self._connection.cursor()

    try:
        source = self._find_tag_by_id(tag_id)
        if source is None:
            raise ValueError("置き換え対象のタグが存在しません。")

        if source["name"] == new_name:
            raise ValueError("同じタグ名には置き換えできません。")

        image_ids = self._find_image_ids_by_tag_id(tag_id)
        target = self._find_tag_by_name(new_name)

        if target is None:
            cursor.execute(
                "UPDATE tag SET name = ? WHERE id = ?",
                (new_name, tag_id),
            )

            self._replace_tag_hashes(image_ids)

            self._connection.commit()

            return MasterReplaceResult(
                source_id=tag_id,
                source_name=source["name"],
                target_id=tag_id,
                target_name=new_name,
                affected_image_count=len(image_ids),
                merged=False,
            )

        target_id = target["id"]

        cursor.execute(
            """
            DELETE FROM tag_image_link
            WHERE tag_id = ?
              AND image_file_id IN (
                SELECT image_file_id
                FROM tag_image_link
                WHERE tag_id = ?
              )
            """,
            (tag_id, target_id),
        )

        cursor.execute(
            """
            UPDATE tag_image_link
            SET tag_id = ?
            WHERE tag_id = ?
            """,
            (target_id, tag_id),
        )

        cursor.execute(
            "DELETE FROM tag WHERE id = ?",
            (tag_id,),
        )

        self._replace_tag_hashes(image_ids)

        self._connection.commit()

        return MasterReplaceResult(
            source_id=tag_id,
            source_name=source["name"],
            target_id=target_id,
            target_name=target["name"],
            affected_image_count=len(image_ids),
            merged=True,
        )

    except Exception:
        self._connection.rollback()
        raise
```

# 26. 注意事項

## 26.1 タグ操作では tag_hash 更新を忘れない

タグ削除・タグ置き換えでは、必ず影響画像の `tag_hash` を再同期する。

忘れると、タグ一覧や画像詳細は新しい状態なのに、重複タグ構成検索だけ古い状態を見ることになる。DB内に過去の亡霊を飼う趣味がないなら、更新必須。

## 26.2 モデル操作では tag_hash を更新しない

モデルはタグ構成ではないため、`tag_hash` 更新は不要。

やる必要のない更新を混ぜると、処理が遅くなるだけでなく、後で「なぜモデル更新でタグハッシュが動くのか」という考古学が始まる。

## 26.3 タグ置き換え時の重複リンク衝突に注意

source タグと target タグの両方を持つ画像が存在する。

この場合、単純な UPDATE は主キー衝突を起こす可能性があるため、先に重複する source リンクを DELETE してから UPDATE する。

## 26.4 削除確認文言を弱くしない

必ず影響件数を表示する。

避ける。

```text
削除しますか？
```

採用する。

```text
タグ「xxx」を削除します。
このタグは 123 件の画像から外されます。
実行しますか？
```

削除確認で遠慮すると、削除後にユーザーが遠い目になる。アプリの責任ではないが、UIの責任ではある。

## 26.5 「未使用一括削除」は使用中マスタを消さない

未使用一括削除は、使用件数 0 件のみ対象とする。

使用中タグ・使用中モデルをまとめて消す機能は今回含めない。

## 26.6 処理中はモーダルを閉じさせない

処理中にモーダルを閉じられると、完了後の再検索や toast の見え方が不安定になる。

`isProcessing` 中は以下を disabled にする。

```text
- モーダルの閉じるボタン
- タブ切り替え
- 検索
- 削除
- 置き換え保存
- 未使用一括削除
```

## 26.7 検索件数は 50 件固定でよい

初回実装ではページング不要。

```text
一度に 50 件まで表示
ID 昇順
```

大量タグがある場合でも、部分一致検索で絞り込む運用とする。
ここでページングまで入れると、マスタメンテナンスが急に市役所の窓口システムになる。今回はそこまで育てない。
