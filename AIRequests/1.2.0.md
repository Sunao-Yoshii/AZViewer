# 実装仕様書

## タグ部分一致検索機能追加

## 1. 目的

画像一覧画面において、登録済みタグ名に含まれる文字列を指定し、その文字列を含むタグが紐づいている画像を検索できるようにする。

既存のタグ検索は、選択済みタグ名を完全一致 AND 条件として扱う。

今回追加する機能は、それとは別に、タグ名の部分一致条件を検索フォームから指定するものである。

```text
既存:
tags = ["blue eyes", "smile"]
→ blue eyes と smile の両方を持つ画像を検索

今回:
tag_keyword = "hair"
→ hair を含むタグを1つ以上持つ画像を検索
```

## 2. 基本方針

## 2.1 検索条件として `tag_keyword` を追加する

既存の `search_image_files` payload に、タグ部分一致検索用の `tag_keyword` を追加する。

```json
{
  "path": "",
  "rating": null,
  "is_checked": null,
  "is_favorite": null,
  "tags": ["blue eyes", "smile"],
  "folder": "set01",
  "model": "waiNSFWIllustrious_v50",
  "tag_hash": null,
  "tag_set": null,
  "tag_keyword": "hair",
  "page": 1,
  "page_size": 25,
  "sort": "id_desc"
}
```

## 2.2 `tags` とは別条件として扱う

`tags` と `tag_keyword` は意味が異なるため、混在させない。

```text
tags:
  タグ完全一致 AND 検索

tag_keyword:
  タグ名部分一致検索
```

両方指定された場合は、既存条件と同様に AND 条件で組み合わせる。

```text
tags = ["blue eyes"]
tag_keyword = "hair"

→ blue eyes を持ち、
  かつ hair を含むタグを1つ以上持つ画像
```

## 2.3 入力が空の場合は条件に含めない

以下の場合、タグ部分一致条件は検索 SQL に追加しない。

```text
- null
- undefined
- 空文字
- trim 後に空文字
```

## 2.4 SQL は `EXISTS` サブクエリを使う

二段階 SQL ではなく、既存の検索 SQL に `EXISTS` 条件を追加する。

理由は、既存の画像検索が複数条件を `ImageFileRepository.search(...)` に集約しており、タグ部分一致だけ別経路にすると検索条件の合成が崩れるためである。

```sql
EXISTS (
    SELECT 1
    FROM tag_image_link til
    INNER JOIN tag t
        ON t.id = til.tag_id
    WHERE til.image_file_id = i.id
      AND t.name LIKE :tag_keyword_like ESCAPE '\'
)
```

ここで二段階 SQL にすると、ページング、ソート、既存条件、`total_count` の整合性が面倒になります。検索処理が二重帳簿になります。だいたい税務調査、じゃなくてバグ調査が来ます。

## 8.3 `_append_tag_keyword_condition(...)` 追加

```python
def _append_tag_keyword_condition(
    self,
    where_clauses: list[str],
    params: dict,
    tag_keyword: str | None,
) -> None:
    keyword = str(tag_keyword or "").strip().lower()

    if not keyword:
        return

    where_clauses.append("""
        EXISTS (
            SELECT 1
            FROM tag_image_link til
            INNER JOIN tag t
                ON t.id = til.tag_id
            WHERE til.image_file_id = i.id
              AND t.name LIKE :tag_keyword_like ESCAPE '\\'
        )
    """)

    params["tag_keyword_like"] = f"%{self._escape_like(keyword)}%"
```

## 8.4 `_escape_like(...)` 追加

`%`, `_`, `\` をリテラルとして検索できるようにする。

```python
def _escape_like(self, value: str) -> str:
    return (
        value
        .replace("\\", "\\\\")
        .replace("%", "\\%")
        .replace("_", "\\_")
    )
```

### 理由

SQLite の `LIKE` では `%` と `_` がワイルドカードとして扱われる。

今回の仕様は「入力文字列を含むタグ名を検索」であり、ユーザーが `%` を入力したときに全件一致のような挙動になるのは不自然である。

## 8.5 SQL 方針

タグ部分一致条件は、以下の SQL を WHERE 句へ追加する。

```sql
EXISTS (
    SELECT 1
    FROM tag_image_link til
    INNER JOIN tag t
        ON t.id = til.tag_id
    WHERE til.image_file_id = i.id
      AND t.name LIKE :tag_keyword_like ESCAPE '\'
)
```

## 8.6 total_count への反映

`items` 取得 SQL と `total_count` 取得 SQL の両方に、同じ WHERE 条件を使う。

既存のタグ検索でも、一覧取得 SQL だけでなく `total_count` 算出 SQL に同じタグ条件を適用する方針です。
今回も同じです。

```text
- items 取得 SQL
- total_count 取得 SQL
```

両方に `tag_keyword` 条件を反映する。

## 8.7 既存タグ AND 条件との併用

既存のタグ AND 条件は変更しない。

既存のタグ AND 条件は、指定されたタグすべてを持つ画像を `GROUP BY` と `HAVING COUNT(DISTINCT t.name)` で絞り込む仕様です。

`tag_keyword` は追加の `EXISTS` 条件として扱う。

```sql
WHERE
    ...
    AND i.id IN (
        SELECT
            til.image_file_id
        FROM tag_image_link til
        INNER JOIN tag t
            ON t.id = til.tag_id
        WHERE t.name IN (:tag_0, :tag_1)
        GROUP BY til.image_file_id
        HAVING COUNT(DISTINCT t.name) = :tag_count
    )
    AND EXISTS (
        SELECT 1
        FROM tag_image_link til_kw
        INNER JOIN tag t_kw
            ON t_kw.id = til_kw.tag_id
        WHERE til_kw.image_file_id = i.id
          AND t_kw.name LIKE :tag_keyword_like ESCAPE '\'
    )
```

実装上は alias の衝突を避けるため、`til_kw`, `t_kw` のように別名にすることを推奨する。

## 14. 注意事項

## 14.1 既存のタグ候補検索とは別物

既存の `fetch_tags_for_search` は、タグ検索ダイアログに表示する候補を取得する API であり、画像一覧そのものを絞り込む API ではない。

既存のタグ候補取得では `keyword` による `tag.name` 部分一致検索が使われています。
今回の `tag_keyword` は、同じ `LIKE` でも **画像一覧検索の WHERE 条件** として使う。

```text
fetch_tags_for_search:
  タグ候補一覧を絞る

search_image_files.tag_keyword:
  画像一覧を絞る
```

## 14.2 既存 tags に混ぜない

`tags` 配列に `"hair"` を入れて部分一致扱いにしてはいけない。

```text
悪い例:
tags = ["hair"]
→ hair 完全一致タグ検索なのか、hair 部分一致検索なのか曖昧

良い例:
tags = ["blue eyes"]
tag_keyword = "hair"
```

曖昧な検索条件は、あとで仕様書を読んだ人間の SAN 値を削ります。たいてい削られるのは未来の自分です。

## 14.3 検索結果全件選択などには影響しない

本機能は検索条件の追加であり、選択状態や一括操作仕様には影響しない。

```text
- 一括選択
- 一括編集
- ワイルドカード出力
- ファイル移動
- 選択画像削除
```

これらは、検索結果として表示された item に対して既存通り動作する。

## 14.4 tag_hash は使わない

`tag_hash` は全く同じタグ構成を持つ画像群を検索するための派生インデックスであり、タグ名部分一致検索には使わない。既存仕様でも `tag_hash` は `hash + tag_set` の両方で完全一致検索に使う設計です。

今回の正本は以下。

```text
tag
tag_image_link
```

`tag_hash` に寄せると、用途が違う道具を無理に使うことになります。ドライバーで味噌汁を混ぜるようなものです。できなくはないですが、やめた方がいいです。
