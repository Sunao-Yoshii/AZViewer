# 実装設計書

## 画像選択・一括物理削除機能追加

## 1. 目的

画像一覧画面で複数画像を選択し、選択した画像について以下を一括削除できるようにする。

- AZViewer 上の画像レコード
- サムネイルキャッシュ
- 実際の画像ファイル

本機能により、AZViewer を単なる画像ビューワーではなく、画像管理ツールとして利用できるようにする。

ただし、物理削除は取り消しできない破壊的操作であるため、既存の単体レコード削除とは明確に区別する。

## 2. 前提

既存の `delete_image_file` は、画像レコードとサムネイル削除を担当する API であり、元画像ファイルの物理削除までは担当していない。

また、既存の `ThumbnailCacheService` には複数サムネイル削除用の `delete_thumbnails(record_ids)` が存在するため、本機能でも再利用する。

`Content.vue` は画像一覧、ソート、ページサイズ、総件数、ページング、タイル表示を担当しているため、画像選択状態の管理・中継対象として自然である。

## 3. 変更概要

以下の機能を追加する。

- tile.vue のタイトル左側に選択チェックボックスを追加
- 選択状態を UI 一時状態として保持
- AppHeader.vue に「選択画像を削除」ボタンを追加
- 選択件数 0 件では削除ボタンを disabled
- 削除確認ダイアログを追加
- 選択画像の実ファイル、サムネイル、DB レコードを一括削除
- 削除後に現在検索条件で再検索
- 再検索・ページ変更・ソート変更・ページサイズ変更時は選択状態をクリア

## 5. 基本方針

## 5.1 既存単体削除とは分離する

既存の単体削除 API は維持する。

```text
delete_image_file
```

これは従来通り、AZViewer 上の登録削除を主目的とする。

本機能では、実ファイルの物理削除を含む新 API を追加する。

```text
delete_image_files_with_physical_files
```

### 理由

物理削除を既存 `delete_image_file` に混ぜると、既存操作の意味が変わる。

```text
従来:
アプリ上から削除するが、実ファイルは残る

今回:
アプリ上から削除し、実ファイルも削除する
```

この2つは操作リスクが大きく異なるため、API・UI・確認文言のすべてで分離する。

削除ボタンは名前で殴るくらいが安全です。短くて曖昧な削除 API は、だいたい後で人間を削ります。

## 5.2 選択状態は DB に保存しない

画像選択状態は、現在表示中の一覧に対する一時的な UI 状態とする。

```text
DB カラムは追加しない
image_file_data に選択状態を持たせない
```

### 理由

選択状態は永続データではない。

```text
- 現在の画面で選択されているか
- 一括削除対象として一時的に選ばれているか
```

だけを表すため、DB 保存すると不要な状態管理が増える。

## 5.3 見えていない選択は保持しない

以下の操作時、選択状態をクリアする。

```text
- 再検索
- ページ変更
- ページサイズ変更
- ソート変更
- 削除完了後
```

### 理由

画面に見えていない画像が選択されたまま残ると、ユーザーが削除対象を把握できない。

```text
見えていないけど選択済み
```

は、物理削除 UI として危険すぎる。地雷に「地雷」と書いてあるだけ、まだ親切です。

## 8.10 実装例

```js
import { deleteImageFilesWithPhysicalFiles } from '../services/backendApi'

async function deleteSelectedImages({ ids, refresh, clearSelection }) {
  const targetIds = [...(ids ?? [])]

  if (targetIds.length === 0) {
    return
  }

  const confirmed = window.confirm(
    [
      `選択した ${targetIds.length} 件の画像を削除します。`,
      '',
      'この操作では、AZViewer 上の登録情報だけでなく、実際の画像ファイルもディスクから削除されます。',
      'この操作は元に戻せません。',
      '',
      '削除しますか？',
    ].join('\n')
  )

  if (!confirmed) {
    return
  }

  loading.showLoading({
    title: '選択画像を削除中',
    message: '実ファイル、サムネイル、登録情報を削除しています...',
  })

  try {
    const result = await deleteImageFilesWithPhysicalFiles({
      ids: targetIds,
    })

    if (!result?.success) {
      pushToast({
        type: 'danger',
        message: result?.message || '選択画像の削除に失敗しました。',
      })
      return
    }

    const data = result.data ?? {}
    const failedCount = data.failedCount ?? 0

    if (failedCount > 0) {
      pushToast({
        type: 'warning',
        message: `一部画像の削除に失敗しました。対象: ${data.targetCount ?? targetIds.length} 件 / 削除: ${data.deletedRecordCount ?? 0} 件 / 失敗: ${failedCount} 件`,
      })
    } else {
      pushToast({
        type: 'success',
        message: `選択画像の削除が完了しました。対象: ${data.targetCount ?? targetIds.length} 件 / 削除: ${data.deletedRecordCount ?? 0} 件`,
      })
    }

    clearSelection?.()
    await refresh?.()
  } catch (error) {
    pushToast({
      type: 'danger',
      message: '選択画像の削除に失敗しました。',
    })
  } finally {
    loading.hideLoading()
  }
}
```

```python
    deleted_file_count: int
    deleted_thumbnail_count: int
    deleted_record_count: int
    missing_file_count: int
    failed_count: int

        }
```

## 13. 物理削除処理設計

## 13.1 削除順序

削除順序は以下とする。

```text
1. id 配列を正規化する
2. DB から対象レコードを取得する
3. 1件ずつ実ファイル削除を試行する
4. 実ファイル削除に成功、または実ファイルが存在しない ID を DB 削除候補に入れる
5. DB 削除候補のサムネイルを削除する
6. DB 削除候補のレコードを削除する
7. 集計結果を返す
```

## 13.2 ファイル削除と DB 削除の関係

```text
ファイル削除成功
→ サムネイル削除
→ DB レコード削除

ファイルが既に存在しない
→ サムネイル削除
→ DB レコード削除

ファイル削除失敗
→ サムネイル削除しない
→ DB レコード削除しない
→ failedFiles に追加
```

### 理由

ファイル削除に失敗したのに DB レコードを削除すると、実ファイルは残っているが AZViewer 上から見えなくなる。

これは「管理ツール」として中途半端で、ユーザーから見るとファイルが忍者になります。存在するのに見えない。困るタイプの忍者です。

## 13.3 実ファイルが存在しない場合

実ファイルが存在しない場合は、失敗扱いにしない。

```text
missingFileCount += 1
DB レコード削除対象に含める
サムネイル削除対象に含める
```

### 理由

起動時クリーンアップでも、DB 上に存在するが実ファイルが存在しないレコードは削除する設計である。

一括削除時も同じ整合性回復として扱う。

## 13.4 サムネイル削除失敗

サムネイル削除に失敗しても、DB レコード削除は継続する。

### 理由

サムネイルは派生キャッシュであり、元画像や DB レコードより重要度が低い。

```text
サムネイル削除失敗
→ ログ出力
→ 可能であれば failedFiles には含めない
→ DB レコード削除は継続
```

ただし、削除件数には反映する。

```text
deletedThumbnailCount
```

は実際に削除できた件数を入れる。

## 13.5 ファイル削除方法

Python 側では `Path.unlink()` を使用する。

```python
from pathlib import Path

image_path = Path(item.path)

if image_path.exists():
    image_path.unlink()
else:
    missing_file_count += 1
```

```python
if image_path.exists() and not image_path.is_file():
```

## 14. ImageCatalogApi 実装方針

## 14.1 正規化メソッド

```python
def _normalize_delete_ids(self, value) -> list[int]:
    if not isinstance(value, list):
        return []

    ids: list[int] = []

    for item in value:
        try:
            record_id = int(item)
        except (TypeError, ValueError):
            continue

        if record_id <= 0:
            continue

        if record_id in ids:
            continue

        ids.append(record_id)

    return ids
```

## 14.3 delete_image_files_with_physical_files

実装イメージ。

```python
def delete_image_files_with_physical_files(self, payload):
    try:
        data = payload or {}
        record_ids = self._normalize_delete_ids(data.get("ids"))

        if not record_ids:
            return ApiResponse(
                success=False,
                message="削除対象が指定されていません。",
                data=self._empty_physical_delete_data(),
            ).to_dict()

        repository = self._get_repository()
        items = repository.find_by_ids(record_ids)

        result = self._delete_physical_files_and_records(
            repository=repository,
            items=items,
        )

        message = (
            "一部画像の削除に失敗しました。"
            if result.failed_count > 0
            else "選択画像の削除が完了しました。"
        )

        return ApiResponse(
            success=True,
            message=message,
            data=result.to_api_data(),
        ).to_dict()

    except Exception:
        return ApiResponse(
            success=False,
            message="選択画像の削除に失敗しました。",
            data=self._empty_physical_delete_data(),
        ).to_dict()
```

## 14.4 内部削除処理

```python
def _delete_physical_files_and_records(
    self,
    repository: ImageFileRepository,
    items: list[ImageFileListItem],
) -> PhysicalDeleteResult:
    deletable_ids: list[int] = []
    failed_files: list[PhysicalDeleteFailure] = []

    deleted_file_count = 0
    missing_file_count = 0

    for item in items:
        try:
            image_path = Path(item.path)

            if image_path.exists():
                if not image_path.is_file():
                    raise ValueError("画像ファイルではありません。")

                image_path.unlink()
                deleted_file_count += 1
            else:
                missing_file_count += 1

            deletable_ids.append(item.id)

        except Exception as error:
            if len(failed_files) < 20:
                failed_files.append(
                    PhysicalDeleteFailure(
                        id=item.id,
                        path=item.path,
                        reason="ファイルを削除できませんでした。",
                    )
                )

    deleted_thumbnail_count = 0

    if deletable_ids:
        deleted_thumbnail_count = self._thumbnail_service.delete_thumbnails(
            deletable_ids
        )
        deleted_record_count = repository.delete_by_ids(deletable_ids)
    else:
        deleted_record_count = 0

    return PhysicalDeleteResult(
        target_count=len(items),
        deleted_file_count=deleted_file_count,
        deleted_thumbnail_count=deleted_thumbnail_count,
        deleted_record_count=deleted_record_count,
        missing_file_count=missing_file_count,
        failed_count=len(items) - len(deletable_ids),
        failed_files=failed_files,
    )
```

### 注意

既存の `ThumbnailCacheService.delete_thumbnails(record_ids)` が戻り値を返さない実装の場合は、削除件数を返すように変更するか、`deletedThumbnailCount` を省略してもよい。

推奨は、件数集計のために戻り値を返す形へ変更する。

```python
def delete_thumbnails(self, record_ids: list[int]) -> int:
    deleted_count = 0
    for record_id in record_ids:
        if self.delete_thumbnail(record_id):
            deleted_count += 1
    return deleted_count
```

その場合、`delete_thumbnail(record_id)` も bool を返すようにする。

## 21. 注意点

## 21.1 物理削除は戻せない

本機能はファイルをディスクから削除する。

OS のゴミ箱へ移動する仕様ではない。

```text
Path.unlink()
```

を使う場合、通常は即時削除になる。

必要になった場合は将来仕様として「ゴミ箱へ移動」を検討する。

## 21.2 DB とファイル削除は完全な原子性を持たない

DB トランザクションで戻せるのは DB 操作のみである。

```text
ファイル削除成功
DB 削除失敗
```

という状態は理論上発生する。

このリスクを完全には消せないため、処理順序・エラーハンドリング・ログで軽減する。

## 21.3 単体削除の意味を変えない

既存の `delete_image_file` に物理削除を混ぜないこと。

単体削除と一括物理削除の意味が混ざると、UI 上も API 上も事故りやすい。

## 21.4 削除対象は必ず ID から再取得する

フロントエンドから `path` を直接渡して削除しない。

```text
フロント: id のみ送信
バックエンド: id から path を取得
```

とする。

理由は、削除対象を DB 上の登録レコードに限定するため。

## 21.5 確認文言は弱くしない

確認文言では必ず以下を明示する。

```text
- 登録情報だけではない
- 実ファイルも削除される
- 元に戻せない
```

削除確認で遠慮すると、だいたい削除後にユーザーが遠い目になります。

## 5. 基本方針

## 5.1 選択状態は既存仕様を流用する

本機能のために新しい DB カラムは追加しない。

```text
image_file_data に選択状態を持たせない
```

選択状態は既存の `selectedImageIds` / `selectedImageIdSet` をそのまま利用する。

## 5.2 物理ファイル移動と DB 更新は成功分のみ反映する

ファイル移動に失敗した画像については、DB を更新しない。

```text
ファイル移動成功
→ DB path / folder 更新対象

ファイル移動失敗
→ DB 更新しない
→ failedFiles に追加
```

## 5.3 同名ファイルは上書きしない

移動先に同名ファイルが既に存在する場合、その画像は移動失敗扱いにする。

```text
old: C:/old/sample.png
new: D:/new/sample.png

D:/new/sample.png が存在する場合
→ 移動しない
```

自動リネームは行わない。

```text
sample_1.png
sample_2.png
```

のようなリネームは便利だが、今回の仕様には含めない。親切が勝手にファイル名を変え始めると、だいたい後で事情聴取になります。

## 5.4 DB 更新失敗時は補償移動を試みる

実ファイルを移動した後に DB 更新へ失敗した場合、可能であればファイルを元の場所へ戻す。

```text
old_path -> new_path 移動成功
DB 更新失敗
new_path -> old_path へ戻す
```

戻しにも失敗した場合は、失敗詳細へ明示する。

## 5.5 サムネイルキャッシュは削除・再生成しない

サムネイルキャッシュが `id` ベースで `data/thumbs/{id}.png` として管理されている場合、画像ファイルの移動ではサムネイルを変更しない。

理由は、画像内容と `id` が変わらないため。

```js
import {
  moveImageFilesToFolder,
  selectFolderDialog,
} from '../services/backendApi'

function buildMoveConfirmMessage(count, destinationFolder) {
  return [
    `選択した ${count} 件の画像ファイルを移動します。`,
    '',
    '移動先:',
    destinationFolder,
    '',
    '移動後は AZViewer 上の登録パスも更新されます。',
    '実行しますか？',
  ].join('\n')
}

function extractSelectedFolderPath(result) {
  const items = result?.data?.items ?? []
  const folderItem = items.find((item) => item?.type === 'directory')
  return folderItem?.path || ''
}

function pushMoveResultToast(pushToast, result, targetCount) {
  const data = result?.data ?? {}
  const failedCount = data.failedCount ?? 0
  const movedCount = data.movedCount ?? 0
  const skippedCount = data.skippedCount ?? 0

  if (failedCount > 0) {
    pushToast({
      type: 'warning',
      message: `一部画像ファイルの移動に失敗しました。対象: ${data.targetCount ?? targetCount} 件 / 移動: ${movedCount} 件 / スキップ: ${skippedCount} 件 / 失敗: ${failedCount} 件`,
    })
    return
  }

  pushToast({
    type: 'success',
    message: `画像ファイルの移動が完了しました。対象: ${data.targetCount ?? targetCount} 件 / 移動: ${movedCount} 件 / スキップ: ${skippedCount} 件`,
  })
}

async function moveSelectedImages({ ids, refresh, clearSelection }) {
  const targetIds = [...(ids ?? [])]

  if (targetIds.length === 0) {
    return
  }

  const folderResult = await selectFolderDialog()

  if (!folderResult?.success) {
    return
  }

  const destinationFolder = extractSelectedFolderPath(folderResult)

  if (!destinationFolder) {
    return
  }

  const confirmed = window.confirm(
    buildMoveConfirmMessage(targetIds.length, destinationFolder)
  )

  if (!confirmed) {
    return
  }

  loading.showLoading({
    title: '画像ファイルを移動中',
    message: '実ファイルの移動と登録情報の更新を行っています...',
  })

  try {
    const result = await moveImageFilesToFolder({
      ids: targetIds,
      destinationFolder,
    })

    if (!result?.success) {
      pushToast({
        type: 'danger',
        message: result?.message || '画像ファイルの移動に失敗しました。',
      })
      return
    }

    pushMoveResultToast(pushToast, result, targetIds.length)

    clearSelection?.()
    await refresh?.()
  } catch (error) {
    pushToast({
      type: 'danger',
      message: '画像ファイルの移動に失敗しました。',
    })
  } finally {
    loading.hideLoading()
  }
}
```

## 12. バックエンド処理設計

## 12.1 処理順序

```text
1. payload から ids / destinationFolder を取得する
2. ids を正規化する
3. destinationFolder を絶対パス化して検証する
4. DB から対象レコードを取得する
5. 各画像について移動前検証を行う
6. 移動可能な画像だけ実ファイルを移動する
7. 移動成功分の path / folder を DB 更新する
8. DB 更新失敗時は可能な範囲で元パスへ戻す
9. 集計 DTO を返す
```

## 12.2 ID 正規化

一括削除の `_normalize_delete_ids` と同等の処理を追加する。

```python
def _normalize_move_ids(self, value) -> list[int]:
    if not isinstance(value, list):
        return []

    ids: list[int] = []

    for item in value:
        try:
            record_id = int(item)
        except (TypeError, ValueError):
            continue

        if record_id <= 0:
            continue

        if record_id in ids:
            continue

        ids.append(record_id)

    return ids
```

共通化するなら `_normalize_record_ids` としてもよい。

```python
def _normalize_destination_folder(self, value) -> Path:
    folder_text = str(value or "").strip()

    if not folder_text:
        raise ValueError("移動先フォルダが指定されていません。")

    folder_path = Path(folder_text).expanduser().resolve()

    if not folder_path.exists():
        raise ValueError("移動先フォルダが存在しません。")

    if not folder_path.is_dir():
        raise ValueError("移動先がフォルダではありません。")

    return folder_path
```

## 13. ImageCatalogApi 実装方針

## 13.1 move_image_files_to_folder

```python
def move_image_files_to_folder(self, payload):
    try:
        data = payload or {}

        record_ids = self._normalize_move_ids(data.get("ids"))
        if not record_ids:
            return ApiResponse(
                success=False,
                message="移動対象が指定されていません。",
                data=self._empty_file_move_data(),
            ).to_dict()

        try:
            destination_folder = self._normalize_destination_folder(
                data.get("destinationFolder")
            )
        except ValueError as error:
            return ApiResponse(
                success=False,
                message=str(error),
                data=self._empty_file_move_data(),
            ).to_dict()

        repository = self._get_repository()
        items = repository.find_by_ids(record_ids)

        result = self._move_files_and_update_records(
            repository=repository,
            items=items,
            destination_folder=destination_folder,
        )

        message = (
            "一部画像ファイルの移動に失敗しました。"
            if result.failed_count > 0
            else "画像ファイルの移動が完了しました。"
        )

        return ApiResponse(
            success=True,
            message=message,
            data=result.to_api_data(),
        ).to_dict()

    except Exception:
        return ApiResponse(
            success=False,
            message="画像ファイルの移動に失敗しました。",
            data=self._empty_file_move_data(),
        ).to_dict()
```

## 13.2 内部移動処理

```python
def _move_files_and_update_records(
    self,
    repository: ImageFileRepository,
    items: list[ImageFileListItem],
    destination_folder: Path,
) -> FileMoveResult:
    moved_items: list[dict] = []
    skipped_count = 0
    failed_files: list[FileMoveFailure] = []

    for item in items:
        try:
            source_path = Path(item.path)

            if not source_path.exists():
                raise FileNotFoundError("移動元ファイルが存在しません。")

            if not source_path.is_file():
                raise ValueError("画像ファイルではありません。")

            destination_path = destination_folder / source_path.name

            if source_path.resolve() == destination_path.resolve():
                skipped_count += 1
                continue

            if destination_path.exists():
                raise FileExistsError("移動先に同名ファイルが既に存在します。")

            if repository.exists_by_path(str(destination_path)):
                raise ValueError("移動後パスは既に登録済みです。")

            shutil.move(str(source_path), str(destination_path))

            moved_items.append({
                "id": item.id,
                "old_path": str(source_path),
                "new_path": str(destination_path),
                "folder": destination_path.parent.name,
            })

        except Exception as error:
            if len(failed_files) < 20:
                failed_files.append(
                    FileMoveFailure(
                        id=item.id,
                        path=item.path,
                        reason=self._to_file_move_failure_reason(error),
                    )
                )

    if moved_items:
        try:
            updates = [
                {
                    "id": item["id"],
                    "path": item["new_path"],
                    "folder": item["folder"],
                }
                for item in moved_items
            ]
            repository.update_paths(updates)

        except Exception:
            self._restore_moved_files(moved_items, failed_files)
            moved_items = []

    return FileMoveResult(
        target_count=len(items),
        moved_count=len(moved_items),
        skipped_count=skipped_count,
        failed_count=len(failed_files),
        failed_files=failed_files,
    )
```

## 13.3 失敗理由変換

```python
def _to_file_move_failure_reason(self, error: Exception) -> str:
    if isinstance(error, FileNotFoundError):
        return "移動元ファイルが存在しません。"

    if isinstance(error, FileExistsError):
        return "移動先に同名ファイルが既に存在します。"

    if isinstance(error, PermissionError):
        return "ファイルへアクセスできません。別のアプリで使用中の可能性があります。"

    if isinstance(error, ValueError):
        return str(error)

    return "ファイルを移動できませんでした。"
```

## 13.4 DB 更新失敗時の復元

```python
def _restore_moved_files(
    self,
    moved_items: list[dict],
    failed_files: list[FileMoveFailure],
) -> None:
    for item in moved_items:
        try:
            new_path = Path(item["new_path"])
            old_path = Path(item["old_path"])

            if new_path.exists() and not old_path.exists():
                shutil.move(str(new_path), str(old_path))

            if len(failed_files) < 20:
                failed_files.append(
                    FileMoveFailure(
                        id=item["id"],
                        path=item["old_path"],
                        reason="DB 更新に失敗したため、ファイル移動を取り消しました。",
                    )
                )

        except Exception:
            if len(failed_files) < 20:
                failed_files.append(
                    FileMoveFailure(
                        id=item["id"],
                        path=item["new_path"],
                        reason="DB 更新に失敗し、移動済みファイルの復元にも失敗しました。",
                    )
                )
```

このケースはかなり嫌な状態なので、ログ出力も推奨。

```text
フォルダ選択ダイアログ表示中は LoadingOverlay を表示しないこと
```

## 17.4 ファイル移動

```text
選択画像の実ファイルが指定フォルダへ移動されること
移動先に同名ファイルがある場合は上書きしないこと
自動リネームしないこと
移動元ファイルが存在しない場合は失敗扱いになること
ファイルロック等で移動できない場合は失敗扱いになること
```

## 17.5 DB 更新

```text
移動成功分のみ image_file_data.path が移動後パスへ更新されること
移動成功分のみ image_file_data.folder が移動後の直接親フォルダ名へ更新されること
移動失敗分の DB は更新されないこと
同一フォルダへの移動は skippedCount として扱われること
```

## 17.6 再検索・選択状態

```text
移動完了後に選択状態がクリアされること
移動完了後に現在検索条件で再検索されること
フォルダ検索条件に一致しなくなった画像は一覧から消えること
```

## 17.7 API 結果

```text
targetCount / movedCount / skippedCount / failedCount が返ること
failedFiles は最大20件まで返ること
一部失敗時も成功分は反映されること
```

## 18. 実装上の注意点

ファイル移動は DB 更新より厄介です。DB はトランザクションで戻せますが、ファイルシステムは「まあ頑張って戻してね」という温度感です。

特に注意すべき点は以下です。

```text
- 同名ファイル上書き禁止
- DB 更新失敗時の補償移動
- ファイルロック時の失敗扱い
- 移動先が現在検索条件から外れる可能性
- サムネイルキャッシュは id ベースなので維持
```

この設計なら、既存の一括物理削除機能の UI・状態管理を流用しつつ、物理ファイル移動と DB 整合性のリスクをある程度抑えられます。

# 実装仕様書

## 重複タグ構成検索機能追加

## 1. 目的

登録済み画像のうち、**全く同じタグ構成を持つ画像群**を検出し、対象画像を一覧検索できるようにする。

本機能では、画像ごとのタグ構成を `tag_hash` テーブルに保持し、タグ構成が重複している候補を高速に取得する。

既存のタグ管理は `tag` / `tag_image_link` を利用しており、検索では `tag_image_link` と `tag` によるタグ AND 条件が使われている。既存検索では、タグ条件を `image_file_data.id IN (...)` のサブクエリで絞り込み、`total_count` 算出にも同じ条件を適用する設計である。

## 2. 基本方針

### 2.1 `tag_hash` は派生インデックスとして扱う

`tag_hash` は正規のタグ情報ではなく、`tag` / `tag_image_link` から生成される検索補助テーブルとする。

正本はあくまで以下とする。

```text
tag
tag_image_link
```

`tag_hash` は、重複タグ構成検索を高速化するための派生データである。

### 2.2 同一性判定は `tag_set` を基準にする

タグ構成の完全一致判定は、タグ名文字列ではなく、タグ ID の集合で行う。

```text
tag_set = tag.id を昇順にカンマ結合した文字列
hash = MD5(tag_set)
```

`tag_names` はダイアログ表示用とする。

```text
tag_names = tag.id 昇順に対応する tag.name をカンマ結合した文字列
```

### 2.3 `hash` 単独ではなく `hash + tag_set` で検索する

MD5 の衝突可能性は実用上かなり低いが、ゼロではない。

そのため、画像検索時は以下の条件で絞り込む。

```text
hash
tag_set
```

`hash` はインデックス利用のため、`tag_set` は同一性の最終確認のために使う。

### 2.4 タグ0件画像は `tag_hash` に登録しない

タグが1件もない画像は、重複タグ構成検索の対象外とする。

```text
タグ0件
→ tag_hash レコードなし
```

タグなし画像を `tag_set = ''` として登録すると、タグ未登録画像が大量に重複候補として表示されるため、本機能の目的から外れる。

### 2.5 `tag_hash` 更新は Repository に閉じ込める

タグ保存経路は複数存在する。

現行構成では `ImageCatalogApi` が画像検索、詳細更新、削除、タグ/フォルダ候補取得、プロンプト一括タグ登録などを担当している。
また、`ImageCatalogApi` には詳細とタグを同一トランザクションで更新する `_update_detail_with_tags(detail, tags)` が存在する。

そのため、`tag_hash` 更新を UI 側や個別 API 側に分散させず、`ImageFileRepository` に集約する。

## 3. 変更概要

以下を追加・変更する。

```text
- tag_hash テーブル追加
- tag_hash 作成・更新・削除・再構築処理追加
- タグ保存時に tag_hash を同期更新
- プロンプト一括タグ登録時に tag_hash を同期更新
- 画像削除時に tag_hash を削除
- tag_hash 件数が 0 の場合のみ、起動時に全件再構築
- 重複タグ構成取得 API 追加
- hash + tag_set による画像検索条件追加
- AppHeader.vue に「重複タグ構成の検索」ボタン追加
- 重複タグ構成選択モーダル追加
- 通常検索フォームから検索した場合は hash 検索条件を解除
```

## 7.2 `replace_tag_hash(image_file_id)` 追加

### 役割

指定画像の現在のタグ構成を読み取り、`tag_hash` を更新する。

```python
def replace_tag_hash(self, image_file_id: int) -> None:
    ...
```

### 処理内容

```text
1. image_file_id に紐づく tag を tag.id 昇順で取得
2. タグが0件なら tag_hash から該当 image_file_data_id を DELETE
3. tag_set を生成
4. tag_names を生成
5. tag_set から MD5 hash を生成
6. tag_hash に UPSERT
```

### SQL 方針

タグ取得。

```sql
SELECT
    t.id,
    t.name
FROM tag_image_link til
INNER JOIN tag t
    ON t.id = til.tag_id
WHERE til.image_file_id = :image_file_id
ORDER BY t.id ASC
```

タグ0件時。

```sql
DELETE FROM tag_hash
WHERE image_file_data_id = :image_file_data_id
```

UPSERT。

```sql
INSERT INTO tag_hash (
    image_file_data_id,
    tag_names,
    tag_set,
    hash
)
VALUES (
    :image_file_data_id,
    :tag_names,
    :tag_set,
    :hash
)
ON CONFLICT(image_file_data_id)
DO UPDATE SET
    tag_names = excluded.tag_names,
    tag_set = excluded.tag_set,
    hash = excluded.hash
```

### Python 実装方針

```python
import hashlib

def _build_tag_hash_value(self, tag_set: str) -> str:
    return hashlib.md5(tag_set.encode("utf-8")).hexdigest()
```

## 8. タグ保存時の同期仕様

## 8.1 `replace_tags(...)` 変更

`ImageFileRepository.replace_tags(image_file_id, tags)` の最後に `replace_tag_hash(image_file_id)` を呼ぶ。

```text
1. 既存 tag_image_link 削除
2. tag を取得または作成
3. tag_image_link 作成
4. replace_tag_hash(image_file_id)
```

### 重要

`tag_image_link` 更新と `tag_hash` 更新は同一トランザクション内で行う。

## 8.2 `replace_tags_atomic(...)` 変更

`replace_tags_atomic(image_file_id, tags)` が存在する場合も、同じく `replace_tag_hash(image_file_id)` を含める。

```text
BEGIN
  replace_tags
  replace_tag_hash
COMMIT
```

失敗時は rollback。

## 8.3 `_update_detail_with_tags(...)` との関係

`ImageCatalogApi._update_detail_with_tags(detail, tags)` は、詳細とタグを同一トランザクションで更新する既存メソッドである。

この中で `repository.replace_tags(...)` または同等処理を呼ぶ場合、`replace_tags(...)` 側に `tag_hash` 更新を閉じ込める。

API 側で別途 `replace_tag_hash(...)` を呼ばない。

理由。

```text
API 側で呼ぶと、タグ更新経路が増えたときに同期漏れしやすい
Repository 側で閉じると、タグ更新の出口を一箇所にできる
```

## 8.4 プロンプト一括タグ登録時

既存のプロンプト一括タグ登録では、タグ未登録画像を抽出し、画像メタ情報からプロンプトを抽出してタグ登録する。処理フローでは `repository.replace_tags(image_file_id, tags)` によりタグ登録する設計である。

したがって、`replace_tags(...)` 側で `tag_hash` を更新すれば、プロンプト一括タグ登録時も自動的に同期される。

## 9. 画像削除時の同期仕様

## 9.1 単体削除

`delete_image_file(payload)` により `image_file_data` を削除する場合、`tag_hash` も削除されること。

推奨は以下。

```text
1. tag_hash 削除
2. tag_image_link 削除
3. image_file_data 削除
4. サムネイル削除
```

ただし、外部キー `ON DELETE CASCADE` が有効であれば、`image_file_data` 削除で `tag_hash` は自動削除される。

## 9.2 一括物理削除

一括物理削除では、実ファイル削除に成功または実ファイルが不存在の画像だけ、サムネイルと DB レコードを削除する設計である。

DB レコード削除対象に含まれた画像について、`tag_hash` も削除されること。

`delete_by_ids(record_ids)` 内で `image_file_data` を削除する前に、明示的に以下を実行してもよい。

```sql
DELETE FROM tag_hash
WHERE image_file_data_id IN (...)
```

## 12.1 `_normalize_tag_hash_condition(...)` 追加

```python
def _normalize_tag_hash_condition(self, data: dict) -> tuple[str | None, str | None]:
    tag_hash = str(data.get("tag_hash") or "").strip()
    tag_set = str(data.get("tag_set") or "").strip()

    if not tag_hash or not tag_set:
        return None, None

    return tag_hash, tag_set
```

## 12.2 `_normalize_duplicate_tag_set_limit(...)` 追加

```python
def _normalize_duplicate_tag_set_limit(self, value) -> int:
    try:
        limit = int(value or 256)
    except (TypeError, ValueError):
        limit = 256

    return max(1, min(limit, 256))
```

## 12.4 `fetch_duplicate_tag_sets(...)` 追加

```python
def fetch_duplicate_tag_sets(self, payload=None):
    try:
        data = payload or {}
        limit = self._normalize_duplicate_tag_set_limit(data.get("limit"))

        repository = self._get_repository()

        items = repository.find_duplicate_tag_sets(limit=limit)
        total_count = repository.count_duplicate_tag_sets()

        return ApiResponse(
            success=True,
            message="",
            data={
                "items": [item.to_dict() for item in items],
                "totalCount": total_count,
                "limit": limit,
            },
        ).to_dict()

    except Exception:
        return ApiResponse(
            success=False,
            message="重複タグ構成一覧を取得できませんでした。",
            data={
                "items": [],
                "totalCount": 0,
                "limit": 256,
            },
        ).to_dict()
```

```js
async function openDuplicateTagSetModal() {
  duplicateTagSetModal.show = true
  duplicateTagSetModal.items = []
  duplicateTagSetModal.totalCount = 0
  duplicateTagSetModal.message = ''
  duplicateTagSetModal.isLoading = true

  try {
    const result = await fetchDuplicateTagSets({ limit: 256 })

    if (!result?.success) {
      duplicateTagSetModal.message =
        result?.message || '重複タグ構成一覧を取得できませんでした。'
      duplicateTagSetModal.items = []
      duplicateTagSetModal.totalCount = 0
      return
    }

    duplicateTagSetModal.items = result.data?.items ?? []
    duplicateTagSetModal.totalCount = result.data?.totalCount ?? 0
    duplicateTagSetModal.limit = result.data?.limit ?? 256

    if (duplicateTagSetModal.items.length === 0) {
      duplicateTagSetModal.message = '重複しているタグ構成はありません。'
    }
  } catch (error) {
    duplicateTagSetModal.message = '重複タグ構成一覧を取得できませんでした。'
    duplicateTagSetModal.items = []
    duplicateTagSetModal.totalCount = 0
  } finally {
    duplicateTagSetModal.isLoading = false
  }
}
```

## 17. 受け入れ条件

### 17.1 DB

```text
- tag_hash テーブルが作成されること
- tag_hash.image_file_data_id が画像1件につき1件のみ保持されること
- tag_hash.hash にインデックスが作成されること
- tag_hash.hash + tag_hash.tag_set にインデックスが作成されること
```

### 17.2 tag_hash 生成

```text
- タグ1件以上の画像について tag_hash が生成されること
- tag_set が tag.id 昇順のカンマ結合になること
- tag_names が tag.id 昇順に対応する tag.name のカンマ結合になること
- hash が tag_set の MD5 になること
- タグ0件の画像では tag_hash が削除されること
```

### 17.3 同期更新

```text
- タグ編集保存時に tag_hash が更新されること
- プロンプト一括タグ登録時に tag_hash が更新されること
- 画像削除時に tag_hash が削除されること
- tag_image_link 更新と tag_hash 更新が同一トランザクションで扱われること
```

### 17.4 起動時補完

```text
- 起動時に tag_hash 件数を確認すること
- tag_hash 件数が 0 の場合だけ全件再構築すること
- tag_hash 件数が 1 件以上ある場合は全件再構築しないこと
- 再構築時、タグ0件画像は tag_hash に登録しないこと
```

### 17.5 API

```text
- fetch_duplicate_tag_sets が追加されていること
- 重複している tag_hash を hash + tag_set + tag_names 単位で取得できること
- imageCount が 2 件以上のものだけ返ること
- search_image_files が tag_hash / tag_set 条件を受け取れること
- tag_hash / tag_set の両方がある場合のみ hash 検索を行うこと
```

### 17.6 UI

```text
- AppHeader.vue に「重複タグ構成の検索」ボタンが表示されること
- ボタン押下で DuplicateTagSetModal.vue が開くこと
- モーダルに重複タグ構成と画像件数が表示されること
- モーダル項目クリックで通常検索条件をクリアし、該当タグ構成の画像を検索できること
- hash 値は検索フォームに表示しないこと
- 特殊検索中であることは検索条件欄に表示されること
- 通常検索フォームから検索した場合、hash 検索条件が解除されること
- ページング時は hash 検索条件が維持されること
```

```text
20. App.vue に composable とモーダルを組み込む
21. search_form.vue または検索条件欄に特殊条件表示を追加
22. CSS を追加
23. 動作確認
```

## 19. 注意点

### 19.1 `tag_hash` はキャッシュではなく検索用派生インデックス

扱いとしてはキャッシュに近いが、検索結果の正しさに直結する。

そのため、同期漏れは許容しない。

```text
tag_image_link は正しい
tag_hash は古い
```

という状態になると、画面検索が嘘をつく。

嘘つき検索はかなりたちが悪いです。バグのくせに堂々と検索結果として座っているので。

### 19.2 全件再構築は初回のみ

今回採用するのは以下。

```text
tag_hash 件数が 0 の場合だけ全件再構築
```

そのため、既に `tag_hash` が一部だけ存在する壊れ方には対応しない。

これは仕様として割り切る。

必要になった場合は、将来 `tag_hash 再構築` のメンテナンス API または管理ボタンを追加する。

### 19.3 通常検索時に hash 条件を必ず解除する

これを忘れると、画面上は検索条件を変えているのに裏で `hash` が効き続ける。

ユーザーから見ると「検索が壊れた」にしか見えないため、通常検索ボタン押下時は必ず解除する。

### 19.4 `hash` は UI に出さない

`hash` は人間が判断する値ではない。

UI には `tag_names` と `imageCount` だけ表示する。

ただし、検索 payload には `hash` と `tag_set` を持たせる。

### 19.5 タグ名変更機能を将来追加する場合

現時点ではタグ名変更機能は想定しない。

将来タグ名変更を追加する場合は、対象タグに紐づく全画像の `tag_hash.tag_names` を再生成する必要がある。

`tag_set` は tag.id ベースなので変わらないが、表示用の `tag_names` は変わる。
ここを忘れると、DBの中で昔の名前で出ています状態になります。昭和歌謡なら味ですが、検索UIではバグです。

# 実装設計書

## タグ最大長 512 文字拡張

## 1. 目的

現在、AZViewer のタグは 1 タグあたり最大 128 文字として扱っている。

Stable Diffusion WebUI 由来のプロンプト文字列や、自然文に近い長いタグを扱いやすくするため、タグ単体の最大長を 512 文字へ拡張する。

## 2. 変更概要

タグ単体の最大長を以下の通り変更する。

| 項目 | 変更前 | 変更後 |
|---|---:|---:|
| タグ単体の最大長 | 128 文字 | 512 文字 |

本変更では、以下を一括して変更する。

- DB の tag.name 長さ制約
- バックエンドのタグ正規化・検証処理
- フロントエンドのタグ入力時検証
- タグ表示 UI の長文対策
- 既存仕様書・エラーメッセージ中の 128 文字表記

## 3. 前提

既存のタグ管理機能では、タグは以下の構成で管理されている。

```text
tag
tag_image_link
```

また、画像詳細更新では `update_image_file_detail` の payload に `tags` を含め、既存の保存フローでタグを保存する。

```text
edit.vue
  ↓ save emit
tile.vue
  ↓ save-detail emit
Content.vue
  ↓ save-detail emit
App.vue
  ↓ useImageMutations.handleSaveDetail
  ↓ updateImageFileDetail
```

この保存経路は変更しない。

## 5. 基本方針

## 5.1 タグ単体の制限のみ変更する

変更対象は「1タグあたりの最大文字数」のみとする。

以下は変更しない。

- タグ件数上限なし
- ASCII カンマ `,` のみ区切り文字扱い
- 全角カンマ `，`、読点 `、` は区切り扱いしない
- 小文字化
- 末尾の :数値 ウェイト除去
- 非エスケープ括弧除去
- 正規化後重複は黙って統合

## 5.2 tagInput 全体の文字数制限は引き続き設けない

`edit.vue` の `tagInput` は、カンマ区切りで複数タグを入力できる。

そのため、入力欄全体の maxlength は引き続き設定しない。

```text
tagInput 全体: 制限なし
タグ単体: 512 文字以内
```

## 5.3 最終検証はバックエンドで行う

フロントエンド側にも事前検証を入れるが、最終的な保存可否はバックエンドの `TagNormalizeService` で判定する。

フロントエンド検証は UX 補助であり、信頼境界ではない。

## 5.4 長文タグの表示崩れを防ぐ

512 文字のタグは、DB 保存よりも UI 表示で問題になりやすい。

そのため、タグ badge 表示では、折り返しまたは省略表示を行い、画面横幅を破壊しないようにする。

CSS を放置すると、長文タグがタイルを突き破って脱獄します。DB より先に画面が泣きます。

## 6. DB 仕様

## 6.1 tag テーブル制約変更


変更後:

```sql
CHECK (length(name) <= 512)
```

`ImageFileRepository.create_table()` 内の `tag` テーブル作成 SQL を変更する。

```python
    cursor.execute(
        """
        CREATE TABLE IF NOT EXISTS tag (
            id INTEGER PRIMARY KEY AUTOINCREMENT,
```

実際の既存 SQL に合わせて、`CHECK (length(name) <= 128)` のみを `512` へ変更する。

現時点では正式リリース前の開発中アプリとみなし、既存 DB の自動マイグレーションは必須としない。

## 7. バックエンド設計

## 7.1 TagNormalizeService 変更

`backend/services/tag_normalize_service.py` にタグ最大長定数を追加する。

```python
MAX_TAG_NAME_LENGTH = 512
```

既にクラス内定数に近い構成がある場合は、それに合わせてよい。

```python
class TagNormalizeService:
    MAX_TAG_NAME_LENGTH = 512
```

## 7.2 長さ検証変更

変更前の意味:

```text
128 文字超過はエラー
```

変更後:

```text
512 文字超過はエラー
```

実装例:

```python
class TagNormalizeService:
    MAX_TAG_NAME_LENGTH = 512

    def _normalize_tag_text(self, value: str) -> str:
        tag = value.strip()
        tag = self._remove_unescaped_brackets(tag)
        tag = re.sub(r":\d+(?:\.\d+)?$", "", tag)
        tag = tag.strip().lower()

        if len(tag) > self.MAX_TAG_NAME_LENGTH:
            raise ValueError(
                f"タグは{self.MAX_TAG_NAME_LENGTH}文字以内で入力してください。"
            )

        return tag
```

既存実装の構造が異なる場合でも、以下を満たすこと。

- 正規化後のタグ文字列に対して長さ検証する
- 検証上限は 512
- エラーメッセージも 512 文字基準にする

## 7.3 正規化順序

正規化順序は既存仕様を維持し、長さ検証値のみ変更する。

変更後の正規化順序:

1. "," で分割
2. trim
3. バックスラッシュでエスケープされていない禁止文字 []{}() を除去
4. 末尾の :数値 を除去
5. trim 再実行
6. 小文字化
7. 空文字除去
8. 512文字超過はエラー
9. 正規化後重複は黙って統合

## 7.4 PromptTagImportService への影響

`PromptTagImportService` は `TagNormalizeService` を利用してプロンプト由来タグを正規化する構成である。

そのため、`PromptTagImportService` 側に個別の文字数変更処理は追加しない。

```text
PromptTagImportService
  ↓
TagNormalizeService.normalize_tags()
  ↓
512 文字制限
```

## 7.5 ImageCatalogApi への影響

`ImageCatalogApi.update_image_file_detail()` 側で独自に 128 文字制限を持っている場合は、削除するか、`TagNormalizeService.MAX_TAG_NAME_LENGTH` と同じ値へ統一する。

推奨は、タグ長検証を `TagNormalizeService` に集約すること。

```js
const MAX_TAG_NAME_LENGTH = 512

function addTagsFromInput() {
  const normalizedTags = form.tagInput
    .split(',')
    .map(normalizeTagText)
    .filter(Boolean)

  const tooLong = normalizedTags.find(
    (tag) => tag.length > MAX_TAG_NAME_LENGTH

  if (tooLong) {
    emit('error', {
      message: `タグは${MAX_TAG_NAME_LENGTH}文字以内で入力してください。`,

    return
  }

  form.tags = [...new Set([...form.tags, ...normalizedTags])]
  form.tagInput = ''
}
```

## 13. 既存 DB の注意事項

開発中 DB を再利用している場合、`CREATE TABLE IF NOT EXISTS` は既存テーブルの制約を変更しない。

そのため、既存 DB に `CHECK (length(name) <= 128)` が残っている場合、コード側を 512 にしても DB 側で失敗する。

対応方法:

- data/az_data.sqlite3 を削除して再作成する
- または tag テーブル再作成マイグレーションを実行する

今回の実装では、正式なマイグレーションは必須としない。

ただし、開発者向けメモとして以下を残す。

```text

```

ここを忘れると「コードは 512 と言っているのに DB が 128 で怒る」という、二重人格バグになります。

## 17.2 512 文字タグは UI 上かなり長い

保存できることと、見やすいことは別問題である。

タイル表示では省略、編集画面や検索モーダルでは折り返しなど、表示箇所ごとに扱いを分ける。

## 17.3 tag_hash 機能との関係

別途追加予定の `tag_hash` 機能では、タグ名をカンマ連結した `tag_names` が長くなる可能性がある。

ただし、検索キーは `hash` を使う想定であるため、512 文字化自体は問題になりにくい。

`tag_names` は表示・確認用、`hash` は検索・同一性判定用として扱う。

```

注意点として、既存 DB を使い回す場合だけは別途確認が必要です。コードを直しても SQLite の既存 `CHECK` 制約は勝手に変わりません。ここを忘れると、アプリが「私は512文字派です」と言いながら、DBが「いや128文字までです」と会議を始めます。
```
