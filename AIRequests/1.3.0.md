# 実装仕様書

## 選択画像タグ出力機能・ヘッダー操作メニュー再編

## 1. 目的

画像一覧画面で選択した画像について、画像に紐づくタグを、画像ファイルと同名の `.txt` ファイルとして出力する。

主用途は LoRA 作成時の caption ファイル生成である。

あわせて、ヘッダー上の一括操作ボタンが増えてきたため、既存の以下操作をドロップダウンメニューへ再編する。

```text
- ワイルドカード出力
- タグ出力
- ファイルを移動
- 選択画像を削除
```

既存構成上、`AppHeader.vue` は一括操作ボタン群を保持し、`AppApi` / `ImageCatalogApi` はワイルドカード出力や一括削除・移動をすでに担当しているため、この構成に合わせて追加する。

---

## 2. 変更概要

以下を追加・変更する。

```text
- AppHeader.vue の「ワイルドカード出力」単体ボタンを廃止
- AppHeader.vue に「アウトプット」ドロップダウンを追加
  - ワイルドカード出力
  - タグ出力

- AppHeader.vue の「ファイルを移動」「選択画像を削除」単体ボタンを廃止
- AppHeader.vue に「ファイル操作」ドロップダウンを追加
  - ファイルを移動
  - 選択画像を削除

- タグ出力確認ダイアログを追加
- 選択画像のタグを、画像と同じフォルダに同名 .txt として出力
- 既存 .txt がある場合は上書き
- タグ 0 件の画像はスキップ
- 出力中は LoadingOverlay で画面ロック
- 完了後に toast で結果件数を通知
```

## 4. 基本方針

## 4.1 選択状態は既存仕様を流用する

画像選択状態は DB に保存しない。

既存の一括削除・一括移動・ワイルドカード出力と同じく、`App.vue` の一時状態である `selectedImageIds` / `selectedImageIdSet` を利用する。

一括選択仕様でも、選択状態はフロントエンド上の一時状態として扱う方針が定義済みである。

## 4.2 タグ出力は DB 更新を行わない

本機能は `.txt` ファイルを出力するだけであり、DB 更新は行わない。

```text
- image_file_data 更新なし
- tag 更新なし
- tag_image_link 更新なし
- tag_hash 更新なし
```

## 4.3 既存 .txt は上書きのみ

既存の `.txt` ファイルがある場合、確認後に上書きする。

```text
追記は行わない
追記モードは持たない
既存末尾改行補正も不要
```

## 4.4 タグ 0 件画像はスキップ

画像に紐づくタグが 0 件の場合、`.txt` は作成しない。

```text
タグ 0 件
→ スキップ
→ 失敗扱いにはしない
```

## 4.5 出力順序

タグの出力順は以下とする。

```text
tag.id 昇順
```

既存の一覧表示に含まれる `tags` 配列ではなく、バックエンド側で対象画像 ID からタグを再取得する。

理由は、出力時点の DB 状態を正とし、フロント表示内容とのズレを避けるためである。既存 Repository には画像 ID ごとのタグ一覧取得である `find_tags_by_image_ids(image_ids)` が存在する。

## 4.6 文字コード

```text
UTF-8 without BOM
```

## 4.7 改行コード

caption ファイルは 1 ファイル 1 行のため、本文末尾に改行を付けるかは実装上どちらでも大きな差はない。

推奨は以下。

```text
本文末尾に LF を 1 つ付与する
```

出力例。

```text
tag1, tag2, tag3\n
```

Windows アプリでは CRLF でもよいが、caption ファイル用途では LF で問題ない。ワイルドカード出力は Windows ローカル用途として CRLF 固定だが、本機能は 1 画像 1 caption ファイルであり、行単位の差分管理が主目的ではないため LF とする。

## 7.4 useImageMutations.js

一括操作系の composable にタグ出力処理を追加する。

既存の一括削除・一括移動と同系統の「選択画像に対するファイル操作」なので、`useImageMutations.js` へ配置する。

### 公開関数

```js
async function exportSelectedTags({ ids, clearSelection }) {
  ...
}
```

### 処理フロー

```text
1. ids が空なら何もしない
2. 確認ダイアログを表示
3. キャンセルなら何もしない
4. LoadingOverlay を表示
5. exportSelectedImageTags API を呼ぶ
6. 結果を toast 表示
7. 成功時は clearSelection() を実行
8. finally で LoadingOverlay を解除
```

### 実装例

```js
import { exportSelectedImageTags } from '../services/backendApi'

async function exportSelectedTags({ ids, clearSelection }) {
  const targetIds = [...(ids ?? [])]

  if (targetIds.length === 0) {
    return
  }

  const confirmed = window.confirm(
    [
      `選択した ${targetIds.length} 件の画像について、画像に設定されているタグを同名の .txt ファイルとして出力します。`,
      '',
      '出力先は各画像ファイルと同じフォルダです。',
      '既に .txt ファイルが存在する場合は上書きされます。',
      'タグが設定されていない画像はスキップされます。',
      '',
      '実行しますか？',
    ].join('\n')
  )

  if (!confirmed) {
    return
  }

  loading.showLoading({
    title: 'タグを出力中',
    message: '選択画像のタグを caption ファイルとして出力しています...',
  })

  try {
    const result = await exportSelectedImageTags({
      ids: targetIds,
    })

    if (!result?.success) {
      pushToast({
        type: 'danger',
        message: result?.message || 'タグ出力に失敗しました。',
      })
      return
    }

    const data = result.data ?? {}
    const failedCount = data.failedCount ?? 0

    const message = `タグ出力が完了しました。対象: ${data.targetCount ?? targetIds.length} 件 / 出力: ${data.exportedCount ?? 0} 件 / スキップ: ${data.skippedCount ?? 0} 件 / 失敗: ${failedCount} 件`

    pushToast({
      type: failedCount > 0 ? 'warning' : 'success',
      message,
    })

    clearSelection?.()
  } catch (error) {
    pushToast({
      type: 'danger',
      message: 'タグ出力に失敗しました。',
    })
  } finally {
    loading.hideLoading()
  }
}
```

## 9. DTO 仕様

```python
@dataclass
class TagCaptionExportFailure:
    id: int
    path: str
    reason: str

    def to_dict(self) -> dict:
        return {
            "id": self.id,
            "path": self.path,

@dataclass
class TagCaptionExportResult:

    exported_count: int
    skipped_count: int
    failed_count: int

            "exportedCount": self.exported_count,
            "skippedCount": self.skipped_count,
```

## 11.5 処理フロー

```text
1. target_count = len(items)
2. items を 1 件ずつ処理
3. item.id に対応するタグ一覧を取得
4. タグ 0 件なら skipped_count 加算
5. item.path から出力先 txt パスを作る

7. UTF-8 without BOM で上書き保存する
8. 成功したら exported_count 加算
9. 例外発生時は failed_count 加算し、failed_files に最大 20 件まで追加
10. 結果 DTO を返す
```

## 12. ImageCatalogApi 実装方針

## 12.1 正規化メソッド

既存の一括削除・移動と同様の ID 正規化を使う。
共通化できるなら private メソッドを使い回す。

```python
def _normalize_ids(self, value) -> list[int]:
    if not isinstance(value, list):
        return []

    ids: list[int] = []

    for item in value:
        try:
            record_id = int(item)
        except (TypeError, ValueError):
            continue

        if record_id <= 0:
            continue

        if record_id in ids:
            continue

        ids.append(record_id)

    return ids
```

## 12.2 空レスポンス

```python
def _empty_tag_caption_export_data(self) -> dict:
    return TagCaptionExportResult(
        target_count=0,
        exported_count=0,
        skipped_count=0,
        failed_count=0,
        failed_files=[],
    ).to_api_data()
```

## 12.3 API 実装イメージ

```python
def export_selected_image_tags(self, payload=None):
    try:
        data = payload or {}
        record_ids = self._normalize_ids(data.get("ids"))

        if not record_ids:
            return ApiResponse(
                success=False,
                message="タグ出力対象が指定されていません。",
                data=self._empty_tag_caption_export_data(),
            ).to_dict()

        repository = self._get_repository()
        items = repository.find_by_ids(record_ids)

        if not items:
            return ApiResponse(
                success=True,
                message="タグ出力対象の画像はありませんでした。",
                data=self._empty_tag_caption_export_data(),
            ).to_dict()

        tags_by_image_id = repository.find_tags_by_image_ids(
            [item.id for item in items]
        )

        service = self._get_tag_caption_export_service()
        result = service.export(
            items=items,
            tags_by_image_id=tags_by_image_id,
        )

        message = (
            "タグ出力が完了しましたが、一部失敗しました。"
            if result.failed_count > 0
            else "タグ出力が完了しました。"
        )

        return ApiResponse(
            success=True,
            message=message,
            data=result.to_api_data(),
        ).to_dict()

    except Exception:
        return ApiResponse(
            success=False,
            message="タグ出力に失敗しました。",
            data=self._empty_tag_caption_export_data(),
        ).to_dict()
```

## 4. 基本方針

## 4.1 選択状態は既存仕様を流用する

画像選択状態は DB に保存しない。

既存の一括操作と同様に、`App.vue` の一時状態である `selectedImageIds` / `selectedImageIdSet` を利用する。

```text
- DB カラム追加なし
- image_file_data に選択状態を持たせない
- チェックされた画像 ID を API payload として送る
```

## 4.2 タグ追加は既存タグを保持する

caption 読み込み・一括タグ追加のどちらも、既存タグを置き換えない。

```text
既存タグ + 追加タグ
```

禁止。

```text
既存タグを削除して caption タグだけに置き換える
既存タグを削除して一括入力タグだけに置き換える
```

ここは明記必須です。
実装者が `replace_tags` をうっかり呼ぶと、caption 読み込み機能が「追加」ではなく「タグ消しゴム」になります。消しゴムにしては殺意が高い。

## 4.3 タグ正規化は既存ルールを使う

caption 読み込み・一括タグ追加の入力文字列は、既存の `TagNormalizeService` 相当の正規化ルールを通す。

想定ルール。

```text
- ASCII カンマ "," で分割
- 前後空白を trim
- 空タグ除外
- 禁止文字除去
- 末尾の :数値 を除去
- 小文字化
- 重複除去
- タグ単体の最大長制限を適用
```

## 4.4 tag_hash を必ず更新する

caption 読み込み・一括タグ追加は、画像のタグ構成を変更する。

そのため、タグリンク更新と `tag_hash` 更新を同一トランザクションで扱う。

既存仕様でも、タグ編集保存時は `replace_tags` と `replace_tag_hash` を同一フローで実行し、プロンプト一括タグ登録時にも `replace_tag_hash` を更新する流れになっています。

追加対象。

```text
- キャプションタグ読み込み成功画像
- 一括タグ追加成功画像
```

## 4.5 処理単位

caption 読み込み・一括タグ追加は、どちらも **1画像単位で処理継続** とする。

```text
1画像で失敗しても、他画像の処理は継続する
```

プロンプト一括読み取りでも、1件ずつ処理して失敗詳細を集計する設計になっているため、それに合わせる。

# 6. キャプションタグ読み込み仕様

## 6.1 概要

チェックした画像に対して、画像ファイルと同じディレクトリにある同名 `.txt` を読み込み、画像タグとして追加登録する。

例。

```text
画像:
C:/images/sample.png

読み込み対象:
C:/images/sample.txt
```

## 6.2 対象

```text
selectedImageIds に含まれる画像
```

## 6.3 ファイル探索

Python 側で対象画像を ID から再取得する。

```text
フロント:
ids のみ送信

バックエンド:
find_by_ids(ids) で path を取得
Path(item.path).with_suffix(".txt") で caption パスを生成
```

フロントから path を送らない。
削除・移動系と同じく、対象は DB 登録済み画像に限定するべきです。フロントから path を受け始めると、入力欄が急にファイル破壊光線になります。

## 6.4 読み込み文字コード

```text
UTF-8
```

BOM 付き UTF-8 は許容してよい。

推奨実装。


Shift_JIS 自動判定は初回仕様に含めない。
必要になったら別仕様でよいです。文字コード自動判定は、見た目より沼です。水たまりかと思ったら底なしタイプ。


# 8. タグ出力仕様変更

## 8.1 変更対象

既存の「タグ出力」。

既存仕様では、選択画像のタグを画像と同名 `.txt` に出力し、既存 `.txt` は上書き、タグ 0 件画像はスキップ、DB 更新は行わない方針です。

## 8.2 出力順序

変更前。

```text
tag.id 昇順
```

変更後。

```text
選択画像内でのタグ出現頻度順
```

## 8.3 出現頻度の定義

対象範囲。

```text
タグ出力対象として選択された画像群
```

頻度の数え方。

```text
そのタグを含む画像数
```

1画像内で同一タグが重複していても、1件として数える。

ただし、既存タグ管理上は画像内重複タグは基本的に排除される前提。

## 8.4 並び順

各 caption ファイル内のタグ順序は以下。

```text
1. 選択画像内での出現画像数 DESC
2. 同頻度の場合は tag.name ASC
```

例。

```text
選択画像:
A: 1girl, blue eyes, smile
B: 1girl, smile
C: 1girl, black hair

出現頻度:
1girl = 3
smile = 2
black hair = 1
blue eyes = 1

A の出力:
1girl, smile, blue eyes
```

## 8.5 DB 取得

出力時点の DB 状態を正とするため、既存どおりバックエンド側で対象画像 ID からタグを再取得する。既存 Repository には画像 ID ごとのタグ一覧取得である `find_tags_by_image_ids(image_ids)` が定義されています。

ただし、今回は頻度計算に `tag.name` が必要なため、内部的には以下のような構造が望ましい。

```python
{
    image_id: [
        {"id": 1, "name": "1girl"},
        {"id": 2, "name": "smile"},
    ]
}
```

現行の `find_tags_by_image_ids` が `dict[int, list[str]]` を返す場合は、以下のどちらかで対応する。

```text
A. find_tags_by_image_ids の戻り値を既存互換のまま使い、name ベースで頻度計算する
B. caption 出力専用に find_tag_items_by_image_ids を追加する
```

推奨は A です。
今回の同頻度ソートは `tag.name ASC` で足りるため、`tag.id` は必須ではありません。

## 8.6 出力処理中の画面ロック

タグ頻度集計とファイル出力を行うため、出力完了まで LoadingOverlay を表示する。

```text
タイトル: タグを出力中
メッセージ: タグ出現頻度を集計し、caption ファイルを出力しています...
```

## 8.7 出力ファイル仕様

既存仕様を維持。

```text
- 画像と同名 .txt
- 画像と同じディレクトリへ出力
- 既存 .txt は上書き
- UTF-8 without BOM
- 本文末尾に LF を 1 つ付与
- タグ 0 件画像はスキップ
```

既存仕様でも、caption 出力は UTF-8 without BOM、末尾 LF 推奨です。

# 11. Repository 仕様

## 11.1 追加メソッド

```python
def merge_tags_atomic(self, image_file_id: int, tags: list[str]) -> bool:
    ...
```

戻り値。

```text
True:
タグ構成が変更された

False:
追加対象タグがすべて既存で、変更なし
```

## 11.2 処理内容

```text

3. 重複を除去する
4. 変更がなければ False を返す

6. tag_hash を更新する
7. commit
8. True を返す
```

## 11.3 注意

`replace_tags` を直接使う場合でも、呼び出し前に必ず既存タグと追加タグを統合する。

```text
replace_tags(image_file_id, caption_tags)
```

は禁止。

正しくは。

```text
merged_tags = existing_tags + caption_tags
replace_tags(image_file_id, merged_tags)
replace_tag_hash(image_file_id)
```

# 12. TagCaptionExportService 変更

既存の `TagCaptionExportService` に頻度順ソートを追加する。

## 12.1 追加処理

```text
1. 対象画像全体のタグ一覧を取得する
2. タグごとの出現画像数を集計する
3. 各画像のタグを以下で並べ替える
   - 出現画像数 DESC
   - tag.name ASC
4. ", " で連結して出力する
```

## 12.2 実装イメージ

```python
def build_frequency_map(tags_by_image_id: dict[int, list[str]]) -> dict[str, int]:
    frequency: dict[str, int] = {}

    for tags in tags_by_image_id.values():
        for tag in set(tags):
            frequency[tag] = frequency.get(tag, 0) + 1

    return frequency


def sort_tags_by_frequency(tags: list[str], frequency: dict[str, int]) -> list[str]:
    return sorted(
        tags,
        key=lambda tag: (-frequency.get(tag, 0), tag),
    )
```

## 注意点

最大の注意点は、**merge と replace を取り違えないこと**です。

今回の 2 機能はどちらも「追加」であって「置換」ではありません。

```text
キャプションタグ読み込み:
既存タグ + caption タグ

一括タグ追加:
既存タグ + 入力タグ
```

ここを誤ると LoRA 用 caption 調整のつもりが、既存タグをきれいに更地にします。更地にするならせめて不動産登記をしてからにしましょう。
