# 実装設計書

## フォルダ単位操作機能追加

## 1. 目的

AZViewer において、`image_folder` に登録されたフォルダ単位で、以下の操作を実行できるようにする。

```text
- フォルダを管理対象から除外
- フォルダごとごみ箱へ移動
```

本機能では、OS 上のフォルダそのものを直接削除・移動するのではなく、選択した `image_folder.id` に紐づく `image_file_data` の登録済み画像を対象にする。

つまり、処理対象は以下である。

```text
選択した image_folder.id に紐づく AZViewer 登録済み画像
```

対象外。

```text
- AZViewer に登録されていないファイル
- 同一 OS フォルダ内に存在する .txt / .psd / 作業ファイルなど
- OS フォルダそのもの
```

これにより、フォルダ単位操作であっても AZViewer 管理外のファイルを巻き込まない。

---

## 2. 前提

本機能は、フォルダ管理リファクタ後の構造を前提とする。

```text
- image_folder テーブルが存在する
- image_file_data.folder_id が image_folder.id を参照する
- image_file_data.folder カラムは廃止済み
- フォルダ検索候補は id / name / path / imageCount を返す
```

対象画像の抽出は、`path LIKE` や親フォルダ文字列推測ではなく、`folder_id` の完全一致で行う。

```sql
SELECT id
FROM image_file_data
WHERE folder_id = :folder_id
ORDER BY id ASC;
```

これにより、同名フォルダ問題を回避する。

---

## 3. 基本方針

## 3.1 UI はヘッダーのドロップダウンに追加する

`AppHeader.vue` に以下のドロップダウンを追加する。

```text
フォルダ単位操作
  - フォルダを管理対象から除外
  - フォルダごとごみ箱へ移動
```

このメニューは選択画像件数に依存しない。

```text
selectedCount === 0 でも有効
```

ただし、検索中・重処理中など `isBusy` の場合は disabled にする。

## 3.2 フォルダ選択は操作専用モーダルで行う

検索条件用の `FolderSearchModal.vue` とは別に、操作専用のモーダルを追加する。

```text
frontend/src/components/form/FolderOperationModal.vue
```

理由。

```text
- 検索条件選択と破壊的操作では責務が異なる
- ごみ箱移動など危険操作用の確認導線が必要
- 将来的にフォルダ単位操作が増えても拡張しやすい
```

見た目や検索 UI の考え方は `FolderSearchModal.vue` に寄せてよいが、コンポーネントは分離する。

## 3.3 実処理は既存画像単位処理へ委譲する

フォルダ単位専用の削除ロジックを新しく作らない。

処理手順。

```text
1. folderId を受け取る
2. image_folder の存在を確認する
3. folder_id に紐づく image_file_data.id 一覧を取得する
4. 既存の画像単位 API 相当処理へ委譲する
```

管理対象から除外。

```text
folder_id -> image ids -> remove_image_files_from_catalog 相当処理
```

ごみ箱へ移動。

```text
folder_id -> image ids -> move_image_files_to_trash 相当処理
```

これにより、既存の関連情報削除処理を再利用する。

対象となる既存処理。

```text
- image_file_data 削除
- tag_image_link 削除
- image_model_to_file_data 削除
- tag_hash 削除
- サムネイルキャッシュ削除
- 実ファイルの send2trash
```

## 3.4 image_folder は即時削除しない

フォルダ単位操作によって、その `image_folder` を参照する画像が 0 件になっても、`image_folder` レコードは即時削除しない。

未使用フォルダ定義の削除は、既存のマスタメンテナンス機能に追加した「未使用フォルダを削除」で行う。

理由。

```text
- 操作結果の確認がしやすい
- 一部失敗時の扱いが複雑にならない
- フォルダマスタの整理責務をマスタメンテナンスに集約できる
```

既存の `fetchFoldersForSearch` を利用する。

```js
async function searchFolderOperationCandidates() {
  folderOperationModal.isLoading = true

  try {
    const result = await fetchFoldersForSearch({
      keyword: folderOperationModal.keyword,
      limit: folderOperationModal.limit,
    })

    if (!result?.success) {
      pushToast({
        type: 'danger',
        message: result?.message || 'フォルダ一覧を取得できませんでした。',
      })
      folderOperationModal.folders = []
      folderOperationModal.totalCount = 0
      return
    }

    folderOperationModal.folders = result.data?.folders ?? []
    folderOperationModal.totalCount = result.data?.totalCount ?? 0
    folderOperationModal.limit = result.data?.limit ?? 256
  } finally {
    folderOperationModal.isLoading = false
  }
}
```

## 8.7 実行

```js
async function executeFolderOperation() {
  const folder = folderOperationModal.selectedFolder

  if (!folder?.id) {
    pushToast({
      type: 'warning',
      message: '操作対象のフォルダを選択してください。',
    })
    return
  }

  folderOperationModal.isExecuting = true

  try {
    if (folderOperationModal.mode === 'remove') {
      await imageMutations.removeFolderFromCatalog({
        folder,
        refresh: imageSearch.executeSearch,
        clearSelection,
      })
    } else {
      await imageMutations.moveFolderToTrash({
        folder,
        refresh: imageSearch.executeSearch,
        clearSelection,
      })
    }

    closeFolderOperationModal()
  } finally {
    folderOperationModal.isExecuting = false
  }
}
```

## 9.2 removeFolderFromCatalog 追加

```js
async function removeFolderFromCatalog({ folder, refresh, clearSelection }) {
  if (!folder?.id) {
    pushToast({
      type: 'warning',
      message: '操作対象のフォルダを選択してください。',
    })
    return
  }

  const confirmed = window.confirm(
    `次のフォルダに属する登録済み画像を AZViewer の管理対象から除外します。\n\n` +
      `対象フォルダ:\n${folder.path}\n\n` +
      `対象画像: ${folder.imageCount ?? 0} 件\n\n` +
      `この操作では、AZViewer 上の登録情報、タグ関連情報、モデル関連情報、サムネイルキャッシュを削除します。\n` +
      `実際の画像ファイルは削除されません。\n\n` +
      `実行しますか？`
  )

  if (!confirmed) {
    return
  }

  showLoading(
    'フォルダを管理対象から除外中',
    '対象フォルダ内の登録情報を AZViewer から除外しています。'
  )

  try {
    const result = await removeFolderFromCatalogApi({
      folderId: folder.id,
    })

    if (!result?.success) {
      pushToast({
        type: 'danger',
        message: result?.message || 'フォルダの管理対象除外に失敗しました。',
      })
      return
    }

    const data = result.data ?? {}

    pushToast({
      type: data.failedCount > 0 ? 'warning' : 'success',
      message:
        `フォルダ内の画像を管理対象から除外しました。` +
        `対象: ${data.targetCount ?? 0} 件 / ` +
        `除外: ${data.removedCount ?? 0} 件 / ` +
        `失敗: ${data.failedCount ?? 0} 件`,
    })

    clearSelection?.()
    await refresh?.()
  } finally {
    hideLoading()
  }
}
```

```js
    pushToast({
      type: data.failedCount > 0 ? 'warning' : 'success',
      message:
        `フォルダ内画像をごみ箱へ移動しました。` +
        `対象: ${data.targetCount ?? 0} 件 / ` +
        `移動: ${data.trashedCount ?? 0} 件 / ` +
        `失敗: ${data.failedCount ?? 0} 件`,
    })

    clearSelection?.()
    await refresh?.()
  } finally {
    hideLoading()
  }
}
```

## 12. ImageCatalogApi 設計

## 12.1 remove_folder_from_catalog 追加

```python
def remove_folder_from_catalog(self, payload):
    try:
        folder_id = self._normalize_folder_id(
            (payload or {}).get("folderId")
        )

        if folder_id is None:
            return ApiResponse(
                success=False,
                message="対象フォルダが指定されていません。",
                data=self._empty_folder_catalog_removal_data(),
            ).to_dict()

        repository = self._get_repository()

        folder = repository.find_folder_by_id(folder_id)

        if folder is None:
            return ApiResponse(
                success=False,
                message="対象フォルダが見つかりません。",
                data=self._empty_folder_catalog_removal_data(),
            ).to_dict()

        image_ids = repository.find_image_ids_by_folder_id(folder_id)

        if not image_ids:
            return ApiResponse(
                success=False,
                message="対象フォルダに登録済み画像がありません。",
                data={
                    **self._empty_folder_catalog_removal_data(),
                    "folderId": folder_id,
                    "folderName": folder.name,
                    "folderPath": folder.path,
                },
            ).to_dict()

        result = self.remove_image_files_from_catalog({
            "ids": image_ids,
        })

        return self._convert_folder_catalog_removal_response(
            result=result,
            folder=folder,
            target_count=len(image_ids),
        )

    except Exception:
        return ApiResponse(
            success=False,
            message="フォルダの管理対象除外に失敗しました。",
            data=self._empty_folder_catalog_removal_data(),
```

## 12.2 move_folder_to_trash 追加

```python
def move_folder_to_trash(self, payload):
    try:
        folder_id = self._normalize_folder_id(
            (payload or {}).get("folderId")
        )

        if folder_id is None:
            return ApiResponse(
                success=False,
                message="対象フォルダが指定されていません。",
                data=self._empty_folder_trash_move_data(),
            ).to_dict()

        repository = self._get_repository()

        folder = repository.find_folder_by_id(folder_id)

        if folder is None:
            return ApiResponse(
                success=False,
                message="対象フォルダが見つかりません。",
                data=self._empty_folder_trash_move_data(),
            ).to_dict()

        image_ids = repository.find_image_ids_by_folder_id(folder_id)

        if not image_ids:
            return ApiResponse(
                success=False,
                message="対象フォルダに登録済み画像がありません。",
                data={
                    **self._empty_folder_trash_move_data(),
                    "folderId": folder_id,
                    "folderName": folder.name,
                    "folderPath": folder.path,
                },
            ).to_dict()

        result = self.move_image_files_to_trash({
            "ids": image_ids,
        })

        return self._convert_folder_trash_move_response(
            result=result,
            folder=folder,
            target_count=len(image_ids),
        )

    except Exception:
        return ApiResponse(
            success=False,
            message="フォルダ内画像のごみ箱移動に失敗しました。",
            data=self._empty_folder_trash_move_data(),
        ).to_dict()
```

## 12.3 _normalize_folder_id 追加

```python
def _normalize_folder_id(self, value) -> int | None:
    if value in (None, ""):
        return None

    try:
        folder_id = int(value)
    except (TypeError, ValueError):
        return None

    if folder_id <= 0:
        return None

    return folder_id
```

## 18. 受け入れ条件

## 18.1 ヘッダー

```text
- ヘッダーに「フォルダ単位操作」ドロップダウンが表示されること
- 「フォルダを管理対象から除外」を選択できること
- 「フォルダごとごみ箱へ移動」を選択できること
- isBusy 中はドロップダウンが disabled になること
- selectedCount === 0 でもドロップダウンが有効であること
```

## 18.2 FolderOperationModal

```text
- 操作モードに応じてタイトルが切り替わること
- フォルダ名またはパスで検索できること
- フォルダ候補にフォルダ名、フルパス、登録画像数が表示されること
- 同名フォルダが複数あってもフルパスで区別できること
- フォルダを1件だけ選択できること
- 選択中フォルダと対象画像数が表示されること
- 対象フォルダ未選択時は実行できないこと
- 対象画像数 0 件のフォルダは実行できないこと
```

## 18.3 管理対象から除外

```text
- folderId を指定して管理対象除外 API を呼び出せること
- バックエンド側で folder_id に紐づく画像 ID が取得されること
- 既存の画像単位管理対象除外処理に委譲されること
- 実画像ファイルは削除されないこと
- 対象画像の DB 登録情報、関連タグ、関連モデル、tag_hash、サムネイルキャッシュが削除されること
- 処理後に現在検索条件で再検索されること
- 処理後に選択状態が解除されること
```

## 18.4 ごみ箱へ移動

```text
- folderId を指定してごみ箱移動 API を呼び出せること
- バックエンド側で folder_id に紐づく画像 ID が取得されること
- 既存の画像単位ごみ箱移動処理に委譲されること
- AZViewer に登録されていないファイルは対象にならないこと
- ごみ箱移動成功分だけ管理対象から除外されること
- 一部失敗時に失敗件数と失敗ファイルが返ること
- 処理後に現在検索条件で再検索されること
- 処理後に選択状態が解除されること
```

## 18.5 image_folder

```text
- フォルダ単位操作後も image_folder は即時削除されないこと
- 参照画像が 0 件になった image_folder は未使用フォルダとして残ること
- 未使用フォルダはマスタメンテナンスから削除できること
```

## 20. 注意点

```text
- OS フォルダそのものは削除しない
- AZViewer 登録済み画像だけを対象にする
- 対象抽出は path LIKE ではなく folder_id で行う
- フォルダ候補では必ず path を表示する
- 同名フォルダを name だけで扱わない
- image_folder は操作後に即時削除しない
- 既存の画像単位除外・ごみ箱移動ロジックを再利用する
- フロントから大量の image ids を渡さない
- 対象画像 ID の確定はバックエンドで行う
- 処理後は clearSelection と再検索を行う
```

今回の肝は、`folder_id` を使って対象画像を正確に確定し、既存の画像単位処理に流すことである。

新しい削除処理を作り込む必要はない。
むしろ作り込むと、タグリンク・モデルリンク・tag_hash・サムネイルキャッシュ削除の整合性処理が二重化し、後で片方だけ修正される事故が起きる。

削除処理の分身はだいたい最後に敵になる。ゲームでも設計でも同じ。
