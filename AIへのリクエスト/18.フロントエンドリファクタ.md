# 実装仕様書

## App.vue 分割リファクタリング

## 1. 目的

`App.vue` が保持している検索、詳細表示、更新、削除、通知、ローディング、backend API 呼び出しの責務を分離し、`App.vue` を画面構成の統括に限定する。

現状 `App.vue` は検索状態、一覧データ、詳細モーダル、トースト、ローディング、backend API 呼び出しを統括しているため、責務が集中している。

---

## 2. 基本方針

### 2.1 backendApi を各コンポーネントから直接呼ばない

`backendApi.js` は pywebview API 呼び出しのラッパーとして維持する。

ただし、以下のような呼び出しは禁止する。

```js
// NG
import { searchImageFiles } from '@/services/backendApi'
```

```vue
<!-- NG: tile.vue / Content.vue / ImageDetailModal.vue から直接 API 呼び出し -->
```

### 2.2 API 呼び出しは composable に集約する

コンポーネントはイベントを emit するだけにし、業務処理は composable に移す。

これは、既存方針である「tile にロジックを持たせない」「API呼び出しは親へ集約」「一方向データフロー維持」を崩さないため。

---

## 3. 追加ディレクトリ

```text
frontend/src/composables/
  useAppStatus.js
  useToast.js
  useLoadingOverlay.js
  useImageSearch.js
  useImageDetail.js
  useImageMutations.js
```

---

## 4. 分離後の責務

## 4.1 App.vue

### 責務

* レイアウト構成
* composable の組み立て
* 子コンポーネントへの state / handler 受け渡し
* 初期ロード呼び出し

### 非責務

* backendApi の直接呼び出し
* 検索 payload 正規化
* サムネイル URL 付与
* 詳細画像取得
* 更新 API 呼び出し
* 削除 API 呼び出し
* トースト配列管理
* ローディング状態の直接制御

---

## 4.2 useAppStatus.js

### 責務

フッター表示用ステータスを管理する。

### 移動対象

`App.vue` から以下を移動する。

```js
setStatus(type, message)
```

### 公開値

```js
return {
  status,
  setStatus,
}
```

---

## 4.3 useToast.js

### 責務

トースト通知を管理する。

### 移動対象

`App.vue` から以下を移動する。

```js
pushToast(notification)
```

### 公開値

```js
return {
  toasts,
  pushToast,
  removeToast,
}
```

---

## 4.4 useLoadingOverlay.js

### 責務

全画面ローディングオーバーレイの状態を管理する。

### 公開値

```js
return {
  loadingOverlay,
  showLoading,
  hideLoading,
  withLoading,
}
```

### 想定状態

```js
const loadingOverlay = reactive({
  show: false,
  title: '',
  message: '',
})
```

---

## 4.5 useImageSearch.js

### 責務

画像検索、ページング、ソート、サムネイル URL 付与を担当する。

現状 `App.vue` は `normalizeSearchPayload`、`applySearchResult`、`addThumbnailUrls`、`addThumbnailUrl`、`executeSearch` を持つため、これらを本 composable に移動する。

### 移動対象

```js
normalizeSearchPayload(overrides = {})
applySearchResult(data, appliedPayload)
addThumbnailUrls(items)
addThumbnailUrl(item)
executeSearch(overrides = {}, overlay = false)
handleSearch()
handlePageChange(page)
handlePageSizeChange(pageSize)
handleSortChange(sort)
handleImportComplete()
```

### 使用 API

```js
searchImageFiles(payload)
fetchLocalImageThumb(id)
```

### 公開値

```js
return {
  filters,
  searchResult,
  isSearching,
  currentSearchPayload,

  executeSearch,
  handleSearch,
  handleImportComplete,
  handlePageChange,
  handlePageSizeChange,
  handleSortChange,
}
```

---

## 4.6 useImageDetail.js

### 責務

詳細モーダル表示、画像本体取得、フォルダオープンを担当する。

`ImageDetailModal.vue` は画像表示専用であり、詳細モーダルでは open / close / 画像読み込みのみ扱う方針。

### 移動対象

```js
handleOpenDetail(item)
handleCloseDetail()
handleOpenContainingFolder(path)
```

### 使用 API

```js
fetchLocalImage(path)
openContainingFolder(path)
```

### 公開値

```js
return {
  selectedItem,
  detailImageUrl,
  isLoadingImage,

  handleOpenDetail,
  handleCloseDetail,
  handleOpenContainingFolder,
}
```

---

## 4.7 useImageMutations.js

### 責務

画像情報の更新、削除を担当する。

既存仕様では、削除・保存後は API 呼び出し後に再検索する。

### 移動対象

```js
handleDelete(id)
handleSaveDetail(payload)
```

### 使用 API

```js
deleteImageFile(id)
updateImageFileDetail(payload)
```

### 公開値

```js
return {
  handleDelete,
  handleSaveDetail,
}
```

### 依存注入

```js
useImageMutations({
  pushToast,
  setStatus,
  refresh,
})
```

`refresh` には `useImageSearch().executeSearch` を渡す。

---

## 5. App.vue の組み立て例

```js
const { status, setStatus } = useAppStatus()
const { toasts, pushToast, removeToast } = useToast()
const loading = useLoadingOverlay()

const imageSearch = useImageSearch({
  pushToast,
  setStatus,
  loading,
})

const imageDetail = useImageDetail({
  pushToast,
  setStatus,
  loading,
})

const imageMutations = useImageMutations({
  pushToast,
  setStatus,
  refresh: imageSearch.executeSearch,
})
```

## 6. イベント連鎖は維持する

`tile.vue`、`display.vue`、`edit.vue`、`Content.vue` の責務は変更しない。

`tile.vue` は現在、表示/編集モード、削除ボタン、イベント送出を担当している。

```text
edit.vue
  ↓ save emit
tile.vue
  ↓ save-detail emit
Content.vue
  ↓ save-detail emit
App.vue
  ↓ useImageMutations.handleSaveDetail
  ↓ updateImageFileDetail
  ↓ 成功時 再検索
```

---

## 7. 変更しないもの

* `backendApi.js` の API ラッパー構造
* backend 側 API
* `tile.vue` の内部編集状態
* `display.vue`
* `edit.vue`
* `ImageDetailModal.vue` の表示専用方針
* `Content.vue` のイベント emit
* `MainLayout.vue` / `AppSidebar.vue` の責務

---

## 8. 受け入れ条件

* `App.vue` から backendApi の直接 import がなくなる
* `App.vue` から検索・更新・削除・詳細画像取得の実処理が消える
* `App.vue` は composable の組み立てとテンプレート連携のみを担当する
* 既存の検索、ページング、ソートが動作する
* インポート完了後に再検索される
* タイル編集成功時に再検索される
* タイル編集失敗時に edit 表示が維持される
* 削除成功時に再検索される
* 詳細モーダルで画像表示ができる
* 詳細モーダルのタイトルクリックで親フォルダを開ける
* トースト通知が従来通り表示される
* ローディングオーバーレイが従来通り表示される

---

## 9. 注意点

このリファクタでは、**イベント連鎖を短くすることを主目的にしない**。

目的は、`App.vue` に集中しているユースケース処理を composable に逃がすこと。

ここでコンポーネントから直接 API を呼ばせると、App.vue 神クラスを倒した代わりに、各コンポーネントが小型神になります。神々の黄昏です。
