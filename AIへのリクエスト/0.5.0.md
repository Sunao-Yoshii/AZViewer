# 実装設計書

## Stable Diffusion WebUI ワイルドカード出力機能追加

## 1. 目的

画像一覧画面で選択した画像のタグを、Stable Diffusion WebUI のワイルドカードとして利用可能な `.txt` ファイルへ出力する。

出力形式は以下とする。

```text
1画像 = 1行
各行 = その画像の有効タグを ASCII カンマ + 半角スペースで接続した文字列
```

例。

```text
masterpiece, 1girl, blue eyes, smile
best quality, solo, black hair
```

本機能では、選択画像ごとのタグ行を確認・編集したうえで、以下のどちらかを実行できるようにする。

```text
- 新規 .txt ファイルとして保存
- 既存 .txt ファイル末尾へ追記
```

## 2. 前提

既存の画像一覧項目にはタグ情報が含まれる前提とする。現行の `ImageFileRepository` には `find_tags_by_image_ids(image_ids)` があり、画像 ID ごとのタグ一覧を取得できる。

また、既存の一括削除・一括移動と同じく、選択画像 ID は DB に保存せず、`App.vue` の一時状態として扱う。選択状態は `Content.vue` 経由で `tile.vue` に渡される既存構成を流用する。

## 3. 変更概要

以下を追加する。

```text
- AppHeader.vue に「ワイルドカード出力」ボタンを追加
- 選択画像 0 件ではボタンを disabled
- 選択画像のタグ情報をもとにワイルドカード出力モーダルを表示
- モーダル上で出力対象タグの ON/OFF を編集
- タグごとに選択画像内での出現件数を表示
- 有効タグだけを使って 1画像1行の出力プレビューを表示
- 新規作成 / 既存ファイル追記を選択
- 保存時に Python 側ネイティブファイルダイアログを表示
- .txt ファイルとして保存または追記
- 出力完了後、選択画像のチェックをすべて解除
```

## 5. 基本方針

## 5.1 DialogService は共有サービス化する

`DialogService` は `AppApi.__init__()` で 1 つ生成し、以下へ渡す。

```text
- ImageRegisterApi
- ImageCatalogApi
```

これにより、既存のファイル/フォルダ選択ダイアログと、今回追加する `.txt` 保存/追記先選択ダイアログを同じサービス責務に集約する。

```text
AppApi
  ├─ DialogService
  ├─ ImageRegisterApi(dialog_service)
  └─ ImageCatalogApi(dialog_service)
```

ダイアログ処理を API ごとに増殖させない。
増殖させると、半年後には「この保存ダイアログだけ挙動が違う」という小さな妖怪が住み着きます。

## 5.2 選択状態は既存仕様を流用する

本機能用の DB カラムは追加しない。

```text
image_file_data にワイルドカード出力選択状態を持たせない
```

既存の `selectedImageIds` / `selectedImageIdSet` を利用する。

## 5.3 出力対象は選択画像に紐づく既存タグ

出力対象は、選択画像に登録済みのタグのみとする。

```text
image_file_data.tags
```

または、必要に応じてバックエンドで以下を利用できる。

```text
ImageFileRepository.find_tags_by_image_ids(image_ids)
```

ただし、本仕様では **プレビューと実出力のズレを避けるため、フロントエンドで確定した previewText を保存 API に渡す**。

つまり、バックエンド側で `ids + enabledTags` から再生成しない。

## 5.4 タグがない画像の扱い

画像にタグが 0 件、または全タグ無効化により有効タグが 0 件になった画像は、出力行から除外する。

```text
空行は出力しない
```

Stable Diffusion WebUI のワイルドカードに空行が混ざると、意図せず「何も出ない候補」が混入するため。

## 5.5 タグ ON/OFF はタグ名単位

モーダル内では、画像単位ではなくタグ名単位で ON/OFF を切り替える。

例。

```text
選択画像:
A: 1girl, blue eyes, smile
B: 1girl, black hair, smile

タグ一覧:
〇 1girl (2)
〇 smile (2)
〇 blue eyes (1)
〇 black hair (1)
```

`blue eyes` を無効化した場合。

```text
A: 1girl, smile
B: 1girl, black hair, smile
```

## 5.6 出力順序

### 画像行の順序

現在の表示順を維持する。

理由は、ユーザーが画面上で選択してプレビュー確認するため、画面表示順とプレビュー順が一致している方が直感的である。

ただし、現在表示されていない選択を保持しない既存方針が前提。

### 行内タグ順序

各画像の `tags` 配列順を維持する。

DB 上でタグ順を明示保持していない場合は、既存の一覧取得時に付与されるタグ順をそのまま採用する。

## 5.7 文字コード・改行コード

出力文字コード。

```text
UTF-8
```

BOM。

```text
なし
```

改行コード。

```text
CRLF
```

Stable Diffusion WebUI では LF でも大きな問題は起きにくいが、Windows ローカルアプリとして扱うため CRLF に固定する。
ここを気分で変えると、差分確認が地味に濁ります。温泉に墨汁を一滴入れるタイプの嫌さです。

## 8.4 openWildcardExportModal

入力。

```js
openWildcardExportModal({
  ids,
  currentItems,
})
```

処理。

```text
1. ids が空なら warning toast
2. currentItems から選択 ID に一致する画像を抽出
3. 各画像の tags を取得
4. タグ出現件数を集計
5. tagItems を全件 enabled=true で初期化
6. previewText を生成
7. モーダルを表示
```

実装例。

```js
function openWildcardExportModal({ ids, currentItems }) {
  const targetIds = [...(ids ?? [])]

  if (targetIds.length === 0) {
    pushToast({
      type: 'warning',
      message: 'ワイルドカード出力する画像を選択してください。',
    })
    return
  }

  const selectedIdSet = new Set(targetIds)
  const sourceItems = (currentItems ?? [])
    .filter((item) => selectedIdSet.has(item.id))
    .map((item) => ({
      id: item.id,
      filename: item.filename,
      tags: [...(item.tags ?? [])],
    }))

  if (sourceItems.length === 0) {
    pushToast({
      type: 'warning',
      message: 'ワイルドカード出力する画像を確認できませんでした。',
    })
    return
  }

  wildcardExportModal.selectedIds = targetIds
  wildcardExportModal.sourceItems = sourceItems
  wildcardExportModal.exportMode = 'create'
  wildcardExportModal.tagItems = buildTagItems(sourceItems)

  refreshWildcardPreview()

  wildcardExportModal.show = true
}
```

## 8.5 タグ集計

タグの出現件数は「選択画像のうち、そのタグを含む画像数」とする。

```js
function buildTagItems(sourceItems) {
  const countMap = new Map()

  for (const item of sourceItems) {
    const uniqueTags = [...new Set(item.tags ?? [])]

    for (const tag of uniqueTags) {
      countMap.set(tag, (countMap.get(tag) ?? 0) + 1)
    }
  }

  return [...countMap.entries()]
    .map(([name, count]) => ({
      name,
      count,
      enabled: true,
    }))
    .sort((a, b) => {
      if (b.count !== a.count) {
        return b.count - a.count
      }

      return a.name.localeCompare(b.name)
    })
}
```

## 8.6 プレビュー生成

```js
function buildWildcardPreview(sourceItems, tagItems) {
  const enabledTagSet = new Set(
    tagItems
      .filter((tag) => tag.enabled)
      .map((tag) => tag.name)
  )

  const lines = []

  for (const item of sourceItems) {
    const tags = (item.tags ?? []).filter((tag) => enabledTagSet.has(tag))

    if (tags.length === 0) {
      continue
    }

    lines.push(tags.join(', '))
  }

  return {
    text: lines.join('\r\n'),
    lineCount: lines.length,
  }
}
```

```js
function refreshWildcardPreview() {
```

## 8.11 saveWildcardExport

処理フロー。

```text
1. previewText が空なら warning toast
2. isSaving = true
3. exportMode と previewText を backend API へ渡す
4. backend 側で保存/追記先のネイティブファイルダイアログを表示
5. キャンセルなら何もしない
6. 保存成功なら toast 表示
7. clearSelection() を実行
8. モーダルを閉じる
9. finally で isSaving = false
```

実装例。

```js
import { exportWildcardText } from '../services/backendApi'

async function saveWildcardExport() {
  const text = wildcardExportModal.previewText

  if (!text) {
    pushToast({
      type: 'warning',
      message: '出力対象のタグがありません。',
    })
    return
  }

  wildcardExportModal.isSaving = true

  try {
    const result = await exportWildcardText({
      mode: wildcardExportModal.exportMode,
      text,
    })

    if (!result?.success) {
      if (result?.data?.cancelled) {
        return
      }

      pushToast({
        type: 'danger',
        message: result?.message || 'ワイルドカード出力に失敗しました。',
      })
      return
    }

    const data = result.data ?? {}

    pushToast({
      type: 'success',
      message: `ワイルドカードを出力しました。出力行: ${data.lineCount ?? wildcardExportModal.outputLineCount} 件`,
    })

    clearSelection?.()
    closeWildcardExportModal()
  } catch (error) {
    pushToast({
      type: 'danger',
      message: 'ワイルドカード出力に失敗しました。',
    })
  } finally {
    wildcardExportModal.isSaving = false
  }
}
```

## 15.5 処理フロー

```text
1. mode を検証
2. text を検証
3. text の空行を除外し、CRLF へ正規化
4. mode=create なら保存ダイアログを表示
5. mode=append なら既存 txt ファイル選択ダイアログを表示
6. キャンセルなら cancelled=True の結果を返す
7. create なら UTF-8 で上書き保存
8. append なら既存末尾改行を確認して追記
9. 出力行数を返す
```

## 15.6 実装例

```python
from __future__ import annotations

from pathlib import Path

from backend.models.image_file import WildcardExportResult
from backend.services.dialog_service import DialogService


class WildcardExportService:
    def __init__(self, dialog_service: DialogService):
        self._dialog_service = dialog_service

    def export_text(self, mode: str, text: str) -> WildcardExportResult:
        normalized_mode = self._normalize_mode(mode)
        normalized_text = self._normalize_output_text(text)

        if not normalized_text:
            raise ValueError("出力対象のテキストがありません。")

        if normalized_mode == "create":
            selected_path = self._dialog_service.select_text_file_for_save()
        else:
            selected_path = self._dialog_service.select_text_file_for_append()

        if not selected_path:
            return WildcardExportResult(
                path="",
                mode=normalized_mode,
                line_count=0,
                cancelled=True,
            )

        output_path = Path(selected_path)

        if normalized_mode == "create":
            self._write_text(output_path, normalized_text)
        else:
            self._append_text(output_path, normalized_text)

        return WildcardExportResult(
            path=str(output_path),
            mode=normalized_mode,
            line_count=self._count_lines(normalized_text),
            cancelled=False,
        )

    def _normalize_mode(self, mode: str) -> str:
        value = str(mode or "").strip()

        if value not in ("create", "append"):
            raise ValueError("出力方式が不正です。")

        return value

    def _normalize_output_text(self, value: str) -> str:
        lines = [
            line.strip()
            for line in str(value or "").splitlines()
            if line.strip()
        ]

        return "\r\n".join(lines)

    def _count_lines(self, text: str) -> int:
        return len([line for line in text.splitlines() if line.strip()])

    def _write_text(self, path: Path, text: str) -> None:
        with path.open("w", encoding="utf-8", newline="") as file:
            file.write(text)
            file.write("\r\n")

    def _append_text(self, path: Path, text: str) -> None:
        needs_newline = False

        if path.exists() and path.stat().st_size > 0:
            with path.open("rb") as file:
                file.seek(-1, 2)
                last_byte = file.read(1)
                needs_newline = last_byte not in (b"\n", b"\r")

        with path.open("a", encoding="utf-8", newline="") as file:
            if needs_newline:
                file.write("\r\n")

            file.write(text)
            file.write("\r\n")
```

# 実装仕様書

## 画像生成元モデル管理・検索機能追加

## 1. 目的

画像ファイルの生成元モデル名を AZViewer 上で管理し、画像検索条件としてモデル名を指定できるようにする。

Stable Diffusion WebUI のメタデータに含まれる `Model: xxx` を読み取り、`image_file_data` に対して生成元モデルを紐づける。

本機能により、以下を実現する。

```text
- 画像ごとの生成元モデル名表示
- edit.vue からのモデル手動設定
- プロンプト読み取り時のモデル自動登録
- モデル名による画像検索
```

## 2. 基本方針

## 2.1 1画像に紐づくモデルは 0 または 1 件

1つの `image_file_data` に紐づく生成元モデルは最大1件とする。

```text
image_file_data : image_model = 1 : 0..1
```

DB上は中間テーブル `image_model_to_file_data` を利用するが、`image_file_data_id` に `UNIQUE` 制約を付け、1画像に複数モデルが紐づく状態を禁止する。

モデル複数持ちを許可すると、検索条件が「いずれか一致」なのか「すべて一致」なのかで仕様が増殖する。今回の用途では不要です。DBに余計な部屋を作ると、だいたい後で変な住人が入ります。

## 2.2 モデル名はタグとは別管理する

モデル名はタグとしては扱わない。

理由は以下。

```text
- モデル名は画像生成環境のメタ情報であり、画像内容タグとは意味が異なる
- タグ検索とは別にモデル検索条件として使いたい
- タグ正規化ルールとは異なる保存ルールにする必要がある
```

## 2.3 モデル名の大文字小文字は維持する

タグとは異なり、モデル名は小文字化しない。

```text
waiNSFWIllustrious_v50
```

のような物理ファイル名・モデル表示名に近い値を扱うため、保存時にケースを変更しない。

## 2.4 保存タイミング

モデル情報の保存タイミングは次の2つとする。

```text
1. edit.vue の「保存して戻る」押下時
2. AppHeader.vue の「プロンプトの読み取り」押下時
```

`edit.vue` のモデル設定ダイアログでは DB 保存しない。
既存のタグ編集と同様、編集フォーム内の一時状態として扱い、「保存して戻る」でまとめて保存する。

## 2.5 起動時の扱い

起動時に `image_model` テーブルが存在しない場合は、テーブル作成のみ行う。

自動で全画像のメタデータ読み取りまでは行わない。

理由は、起動時に全画像のメタデータを読むと、件数次第で起動処理が重くなるため。起動直後から朝礼が長いアプリは嫌われます。

モデル自動登録は、ユーザーが明示的に `AppHeader.vue` の「プロンプトの読み取り」を押下したタイミングで行う。

## 4.2 `image_model_to_file_data`

### 役割

```sql
CREATE TABLE IF NOT EXISTS image_model_to_file_data (
```

# 5. モデル名正規化仕様

## 5.1 保存時正規化

モデル名は以下の処理のみ行う。

```text
1. None は未設定扱い
2. 前後空白を trim
3. 空文字は未設定扱い
```

## 5.2 行わないこと

```text

- カンマ分割
- 括弧除去
- 重み表記除去
- 全角半角変換
```

タグ正規化とは別物として扱う。

## 5.3 最大長

```text
512文字
```

理由は、タグ最大長を 512 文字へ拡張している既存方針との整合を取りつつ、モデル名としては十分な長さであるため。

# 8. ImageMetadataService仕様

## 8.1 公開メソッド追加

```python
def extract_stable_diffusion_model_name(self, path: str) -> str | None:
    ...
```

## 8.2 役割

```text
1. 画像ファイルを開く
2. 既存の _build_metadata_list を利用してメタ情報行リストを取得する
3. 拡張子に応じて対象キーを探す
4. 対象キーの値を復元する
5. Stable Diffusion WebUI パラメータ内の Model: を抽出する
6. モデル名を返す
```

## 8.3 対象キー

| 拡張子     | 対象キー               |
| ------- | ------------------ |
| `.png`  | `parameters`       |
| `.jpg`  | `exif.UserComment` |
| `.jpeg` | `exif.UserComment` |
| `.webp` | `exif.UserComment` |
| `.avif` | `exif.UserComment` |

既存の `extract_stable_diffusion_prompt()` と同じメタ情報復元処理を使う。

## 8.4 抽出ルール

`Model:` の値を抽出する。

例。

```text
Model: waiNSFWIllustrious_v50
```

抽出結果。

```text
waiNSFWIllustrious_v50
```

## 8.5 誤抽出防止

以下は対象外とする。

```text
Model hash:
ADetailer model:
ADetailer model classes:
```

このため、単純な `if "Model" in item` ではなく、`Model:` キーを判定する。

推奨実装。

```python
import re

_MODEL_NAME_PATTERN = re.compile(
    r"(?:^|,\s*)Model:\s*([^,\n]+)"
)

def _extract_model_name(self, value: str) -> str | None:
    match = _MODEL_NAME_PATTERN.search(value or "")
    if not match:
        return None

    model_name = match.group(1).strip()
    return model_name or None
```

## 8.6 注意

`TI hashes` のように引用符内にカンマを含む項目があるため、最終行を単純に `split(",")` する実装は避ける。

CSVパーサを持ち込むほどでもないですが、雑な split はここでは地雷です。薄く埋まっているタイプの地雷です。

# 9. PromptTagImportService変更

## 9.1 既存責務

既存の `PromptTagImportService` は、タグ未登録画像だけを対象に、メタ情報からプロンプトを抽出し、タグ登録・件数集計・失敗詳細収集を行う構成である。

## 9.2 追加方針

「プロンプトの読み取り」ボタン押下時に、タグ登録とモデル登録の両方を行う。

ただし、対象条件は分ける。

```text
タグ登録対象:
tag_image_link が 1 件もない画像

モデル登録対象:
image_model_to_file_data が 1 件もない画像
```

タグ登録済みでも、モデル未設定ならモデル抽出対象とする。

## 9.3 処理フロー

```text
1. find_items_without_tags() でタグ未登録画像を取得
2. 既存通りタグ登録処理を行う
3. find_items_without_model() でモデル未設定画像を取得
4. 画像ごとに ImageMetadataService.extract_stable_diffusion_model_name(path) を呼ぶ
5. モデル名が取得できなければ modelSkippedCount 加算
6. モデル名が取得できたら repository.replace_image_model(id, model_name) を実行
7. 成功したら modelLinkedCount 加算
8. 例外発生時は modelFailedCount 加算し、失敗詳細を最大20件まで保持する
9. タグ結果とモデル結果をまとめて返す
```

## 9.4 トランザクション方針

一括全体を1トランザクションにはしない。

```text
- タグ登録は1画像単位
- モデル登録も1画像単位
- 1画像で失敗しても全体処理は継続
```

既存のプロンプトタグ登録と同じ方針に揃える。

```python
def fetch_models_for_search(self, payload):
    try:
        data = payload or {}
        keyword = str(data.get("keyword") or "").strip()
        limit = self._normalize_model_search_limit(data.get("limit"))

        repository = self._get_repository()

        models = repository.find_models_for_search(
            keyword=keyword,
            limit=limit,
        )
        total_count = repository.count_models_for_search(
            keyword=keyword,
        )

        return ApiResponse(
            success=True,
            message="",
            data={
                "models": [model.to_dict() for model in models],
                "total_count": total_count,
                "limit": limit,
            },
        ).to_dict()

    except Exception:
        return ApiResponse(
            success=False,
            message="モデル一覧を取得できませんでした。",
            data={
                "models": [],
                "total_count": 0,
                "limit": 256,
            },
        ).to_dict()
```

## 10.7 `_normalize_model_search_limit` 追加

```python
def _normalize_model_search_limit(self, value) -> int:
    try:
        limit = int(value or 256)
    except (TypeError, ValueError):
        limit = 256

    return max(1, min(limit, 256))
```

```js
async function searchModelsForSearch({ keyword = '' } = {}) {
```

# 25. 注意点

## 25.1 起動時自動抽出は行わない

今回の仕様では、`image_model` テーブルが存在しない場合でも自動抽出は行わない。

```text
起動時:
- テーブル作成のみ

ユーザー操作:
- 「プロンプトの読み取り」でモデル抽出実行
```

ここを自動化すると、初回起動が重くなる可能性がある。あと、ユーザーが「起動しただけなのに何か始まった」と感じる挙動は地味に嫌われます。

## 25.2 モデル名はタグ化しない

モデル名をタグにも追加しない。

タグ検索とモデル検索の責務を分ける。

## 25.3 中間テーブルだが実質1対1

`image_model_to_file_data` は中間テーブルだが、`UNIQUE (image_file_data_id)` により、1画像1モデルを保証する。

ここを忘れると、仕様書では1モデルなのにDBでは複数入るという、二重帳簿みたいな状態になります。

## 25.4 `Model:` 抽出は雑に split しない

`TI hashes` など、カンマを含む複雑な項目があるため、単純なカンマ分割は避ける。

`Model:` キーを正規表現で抽出する。
