# pywebview + Vue + Bootstrap アプリケーション基盤 要件定義書

## 1. 目的

本アプリケーションは、Python と pywebview を用いたデスクトップアプリケーション開発において、今後の各種業務機能・画面機能を実装するための**共通基盤**として構築する。

フロントエンドは Vue を用いて実装し、画面レイアウトは以下の共通構成を採用する。

* ヘッダ
* フッタ
* サイドバー
* メインコンテンツエリア

また、画面デザインおよびレイアウト補助のため、CSS フレームワークとして Bootstrap を採用する。

本基盤の目的は、個別機能を迅速に追加できる拡張性と、画面構成の統一性を確保することである。

## 2. 開発方針

### 2.1 基本方針

* 本アプリケーションは単体で価値を持つ完成品ではなく、将来の機能追加を前提とした基盤とする
* UI の共通枠組みを先に固定し、各機能はメインコンテンツエリアへ差し込む構成とする
* Python 側は業務ロジックとローカル処理を担当し、Vue 側は画面表示とユーザー操作を担当する
* 小規模から中規模のデスクトップツールを想定し、過剰な構成は避ける

### 2.2 設計思想

* レイアウト共通化を優先する
* 機能追加時に既存レイアウトを崩さない
* Vue コンポーネント単位で差し替えや追加ができる
* Python 連携方法を標準化する
* Bootstrap によって基本的なUI整形を迅速に行えるようにする

## 3. システム構成

### 3.1 使用技術

* バックエンド: Python
* デスクトップ表示: pywebview
* フロントエンド: Vue
* UIフレームワーク: Bootstrap
* フロントとバックエンドの橋渡し: pywebview の JavaScript API

### 3.2 構成概要

* Python がアプリケーション起動・ロジック・データ処理を担当する
* pywebview が Vue アプリケーションを内包したウィンドウを表示する
* Vue は共通レイアウトの描画と画面遷移、ユーザー操作を担当する
* Bootstrap はレイアウトおよび基本スタイルの整備に利用する

## 4. スコープ

## 4.1 本基盤に含むもの

* アプリケーション起動処理
* 共通レイアウトの表示
* ヘッダ、フッタ、サイドバー、メインエリアの固定構造
* メインコンテンツ差し替えのための画面領域
* Vue から Python を呼び出す基本的なAPI接続
* Bootstrap による基本UIスタイル
* ベースとなるディレクトリ構成

## 4.2 本基盤に含まないもの

* 個別業務機能
* 高度な認証認可
* サーバー通信前提の API 設計
* 複雑な権限管理
* 多ウィンドウ制御
* 高度なテーマ切替
* 多言語化
* 通知基盤の完成実装

ここを明確にしないと、基盤開発のはずが「全部入り starter kit」になって転びます。
**“将来使うかも”は、だいたい今は使わない**です。

## 5. 画面要件

## 5.1 共通レイアウト

画面は以下の4領域で構成する。

### ヘッダ

* アプリケーション名またはタイトルを表示する
* 必要に応じてグローバル操作領域を配置できる
* 画面上部に常時表示する

### フッタ

* ステータス表示や補助情報を表示できる
* 画面下部に常時表示する

### サイドバー

* メニューまたは画面切替導線を配置する
* 左側固定を基本とする
* 項目数増加に備え、縦スクロール可能とする

### メインコンテンツエリア

* 個別機能画面を表示する主領域
* サイドバー選択に応じて内容を切り替える
* 今後追加される機能画面は原則ここに描画する

## 5.2 レイアウト方針

* ウィンドウ全体に対して安定した表示を行う
* 各領域の責務を明確にする
* 画面サイズ変更時にも大きく破綻しない構成とする
* Bootstrap の grid / flex ユーティリティを用いて整列する

## 5.3 UI方針

* 見た目はシンプルでよい
* Bootstrap 標準コンポーネントを優先利用する
* 不要な独自CSSを増やしすぎない
* 配色や余白は最低限の統一を保つ

ここで独自CSS祭りを始めると、**Bootstrap 採用理由が“入れただけ”になります**。
それはフレームワーク導入ではなく、観葉植物です。

## 6. 機能要件

## 6.1 起動機能

* Python から pywebview を起動できること
* 起動時に Vue アプリケーションを表示できること
* 初期画面として共通レイアウトが表示されること

## 6.2 画面表示機能

* ヘッダ、フッタ、サイドバー、メインコンテンツエリアを表示できること
* 各領域が定義された位置に表示されること
* メインエリアに初期画面コンテンツを表示できること

## 6.3 メニュー切替機能

* サイドバーの選択に応じてメインエリアの表示を切り替えられること
* 切替対象は Vue コンポーネント単位で管理できること
* 将来の画面追加時に、既存構造へ容易に組み込めること

## 6.4 Python 連携機能

* Vue から Python 側の公開APIを呼び出せること
* Python の応答を Vue 側で受け取り表示できること
* エラー時にはフロント側で適切に扱えること

## 6.5 状態表示機能

* フッタまたはメイン領域で簡易ステータスを表示できること
* 処理中、成功、失敗などの基本状態を表示できること

## 7. 非機能要件

## 7.1 拡張性

* 新規画面追加時に共通レイアウトの変更を最小限に抑えること
* Vue コンポーネントを追加するだけで画面を拡張しやすいこと
* Python API を追加しやすい構造とすること

## 7.2 保守性

* フロントエンドとバックエンドを分離したディレクトリ構成とすること
* 共通コンポーネントを明確に分離すること
* 命名規則と責務分担を統一すること

## 7.3 操作性

* 画面構造が直感的に理解できること
* 主要導線がサイドバーで明確に分かること
* 過剰な画面遷移を避けること

## 7.4 性能

* 起動時間は過度に長くならないこと
* 画面切替は実用上ストレスのない速度で行えること
* 重い処理が必要な場合、UI 側で状態を表示できること

## 7.5 一貫性

* 全画面で共通レイアウトを維持すること
* ボタン、フォーム、テーブル等の基本UIは Bootstrap ベースで統一すること

## 8. コンポーネント要件

## 8.1 共通コンポーネント

* Header
* Footer
* Sidebar
* MainLayout

## 8.2 画面コンポーネント

* Dashboard または Home
* SamplePage などのプレースホルダ画面

## 8.3 コンポーネント設計方針

* レイアウト部品と機能画面部品を分離する
* サイドバー項目と表示コンポーネントの対応関係を明確化する
* 共通部品の責務を増やしすぎない

ありがちな事故として、Sidebar が
**メニュー表示、権限制御、ルーティング、状態保持、通知バッジ、折りたたみ制御** を全部持ち始めます。
そうなると Sidebar ではなく、もはや地方行政です。
最初は「表示と選択」くらいに留めるのが無難です。

## 9. Python-Vue 連携要件

## 9.1 連携方式

* pywebview の API 機能を用いて、JavaScript から Python の公開メソッドを呼び出す
* 公開メソッドは明示的に定義する

## 9.2 API設計方針

* 返却形式は統一する
* 正常系と異常系を画面側で判別可能にする
* UI専用の細かすぎる API を乱立させない

### 応答形式例

```json
{
  "success": true,
  "message": "",
  "data": {}
}
```

異常時:

```json
{
  "success": false,
  "message": "エラー内容",
  "data": null
}
```

## 9.3 想定ベースAPI

* `initialize()`
* `get_app_info()`
* `get_menu_definitions()`
* `health_check()`

業務機能が未実装でも、この程度のベースAPIがあると土台として整います。

## 11. Bootstrap 採用要件

* Bootstrap を標準UIフレームワークとして採用する
* レイアウト、フォーム、ボタン、ナビゲーションに Bootstrap を利用する
* 原則として Bootstrap の設計思想に沿って実装する
* 独自CSSは不足分の補完に限定する
* Bootstrap クラスと Vue コンポーネントの責務を混在させすぎない

補足すると、Bootstrap 採用は良い判断です。
pywebview + Vue の土台段階では、**最初からデザイン凝るより “壊れにくい既製品” を使う方が強い**です。
Tailwind でもできますが、基盤段階では Bootstrap の方が「枠を作る」には雑に強いです。

## 12. 制約事項

* デスクトップアプリとしてローカル動作を前提とする
* 基本構成は単一ウィンドウとする
* Web サービスとしての公開は前提としない
* まずは基盤構築を優先し、業務固有機能は後続実装とする

## 13. 受け入れ条件

以下を満たした場合、本基盤は完成とみなす。

* pywebview でアプリが起動する
* Vue 画面が表示される
* 画面がヘッダ、フッタ、サイドバー、メインコンテンツエリアで構成される
* Bootstrap により基本的な見た目が整っている
* サイドバー選択によりメインコンテンツが切り替わる
* Vue から Python API を1つ以上呼び出せる
* Python の応答結果を画面に表示できる

## 3. コンポーネント分割方針

本機能の実装にあたっては、実ファイル配置やディレクトリ構成を本設計書では固定しない。
これは、既存のアプリケーション基盤側で定義されている土台構成と不整合を起こさないようにするためである。

その代わり、本機能は**役割ごとに明確に分離されたコンポーネント／モジュールとして設計する**ことを基本方針とする。

### 3.1 分割方針

以下の責務単位で分離する。

* DB 接続管理
* DB 初期化
* 起動時クリーンアップ
* ファイル走査
* 画像ファイル登録
* ネイティブダイアログ制御
* サイドバー上部 UI
* エラーモーダル UI
* API 呼び出し境界

### 3.2 分離の目的

* 責務の混在を防ぐ
* 土台側構成へ柔軟に組み込みやすくする
* 将来的な機能追加時に影響範囲を局所化する
* UI と業務ロジック、DB アクセスを分離する

### 3.3 設計原則

* Vue 側は表示とユーザー操作を担当する
* Python 側はファイル処理、DB 処理、トランザクション管理を担当する
* DB アクセスは接続管理責務を通じて行う
* 登録処理はファイル選択とフォルダ選択で共通化する
* 各機能は再利用可能な責務単位で分解する
* 実配置は土台側の既存規約に従って決定する

## 4.2 テーブル定義

### テーブル名

`image_file_data`

### カラム一覧

| カラム名        | 型       | NOT NULL | 主キー | 備考                            |
| ----------- | ------- | -------: | --: | ----------------------------- |
| id          | INTEGER |        ○ |   ○ | AUTOINCREMENT                 |
| filename    | TEXT    |        ○ |     | ファイル名                         |
| path        | TEXT    |        ○ |     | 絶対パス、一意                       |
| folder      | TEXT    |        ○ |     | 直近フォルダ名                       |

| comment     | TEXT    |          |     | 改行可                           |

### 制約

* `PRIMARY KEY AUTOINCREMENT (id)`
* `UNIQUE (path)`
* `CHECK (rating IN ('General', 'R-15', 'R-18', 'R-18G'))`
* `CHECK (is_checked IN (0, 1))`
* `CHECK (is_favorite IN (0, 1))`

### CREATE TABLE 想定

```sql
CREATE TABLE IF NOT EXISTS image_file_data (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    filename TEXT NOT NULL,
    path TEXT NOT NULL UNIQUE,
    folder TEXT NOT NULL,
    rating TEXT NOT NULL CHECK (rating IN ('General', 'R-15', 'R-18', 'R-18G')),
    is_checked INTEGER NOT NULL CHECK (is_checked IN (0, 1)),
    is_favorite INTEGER NOT NULL CHECK (is_favorite IN (0, 1)),
    comment TEXT NULL
);
```

## 5. 接続管理設計

## 5.1 接続管理方針

* アプリケーション全体で **ConnectionManager** を 1 つ保持する
* ConnectionManager は SQLite 接続インスタンスを内部保持する
* 必要に応じて同一接続を返す
* DB アクセスは Repository または Service から直接 `sqlite3.connect()` せず、必ず ConnectionManager 経由で行う

## 5.2 ConnectionManager の責務

* DB ファイルパスの解決
* SQLite 接続生成
* 接続取得
* 接続クローズ
* 二重接続生成防止

## 5.3 想定インターフェース

```python
class ConnectionManager:
    def initialize(self) -> None
    def get_connection(self)
    def close(self) -> None
```

## 5.4 設計上の注意

* 接続生成は起動時に 1 回のみ行う
* 終了時に必ず close する
* 将来の拡張を考慮し、接続設定は ConnectionManager に閉じ込める
* 他クラスが DB パスや接続生成方法を知らない構造にする

ここを破ると、後で `repository_a.py` と `repository_b.py` がそれぞれ勝手に接続を開き始め、
設計が「共有管理」から「野生の SQLite」になります。

# 設計変更仕様書

## 初回画面表示後の画面ロック方式による整合性確認実行への変更

## 1. 目的

本仕様書は、起動時整合性確認処理の実行方式を見直し、SQLite のスレッド制約を回避しつつ、初回画面表示までの体感速度を維持することを目的とする。

主な目的は以下の通りとする。

* 初回画面を早期に表示する
* SQLite のスレッド制約に起因するエラーを回避する
* DB 接続運用を単純化する
* 整合性確認中の不整合なユーザー操作を防止する

## 2. 変更概要

従来検討では、整合性確認を動機処理として実行する案を検討していた。
しかし、それまで真っ白な画面が表示されていたため、その挙動を改善し、画面表示後に同期で整合性確認を行うようにする。

* **初回画面は先に表示する**
* **画面表示直後に、操作を禁止するロック画面を前面表示する**
* **整合性確認はバックグラウンドではなく、同一スレッド上で同期実行する**
* **整合性確認完了後にロック画面を解除する**
* **DB 接続は単一コネクションを使い回す**

## 3. 非同期ではない理由

参照記事では、Python の `sqlite3` を用いたインメモリ DB において、異なるスレッドから同一 Connection を参照すると `ProgrammingError` が発生すること、および別コネクションでは同一のインメモリ内容を参照できないことが示されている。

また、Python 公式ドキュメントでは、`check_same_thread=True` が既定値であり、この場合は Connection を作成したスレッド以外で利用すると `ProgrammingError` が発生するとされている。複数スレッド共有を有効化することは可能だが、その場合は利用者側で書き込みを直列化する必要がある。 

本システムでは tempfile を利用しない方針であるため、スレッドをまたぐ DB 利用を避け、**同一スレッド・同一コネクション・同期実行** に統一する。

## 4. 起動処理方式

### 4.1 起動シーケンス（変更後）

```text
1. アプリケーション起動
2. ./data ディレクトリ存在確認
3. ConnectionManager 初期化
4. 単一 DB コネクション生成
5. テーブル存在確認
6. 必要テーブル作成
7. UI 初回画面を表示
8. 画面ロック用オーバーレイを最前面表示
9. 同期的に整合性確認を実行
10. 結果に応じて必要時のみ通知
11. オーバーレイを除去
12. 通常操作可能状態へ遷移
```

### 4.2 動作要件

* 初回画面の描画自体は整合性確認前に完了させる
* ただし、整合性確認完了まではユーザー操作を受け付けない
* 整合性確認は UI 表示後に同期実行する
* 整合性確認処理中は他の DB 操作を開始しない

## 5. 整合性確認処理

### 5.1 処理内容

整合性確認処理の業務内容自体は変更しない。

1. DB から全 `path` を取得
2. 各 `path` について実ファイル存在確認
3. 存在しないファイルのレコードを削除

既存設計上の起動時クリーンアップの内容はこの処理である。 

### 5.2 実行方式

* バックグラウンドスレッドでは実行しない
* 初回描画後に、同一スレッドで同期実行する
* 整合性確認中は UI をロックする

### 5.3 実行制御

* 起動ごとに 1 回のみ実行する
* 多重実行は禁止する
* オーバーレイ表示中は他の起動時処理を発火しない

## 6. DB 接続運用方針

### 6.1 基本方針

DB 接続は **アプリケーション全体で 1 つのみ保持する**。

* `ConnectionManager` は単一の SQLite Connection を内部保持する
* 常に同一 Connection を返却する
* 追加 Connection の生成は行わない

この方針は既存の基本設計に近い。 

### 6.2 ConnectionManager の責務

* DB ファイルパスの解決
* SQLite コネクション生成
* 同一コネクション返却
* コネクションクローズ
* 二重接続生成防止

### 6.3 想定インターフェース

```python
class ConnectionManager:
    def initialize(self) -> None
    def get_connection(self)
    def close(self) -> None
```

### 6.4 運用上の注意

* Repository / Service から直接 `sqlite3.connect()` を呼び出さない
* DB 操作はすべて ConnectionManager 経由とする
* 起動時整合性確認中に別 DB 操作を割り込ませない
* Connection の生成・利用・クローズは同一スレッド内で完結させる

## 7. 画面ロック仕様

### 7.1 目的

* 整合性確認中のユーザー操作を禁止する
* 整合性確認前後で UI 状態の不整合を防止する
* 初回表示後に「処理中」であることを明示する

### 7.2 表示方式

画面全体を覆う半透明オーバーレイを最前面に表示する。

#### 要件

* `position: fixed`
* 全画面を覆う
* 十分に高い `z-index` を設定する
* 背面クリックを遮断する
* 読み込み中メッセージを表示する
* 半透明オーバーレイ自体もコンポーネント化し、メッセージは呼び出し元から任意に設定できるものとする

### 7.3 表示文言例

* `ファイル状態確認中です...`
* `起動処理を完了しています。しばらくお待ちください。`

### 7.4 UI 挙動

* オーバーレイ表示中はクリック・キーボード操作を受け付けない
* オーバーレイ表示中は一覧操作、登録操作、メニュー操作を禁止する
* 整合性確認完了後にオーバーレイを除去する

## 8. フロントエンド通知仕様

### 8.1 通知方針

整合性確認結果は常時通知しない。
以下の場合のみトースト通知を表示する。

### 8.2 通知条件

#### 通知する場合

* 削除件数が 1 件以上
* 整合性確認処理が失敗した場合

#### 通知しない場合

* 正常終了かつ削除件数 0 件

### 8.3 通知内容

#### 削除時

* `ファイル状態確認が完了しました`
* `存在しないファイルの登録情報を X 件削除しました`

#### エラー時

* `ファイル状態確認の実行中にエラーが発生しました`

### 8.4 表示方式

* トースト通知を使用する
* モーダル表示は行わない
* オーバーレイ解除後に表示する

## 9. エラー処理

### 9.1 基本方針

* 整合性確認失敗は致命的障害として扱わない
* エラー発生時も、オーバーレイは必ず解除する
* ログ出力とトースト通知を行う

### 9.2 処理方針

* `try/finally` 等によりオーバーレイ解除を保証する
* 整合性確認処理失敗時も UI を操作可能状態へ戻す
* 必要に応じて失敗通知を表示する

### 9.3 想定エラー

* ファイルアクセスエラー
* DB 接続エラー
* SQL 実行エラー
* 起動処理途中例外
* UI オーバーレイ解除漏れ

### 10.1 影響対象

* 起動処理
* ConnectionManager
* StartupCleanupService
* フロントエンド起動直後 UI
* 通知機構

### 10.2 非影響範囲

## 11. 受け入れ条件

### 起動表示

* 初回画面が表示されること
* 表示直後に画面全体を覆うオーバーレイが表示されること
* オーバーレイ表示中はユーザー操作できないこと

### 整合性確認

* 起動時に同期的に整合性確認が実行されること
* 実在しないファイルのレコードが削除されること
* 起動ごとに 1 回のみ実行されること

### 接続管理

* 単一コネクションのみを使用すること
* 別スレッド用コネクションを生成しないこと
* Connection の生成・利用・クローズが同一スレッドで完結すること

### 通知

* 削除件数が 1 件以上の場合のみトースト通知されること
* 実行失敗時のみトースト通知されること
* 削除件数 0 件の正常終了では通知されないこと

### 終了制御

* 正常終了時にオーバーレイが解除されること
* 異常終了時もオーバーレイが解除されること

## 12. 注意点

* 本方式は「初回描画は速い」が、「操作可能になるまでの待ち時間」は残る
* 実質的には、**真っ白待機をやめて、画面を見せた上で待たせる** 方式である
* DB と UI の整合性は保ちやすいが、整合性確認の所要時間が長い場合は体感的に重く感じる可能性がある
* そのため、オーバーレイ文言やローディング表示は手抜きしない方がよいです。無言で覆うと、ただのアプリ反抗期に見えます

本変更は、非同期バックグラウンド処理によるスレッド複雑性を避け、

tempfile を用いた別方式は採用しない。

## 1. 目的

画像一覧画面のタイル表示サイズを拡張し、タイル内簡易編集フォームでコメントまで編集できるようにする。
また、`ImageDetailModal.vue` は編集機能を廃止し、画像表示専用ダイアログとして再定義する。

## 2. 対象ファイル

### フロントエンド

* `frontend/src/components/tile/tile.vue`
* `frontend/src/components/tile/display.vue`
* `frontend/src/components/tile/edit.vue`
* `frontend/src/components/tile/ImageDetailModal.vue`
* `frontend/src/components/layout/Content.vue`
* `frontend/src/App.vue`
* `frontend/src/styles/app.css`

### バックエンド

* `backend/services/thumbnail_cache_service.py`
* `backend/api/image_catalog_api.py`
* `backend/repositories/image_file_repository.py`

## 3. 変更概要

* `tile.vue` の幅を 300px 固定へ変更する
* サムネイル生成サイズを長辺 300px へ変更する
* 既存サムネイルキャッシュの自動再生成は行わない
* `edit.vue` に `comment` 編集項目を追加する
* `edit.vue` の `rating` を Bootstrap 5.3 の radio button group に変更する
* `ImageDetailModal.vue` を画像表示専用モーダルへ変更する
* `ImageDetailModal.vue` からフォーム類、Save ボタン、フッター、右側領域を削除する
* `ImageDetailModal.vue` のタイトルを `filename` から `path` へ変更する

## 4. tile.vue 仕様変更

### 4.1 タイル幅

`tile.vue` のカード幅を 300px 固定とする。

```css
.card.image-tile,
.image-tile {
  width: 300px;
}
```

### 4.2 レイアウト

* 高さはコンテンツ内容に応じて可変とする
* カラム数指定は行わない
* 既存通り、横方向に敷き詰め、横スクロールが発生しない範囲で折り返す
* タイル間余白は既存仕様を維持する
* 既存の一覧表示レイアウトを変更しない

## 5. サムネイル仕様変更

### 5.1 生成サイズ

`ThumbnailCacheService.create_thumbnail()` の生成サイズを長辺 300px に変更する。

```python
image.thumbnail((300, 300))
```

### 5.2 生成ルール

* 縦横比を維持する
* 拡大は行わない
* 短辺方向は既存通り中央寄せ表示とする
* 生成形式は既存通り PNG
* 保存先は既存通り `data/thumbs/{id}.png`

### 5.3 既存キャッシュ

* 既存サムネイルキャッシュの自動再生成処理は実装しない
* 既存キャッシュは手動削除前提とする
* 正式リリース前のため、移行処理は不要とする

## 6. edit.vue 仕様変更

### 6.1 編集対象

`edit.vue` の編集対象を以下に変更する。

* `rating`
* `is_checked`
* `is_favorite`
* `comment`

既存仕様では `comment` は編集対象外だったが、本仕様で編集対象に追加する。

### 6.2 comment 入力

`comment` は textarea として表示する。

#### 入力仕様

* 最大長: 255 文字
* 改行許可
* 空文字許可
* 空文字保存時は DB 上 `NULL` として保存する
* 保存 payload には編集後の `comment` を含める

#### HTML 例

```html
<textarea
  v-model="form.comment"
  class="form-control form-control-sm"
  maxlength="255"
  rows="3"
></textarea>
```

### 6.3 rating 入力

`rating` は Bootstrap 5.3 の radio button group を使用する。

#### 選択肢

* `General`
* `R-15`
* `R-18`
* `R-18G`

#### 制約

* 未選択は禁止
* 選択解除は禁止
* 常に 1 つの値が選択されていること
* DB の `NOT NULL` / `CHECK` 制約に違反しないこと

#### 実装方針

* `input type="radio"` と `label.btn` の組み合わせで構成する
* `v-model` は `form.rating` に紐づける
* radio input の `id` は rating 値ベースでよい
* Bootstrap の対象仕様は「チェックボックスとラジオボタングループ」とする

### 6.4 内部状態

`props.item` の値をコピーして内部フォーム状態を作成する。

```js
const form = reactive({
  rating: props.item.rating,
  is_checked: props.item.is_checked,
  is_favorite: props.item.is_favorite,
  comment: props.item.comment ?? "",
})
```

### 6.5 変更判定

`isDirty` の判定対象に `comment` を追加する。

```js
const isDirty = computed(() => (
  form.rating !== props.item.rating ||
  form.is_checked !== props.item.is_checked ||
  form.is_favorite !== props.item.is_favorite ||
  form.comment !== (props.item.comment ?? "")
))
```

### 6.6 戻るボタン仕様

| 状態   | 表示文言   | 動作            |
| ---- | ------ | ------------- |
| 未変更  | 表示に戻る  | `cancel` emit |
| 変更あり | 保存して戻る | `save` emit   |

### 6.7 保存 payload

```json
{
  "id": 1,
  "rating": "General",
  "is_checked": 1,
  "is_favorite": 0,
  "comment": "memo"
}
```

## 7. 保存処理フロー

```text
edit.vue
  ↓ save emit
tile.vue
  ↓ save-detail emit
Content.vue
  ↓ save-detail emit
App.vue
  ↓ update_image_file_detail API
  ↓ 成功時: 再検索
  ↓ 成功時: edit状態解除
  ↓ 失敗時: トースト表示、edit状態維持
```

### 7.1 成功時

* `update_image_file_detail` を呼び出す
* 成功時は再検索する
* 再検索によりタイル状態は通常表示へ戻る

### 7.2 失敗時

* edit 表示を維持する
* 入力内容を保持する
* トースト通知を表示する
* 再検索は行わない

## 8. ImageDetailModal.vue 再定義

### 8.1 役割

`ImageDetailModal.vue` は画像表示専用モーダルとする。
編集機能は持たない。

### 8.2 削除する機能

以下を削除する。

* `rating` 編集フォーム
* `is_checked` 編集フォーム
* `is_favorite` 編集フォーム
* `comment` 編集フォーム

### 8.3 タイトル


長いパスでレイアウトが崩れないようにする。


### 8.4 本文レイアウト

* モーダル本文は画像表示領域のみとする
* 右側領域は設けない
* フォーム領域は設けない

### 8.5 画像表示

* 画像クリックによる縮小表示 / 原寸表示の切替は既存仕様を維持する
* 画像表示領域はモーダル本文全体を使用する

## 9. App.vue 側の変更

### 9.1 維持する処理

`handleSaveDetail(payload)` は削除しない。
`edit.vue` からの保存要求で引き続き使用する。

### 9.2 削除・変更する接続

* `ImageDetailModal.vue` への `isSaving` prop 渡しを削除する
* `ImageDetailModal.vue` からの `save` イベント受信を削除する
* 詳細モーダルでは `open` / `close` / 画像読み込みのみ扱う

### 10.1 update_image_file_detail

既存の詳細更新 API を継続利用する。

#### 入力

```json
  "is_favorite": 0,

```

### 10.2 comment 正規化

* `comment` が空文字の場合、DB 保存値は `NULL` とする
* `comment` が文字列の場合、最大 255 文字以内で保存する
* 改行は許可する

## 11. 受け入れ条件

### 11.1 タイル

* tile 幅が 300px で表示されること
* 既存同様、横スクロールせず折り返し表示されること
* タイル間余白が既存から変わらないこと
* タイル高さは内容に応じて可変であること

### 11.2 サムネイル

* 新規生成サムネイルが長辺 300px になること
* 縦横比が維持されること
* 小さい画像が拡大されないこと
* 既存キャッシュ自動再生成処理が追加されていないこと

### 11.3 edit.vue

* `comment` textarea が表示されること
* `comment` が最大 255 文字であること
* `comment` の改行が入力できること
* `comment` 空文字保存時に DB 上 `NULL` として保存されること
* `comment` 変更時も「保存して戻る」扱いになること

# 実装仕様書

## App.vue 分割リファクタリング

## 1. 目的

`App.vue` が保持している検索、詳細表示、更新、削除、通知、ローディング、backend API 呼び出しの責務を分離し、`App.vue` を画面構成の統括に限定する。

現状 `App.vue` は検索状態、一覧データ、詳細モーダル、トースト、ローディング、backend API 呼び出しを統括しているため、責務が集中している。

## 2. 基本方針

### 2.1 backendApi を各コンポーネントから直接呼ばない

`backendApi.js` は pywebview API 呼び出しのラッパーとして維持する。

ただし、以下のような呼び出しは禁止する。

```js
// NG
import { searchImageFiles } from '@/services/backendApi'
```

```vue
<!-- NG: tile.vue / Content.vue / ImageDetailModal.vue から直接 API 呼び出し -->
```

### 2.2 API 呼び出しは composable に集約する

コンポーネントはイベントを emit するだけにし、業務処理は composable に移す。

これは、既存方針である「tile にロジックを持たせない」「API呼び出しは親へ集約」「一方向データフロー維持」を崩さないため。

## 3. 追加ディレクトリ

```text
frontend/src/composables/
  useAppStatus.js
  useToast.js
  useLoadingOverlay.js
  useImageSearch.js
  useImageDetail.js
  useImageMutations.js
```

## 4. 分離後の責務

## 4.1 App.vue

### 責務

* レイアウト構成
* composable の組み立て
* 子コンポーネントへの state / handler 受け渡し
* 初期ロード呼び出し

### 非責務

* backendApi の直接呼び出し
* 検索 payload 正規化
* サムネイル URL 付与
* 詳細画像取得
* 更新 API 呼び出し
* 削除 API 呼び出し
* トースト配列管理
* ローディング状態の直接制御

## 4.2 useAppStatus.js

### 責務

フッター表示用ステータスを管理する。

### 移動対象

`App.vue` から以下を移動する。

```js
setStatus(type, message)
```

### 公開値

```js
return {
  status,
  setStatus,
}
```

## 4.3 useToast.js

### 責務

トースト通知を管理する。

### 移動対象

`App.vue` から以下を移動する。

```js
pushToast(notification)
```

### 公開値

```js
return {
  toasts,
  pushToast,
  removeToast,
}
```

## 4.4 useLoadingOverlay.js

### 責務

全画面ローディングオーバーレイの状態を管理する。

### 公開値

```js
return {
  loadingOverlay,
  showLoading,
  hideLoading,

}
```

### 想定状態

```js
const loadingOverlay = reactive({

```

## 4.5 useImageSearch.js

### 責務

画像検索、ページング、ソート、サムネイル URL 付与を担当する。

現状 `App.vue` は `normalizeSearchPayload`、`applySearchResult`、`addThumbnailUrls`、`addThumbnailUrl`、`executeSearch` を持つため、これらを本 composable に移動する。

### 移動対象

```js
normalizeSearchPayload(overrides = {})
applySearchResult(data, appliedPayload)
addThumbnailUrls(items)
addThumbnailUrl(item)
executeSearch(overrides = {}, overlay = false)
handleSearch()
handlePageChange(page)
handlePageSizeChange(pageSize)
handleSortChange(sort)
handleImportComplete()
```

### 使用 API

```js
searchImageFiles(payload)
fetchLocalImageThumb(id)
```

### 公開値

```js
return {
  filters,
  searchResult,
  isSearching,
  currentSearchPayload,

  executeSearch,
  handleSearch,
  handleImportComplete,
  handlePageChange,
  handlePageSizeChange,
  handleSortChange,
}
```

## 4.6 useImageDetail.js

### 責務

詳細モーダル表示、画像本体取得、フォルダオープンを担当する。

`ImageDetailModal.vue` は画像表示専用であり、詳細モーダルでは open / close / 画像読み込みのみ扱う方針。

### 移動対象

```js
handleOpenDetail(item)
handleCloseDetail()
handleOpenContainingFolder(path)
```

### 使用 API

```js
fetchLocalImage(path)
openContainingFolder(path)
```

### 公開値

```js
return {
  selectedItem,
  detailImageUrl,
  isLoadingImage,

  handleOpenDetail,
  handleCloseDetail,
  handleOpenContainingFolder,
}
```

## 4.7 useImageMutations.js

### 責務

画像情報の更新、削除を担当する。

既存仕様では、削除・保存後は API 呼び出し後に再検索する。

### 移動対象

```js
handleDelete(id)
handleSaveDetail(payload)
```

### 使用 API

```js
deleteImageFile(id)
updateImageFileDetail(payload)
```

### 公開値

```js
return {
  handleDelete,
  handleSaveDetail,
}
```

### 依存注入

```js
useImageMutations({
  pushToast,
  setStatus,
  refresh,
})
```

`refresh` には `useImageSearch().executeSearch` を渡す。

## 5. App.vue の組み立て例

```js
const { status, setStatus } = useAppStatus()
const { toasts, pushToast, removeToast } = useToast()
const loading = useLoadingOverlay()

const imageSearch = useImageSearch({
  pushToast,
  setStatus,
  loading,
})

const imageDetail = useImageDetail({
  pushToast,
  setStatus,
  loading,
})

const imageMutations = useImageMutations({
  pushToast,
  setStatus,
  refresh: imageSearch.executeSearch,
})
```

## 6. イベント連鎖は維持する

`tile.vue`、`display.vue`、`edit.vue`、`Content.vue` の責務は変更しない。

`tile.vue` は現在、表示/編集モード、削除ボタン、イベント送出を担当している。

```text
edit.vue
  ↓ save emit
tile.vue
  ↓ save-detail emit
Content.vue
  ↓ save-detail emit
App.vue
  ↓ useImageMutations.handleSaveDetail
  ↓ updateImageFileDetail
  ↓ 成功時 再検索
```

## 7. 変更しないもの

* `backendApi.js` の API ラッパー構造
* backend 側 API
* `tile.vue` の内部編集状態
* `display.vue`
* `edit.vue`
* `ImageDetailModal.vue` の表示専用方針
* `Content.vue` のイベント emit
* `MainLayout.vue` / `AppSidebar.vue` の責務

## 8. 受け入れ条件

* `App.vue` から backendApi の直接 import がなくなる
* `App.vue` から検索・更新・削除・詳細画像取得の実処理が消える
* `App.vue` は composable の組み立てとテンプレート連携のみを担当する
* 既存の検索、ページング、ソートが動作する
* インポート完了後に再検索される
* タイル編集成功時に再検索される
* タイル編集失敗時に edit 表示が維持される
* 削除成功時に再検索される
* 詳細モーダルで画像表示ができる
* 詳細モーダルのタイトルクリックで親フォルダを開ける
* トースト通知が従来通り表示される
* ローディングオーバーレイが従来通り表示される

## 9. 注意点

このリファクタでは、**イベント連鎖を短くすることを主目的にしない**。

目的は、`App.vue` に集中しているユースケース処理を composable に逃がすこと。

ここでコンポーネントから直接 API を呼ばせると、App.vue 神クラスを倒した代わりに、各コンポーネントが小型神になります。神々の黄昏です。

---

既存のファイル/フォルダのドラッグ&ドロップ受付は完了時のみ再検索という仕様ですが、ドロップ時点から取り込み完了までの間、ローディング画面を表示するようにしてください。


- drag イベントでローディング画面を出してもイベント自体は取り込みの完了時点で発生するので、ローディング開始直前に backend 側から独自イベントで画面に通知してローディング画面表示を行うようにしてください。
- ローディング画面解除は画像の取り込み完了時に backend から azviewer:import-complete イベントを発生させているので、これをトリガーとしてロック解除します。
- backend 側はインポート失敗時でも新規にイベントを発生させるものとし、画面側ではこの場合もローディング画面があれば解除するようにしてください。
- エラー内容はイベント内に含み、エラートーストの形で通知してください。

## 画像タグ管理機能追加

## 1. 目的

画像に対してタグ情報を付与・表示・編集できるようにする。
タグは将来のタグ検索を前提とし、`tag` マスタテーブルと `tag_image_link` リンクテーブルで管理する。

既存の画像一覧・更新系は `ImageCatalogApi`、`ImageFileRepository`、`ImageFileListItem` を中心に構成されているため、この流れにタグ情報を追加する。 

## 2. 対象ファイル

### バックエンド

```text
backend/models/image_file.py
backend/repositories/image_file_repository.py
backend/api/image_catalog_api.py
```

必要に応じて追加。

```text
backend/services/tag_normalize_service.py
```

### フロントエンド

```text
frontend/src/components/tile/display.vue
frontend/src/components/tile/edit.vue
frontend/src/components/tile/tile.vue
frontend/src/composables/useImageMutations.js
frontend/src/services/backendApi.js
```

現在の構成上、タイル表示・編集は `display.vue` / `edit.vue` / `tile.vue` に分離済みである。

## 3. DB仕様

### 3.1 tag

```sql
CREATE TABLE IF NOT EXISTS tag (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL UNIQUE,
    CHECK (length(name) <= 128)
);
```

### 3.2 tag_image_link

```sql
CREATE TABLE IF NOT EXISTS tag_image_link (
    image_file_id INTEGER NOT NULL,
    tag_id INTEGER NOT NULL,
    PRIMARY KEY (image_file_id, tag_id),
    FOREIGN KEY (image_file_id)
        REFERENCES image_file_data(id)
        ON DELETE CASCADE,
    FOREIGN KEY (tag_id)
        REFERENCES tag(id)
        ON DELETE CASCADE
);
```

### 3.3 削除時

`image_file_data` 削除時、`tag_image_link` は外部キー制約で削除される。

`tag` は削除しない。
未使用タグの掃除は今回対象外。

## 4. タグ正規化仕様

### 4.1 入力分割

`,` のみ区切り文字とする。

```text
"girl, blue eyes, smile"
→ ["girl", "blue eyes", "smile"]
```

全角カンマ `，`、読点 `、` は区切り文字扱いしない。

### 4.2 正規化順序

```text
1. "," で分割
2. trim
3. 禁止文字 []{}() を除去
4. 末尾の :数値 を除去
5. trim 再実行
6. 小文字化
7. 空文字除去
8. 128文字超過はエラー
9. 正規化後重複は黙って統合
```

### 4.3 禁止文字

以下を除去する。

```regex
[\[\]\{\}\(\)]
```

例。

```text
"(masterpiece)" → "masterpiece"
"[blue eyes]"  → "blue eyes"
```

### 4.4 ウェイト除去

末尾が `:数値` の場合のみ除去する。

```regex
:\d+(?:\.\d+)?$
```

例。

```text
girl:1.2       → girl
blue eyes:0.8  → blue eyes
artist:name    → artist:name
tag:abc        → tag:abc
```

### 6.3 replace_tags 処理

```text

2. tags を順に処理
3. tag に存在しなければ INSERT
4. tag_id を取得
```

### 6.4 search 結果へのタグ付与

`search(...)` で画像一覧を取得後、取得した `id` 一覧からタグをまとめて取得する。

```text

2. items を生成
3. item ids を抽出
4. find_tags_by_image_ids(ids)
5. 各 ImageFileListItem.tags に設定
```

N+1クエリは禁止。ここで1件ずつタグ取得すると、検索が「小刻みに足を引っ張る妖怪」になる。

### 8.5 addTagsFromInput

```js
function normalizeTagText(value) {
  return value
    .trim()
    .replace(/[\[\]\{\}\(\)]/g, '')
    .replace(/:\d+(?:\.\d+)?$/, '')
    .trim()
    .toLowerCase()
}

function addTagsFromInput() {
  const normalizedTags = form.tagInput
    .split(',')
    .map(normalizeTagText)
    .filter(Boolean)

  const tooLong = normalizedTags.find((tag) => tag.length > 128)
  if (tooLong) {
    // トーストまたはインラインエラー
    return
  }

  form.tags = [...new Set([...form.tags, ...normalizedTags])]
  form.tagInput = ''
}
```

### 8.6 removeTag

```js
function removeTag(tag) {
  form.tags = form.tags.filter((value) => value !== tag)
}
```

### 8.7 isDirty

```js
function normalizeTagsForCompare(tags) {

const isDirty = computed(() => (
```

## 9. 保存フロー

既存フローを維持する。

```text
edit.vue
  ↓ save emit
tile.vue
  ↓ save-detail emit
Content.vue
  ↓ save-detail emit
App.vue
  ↓ useImageMutations.handleSaveDetail
  ↓ updateImageFileDetail
  ↓ 成功時 再検索
```

更新・削除は `useImageMutations.js` に集約されているため、コンポーネントから直接 API を呼ばない方針を維持する。

失敗時も既存通り。

```text
edit表示維持
入力内容保持
トースト通知
再検索しない
```

この失敗時挙動は既存仕様とも一致する。

## 11. 受け入れ条件

### DB

* `tag` テーブルが作成されること
* `tag_image_link` テーブルが作成されること
* 同一タグ名が `tag` に重複登録されないこと
* 同一画像に同一タグリンクが重複登録されないこと
* 画像削除時に `tag_image_link` が削除されること

### 正規化

* タグは小文字で保存されること
* `[]{}()` が除去されること
* 末尾の `:1.2` 形式が除去されること
* 全角カンマ、読点では分割されないこと
* 正規化後の重複タグは黙って1つに統合されること
* 128文字超過タグは保存できないこと

### 表示

* `display.vue` にタグが灰色 badge で表示されること
* タグ0件時に不要な空欄が出ないこと

### 編集

* `edit.vue` でタグを入力できること
* Enterでタグ追加できること
* `,` 区切りで複数タグ追加できること
* xボタンでタグ削除できること
* 「保存して戻る」時のみDB反映されること
* 保存失敗時に入力中タグが消えないこと

### 一覧取得

* `search_image_files` の各 item に `tags` が含まれること
* タグ取得がN+1クエリにならないこと
