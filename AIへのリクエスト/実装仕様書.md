# 実装仕様書

## 画像生成元モデル管理・検索機能追加

## 1. 目的

画像ファイルの生成元モデル名を AZViewer 上で管理し、画像検索条件としてモデル名を指定できるようにする。

Stable Diffusion WebUI のメタデータに含まれる `Model: xxx` を読み取り、`image_file_data` に対して生成元モデルを紐づける。

本機能により、以下を実現する。

```text
- 画像ごとの生成元モデル名表示
- edit.vue からのモデル手動設定
- プロンプト読み取り時のモデル自動登録
- モデル名による画像検索
```

## 2. 基本方針

## 2.1 1画像に紐づくモデルは 0 または 1 件

1つの `image_file_data` に紐づく生成元モデルは最大1件とする。

```text
image_file_data : image_model = 1 : 0..1
```

DB上は中間テーブル `image_model_to_file_data` を利用するが、`image_file_data_id` に `UNIQUE` 制約を付け、1画像に複数モデルが紐づく状態を禁止する。

モデル複数持ちを許可すると、検索条件が「いずれか一致」なのか「すべて一致」なのかで仕様が増殖する。今回の用途では不要です。DBに余計な部屋を作ると、だいたい後で変な住人が入ります。

## 2.2 モデル名はタグとは別管理する

モデル名はタグとしては扱わない。

理由は以下。

```text
- モデル名は画像生成環境のメタ情報であり、画像内容タグとは意味が異なる
- タグ検索とは別にモデル検索条件として使いたい
- タグ正規化ルールとは異なる保存ルールにする必要がある
```

## 2.3 モデル名の大文字小文字は維持する

タグとは異なり、モデル名は小文字化しない。

```text
waiNSFWIllustrious_v50
```

のような物理ファイル名・モデル表示名に近い値を扱うため、保存時にケースを変更しない。

## 2.4 保存タイミング

モデル情報の保存タイミングは次の2つとする。

```text
1. edit.vue の「保存して戻る」押下時
2. AppHeader.vue の「プロンプトの読み取り」押下時
```

`edit.vue` のモデル設定ダイアログでは DB 保存しない。
既存のタグ編集と同様、編集フォーム内の一時状態として扱い、「保存して戻る」でまとめて保存する。

## 2.5 起動時の扱い

起動時に `image_model` テーブルが存在しない場合は、テーブル作成のみ行う。

自動で全画像のメタデータ読み取りまでは行わない。

理由は、起動時に全画像のメタデータを読むと、件数次第で起動処理が重くなるため。起動直後から朝礼が長いアプリは嫌われます。

モデル自動登録は、ユーザーが明示的に `AppHeader.vue` の「プロンプトの読み取り」を押下したタイミングで行う。

# 4. DB設計

## 4.1 `image_model`

### 役割

生成元モデル名を管理するマスタテーブル。

### 定義

```sql
CREATE TABLE IF NOT EXISTS image_model (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL UNIQUE
);
```

### カラム

| カラム    | 型       | 内容   |
| ------ | ------- | ---- |
| `id`   | INTEGER | 主キー  |
| `name` | TEXT    | モデル名 |

### 制約

```text
- name は NOT NULL
- name は UNIQUE
```

## 4.2 `image_model_to_file_data`

### 役割

`image_file_data` と `image_model` の関連を管理する。

DB構成上は中間テーブルだが、1画像1モデル制約を持たせる。

### 定義

```sql
CREATE TABLE IF NOT EXISTS image_model_to_file_data (
    image_file_data_id INTEGER NOT NULL,
    image_model_id INTEGER NOT NULL,
    PRIMARY KEY (image_file_data_id, image_model_id),
    UNIQUE (image_file_data_id),
    FOREIGN KEY (image_file_data_id)
        REFERENCES image_file_data(id)
        ON DELETE CASCADE,
    FOREIGN KEY (image_model_id)
        REFERENCES image_model(id)
        ON DELETE CASCADE
);
```

### 重要制約

```sql
UNIQUE (image_file_data_id)
```

これにより、1つの画像に複数モデルが紐づくことを禁止する。

## 4.3 インデックス

検索用に以下を追加する。

```sql
CREATE INDEX IF NOT EXISTS idx_image_model_name
ON image_model(name);

CREATE INDEX IF NOT EXISTS idx_image_model_to_file_data_model_id
ON image_model_to_file_data(image_model_id);
```

# 5. モデル名正規化仕様

## 5.1 保存時正規化

モデル名は以下の処理のみ行う。

```text
1. None は未設定扱い
2. 前後空白を trim
3. 空文字は未設定扱い
4. 512文字を超える場合はエラー
```

## 5.2 行わないこと

以下は行わない。

```text
- 小文字化
- カンマ分割
- 括弧除去
- 重み表記除去
- 全角半角変換
```

タグ正規化とは別物として扱う。

## 5.3 最大長

```text
512文字
```

理由は、タグ最大長を 512 文字へ拡張している既存方針との整合を取りつつ、モデル名としては十分な長さであるため。

# 8. ImageMetadataService仕様

## 8.1 公開メソッド追加

```python
def extract_stable_diffusion_model_name(self, path: str) -> str | None:
    ...
```

## 8.2 役割

```text
1. 画像ファイルを開く
2. 既存の _build_metadata_list を利用してメタ情報行リストを取得する
3. 拡張子に応じて対象キーを探す
4. 対象キーの値を復元する
5. Stable Diffusion WebUI パラメータ内の Model: を抽出する
6. モデル名を返す
```

## 8.3 対象キー

| 拡張子     | 対象キー               |
| ------- | ------------------ |
| `.png`  | `parameters`       |
| `.jpg`  | `exif.UserComment` |
| `.jpeg` | `exif.UserComment` |
| `.webp` | `exif.UserComment` |
| `.avif` | `exif.UserComment` |

既存の `extract_stable_diffusion_prompt()` と同じメタ情報復元処理を使う。

## 8.4 抽出ルール

`Model:` の値を抽出する。

例。

```text
Model: waiNSFWIllustrious_v50
```

抽出結果。

```text
waiNSFWIllustrious_v50
```

## 8.5 誤抽出防止

以下は対象外とする。

```text
Model hash:
ADetailer model:
ADetailer model classes:
```

このため、単純な `if "Model" in item` ではなく、`Model:` キーを判定する。

推奨実装。

```python
import re

_MODEL_NAME_PATTERN = re.compile(
    r"(?:^|,\s*)Model:\s*([^,\n]+)"
)

def _extract_model_name(self, value: str) -> str | None:
    match = _MODEL_NAME_PATTERN.search(value or "")
    if not match:
        return None

    model_name = match.group(1).strip()
    return model_name or None
```

## 8.6 注意

`TI hashes` のように引用符内にカンマを含む項目があるため、最終行を単純に `split(",")` する実装は避ける。

CSVパーサを持ち込むほどでもないですが、雑な split はここでは地雷です。薄く埋まっているタイプの地雷です。

# 9. PromptTagImportService変更

## 9.1 既存責務

既存の `PromptTagImportService` は、タグ未登録画像だけを対象に、メタ情報からプロンプトを抽出し、タグ登録・件数集計・失敗詳細収集を行う構成である。

## 9.2 追加方針

「プロンプトの読み取り」ボタン押下時に、タグ登録とモデル登録の両方を行う。

ただし、対象条件は分ける。

```text
タグ登録対象:
tag_image_link が 1 件もない画像

モデル登録対象:
image_model_to_file_data が 1 件もない画像
```

タグ登録済みでも、モデル未設定ならモデル抽出対象とする。

## 9.3 処理フロー

```text
1. find_items_without_tags() でタグ未登録画像を取得
2. 既存通りタグ登録処理を行う
3. find_items_without_model() でモデル未設定画像を取得
4. 画像ごとに ImageMetadataService.extract_stable_diffusion_model_name(path) を呼ぶ
5. モデル名が取得できなければ modelSkippedCount 加算
6. モデル名が取得できたら repository.replace_image_model(id, model_name) を実行
7. 成功したら modelLinkedCount 加算
8. 例外発生時は modelFailedCount 加算し、失敗詳細を最大20件まで保持する
9. タグ結果とモデル結果をまとめて返す
```

## 9.4 トランザクション方針

一括全体を1トランザクションにはしない。

```text
- タグ登録は1画像単位
- モデル登録も1画像単位
- 1画像で失敗しても全体処理は継続
```

既存のプロンプトタグ登録と同じ方針に揃える。

## 10.6 `fetch_models_for_search` 追加

```python
def fetch_models_for_search(self, payload):
    try:
        data = payload or {}
        keyword = str(data.get("keyword") or "").strip()
        limit = self._normalize_model_search_limit(data.get("limit"))

        repository = self._get_repository()

        models = repository.find_models_for_search(
            keyword=keyword,
            limit=limit,
        )
        total_count = repository.count_models_for_search(
            keyword=keyword,
        )

        return ApiResponse(
            success=True,
            message="",
            data={
                "models": [model.to_dict() for model in models],
                "total_count": total_count,
                "limit": limit,
            },
        ).to_dict()

    except Exception:
        return ApiResponse(
            success=False,
            message="モデル一覧を取得できませんでした。",
            data={
                "models": [],
                "total_count": 0,
                "limit": 256,
            },
        ).to_dict()
```

## 10.7 `_normalize_model_search_limit` 追加

```python
def _normalize_model_search_limit(self, value) -> int:
    try:
        limit = int(value or 256)
    except (TypeError, ValueError):
        limit = 256

    return max(1, min(limit, 256))
```

async function searchModelsForSearch({ keyword = '' } = {}) {

# 25. 注意点

## 25.1 起動時自動抽出は行わない

今回の仕様では、`image_model` テーブルが存在しない場合でも自動抽出は行わない。

```text
起動時:
- テーブル作成のみ

ユーザー操作:
- 「プロンプトの読み取り」でモデル抽出実行
```

ここを自動化すると、初回起動が重くなる可能性がある。あと、ユーザーが「起動しただけなのに何か始まった」と感じる挙動は地味に嫌われます。

## 25.2 モデル名はタグ化しない

モデル名をタグにも追加しない。

タグ検索とモデル検索の責務を分ける。

## 25.3 中間テーブルだが実質1対1

`image_model_to_file_data` は中間テーブルだが、`UNIQUE (image_file_data_id)` により、1画像1モデルを保証する。

ここを忘れると、仕様書では1モデルなのにDBでは複数入るという、二重帳簿みたいな状態になります。

## 25.4 `Model:` 抽出は雑に split しない

`TI hashes` など、カンマを含む複雑な項目があるため、単純なカンマ分割は避ける。

`Model:` キーを正規表現で抽出する。
