# 実装設計書

## 画像選択・一括物理削除機能追加

## 1. 目的

画像一覧画面で複数画像を選択し、選択した画像について以下を一括削除できるようにする。

```text
- AZViewer 上の画像レコード
- サムネイルキャッシュ
- 実際の画像ファイル
```

本機能により、AZViewer を単なる画像ビューワーではなく、画像管理ツールとして利用できるようにする。

ただし、物理削除は取り消しできない破壊的操作であるため、既存の単体レコード削除とは明確に区別する。

## 2. 前提

既存の `delete_image_file` は、画像レコードとサムネイル削除を担当する API であり、元画像ファイルの物理削除までは担当していない。

また、既存の `ThumbnailCacheService` には複数サムネイル削除用の `delete_thumbnails(record_ids)` が存在するため、本機能でも再利用する。

`Content.vue` は画像一覧、ソート、ページサイズ、総件数、ページング、タイル表示を担当しているため、画像選択状態の管理・中継対象として自然である。

## 3. 変更概要

以下の機能を追加する。

```text
- tile.vue のタイトル左側に選択チェックボックスを追加
- 選択状態を UI 一時状態として保持
- AppHeader.vue に「選択画像を削除」ボタンを追加
- 選択件数 0 件では削除ボタンを disabled
- 削除確認ダイアログを追加
- 選択画像の実ファイル、サムネイル、DB レコードを一括削除
- 削除後に現在検索条件で再検索
- 再検索・ページ変更・ソート変更・ページサイズ変更時は選択状態をクリア
```

## 5. 基本方針

## 5.1 既存単体削除とは分離する

既存の単体削除 API は維持する。

```text
delete_image_file
```

これは従来通り、AZViewer 上の登録削除を主目的とする。

本機能では、実ファイルの物理削除を含む新 API を追加する。

```text
delete_image_files_with_physical_files
```

### 理由

物理削除を既存 `delete_image_file` に混ぜると、既存操作の意味が変わる。

```text
従来:
アプリ上から削除するが、実ファイルは残る

今回:
アプリ上から削除し、実ファイルも削除する
```

この2つは操作リスクが大きく異なるため、API・UI・確認文言のすべてで分離する。

削除ボタンは名前で殴るくらいが安全です。短くて曖昧な削除 API は、だいたい後で人間を削ります。

## 5.2 選択状態は DB に保存しない

画像選択状態は、現在表示中の一覧に対する一時的な UI 状態とする。

```text
DB カラムは追加しない
image_file_data に選択状態を持たせない
```

### 理由

選択状態は永続データではない。

```text
- 現在の画面で選択されているか
- 一括削除対象として一時的に選ばれているか
```

だけを表すため、DB 保存すると不要な状態管理が増える。

## 5.3 見えていない選択は保持しない

以下の操作時、選択状態をクリアする。

```text
- 再検索
- ページ変更
- ページサイズ変更
- ソート変更
- 削除完了後
```

### 理由

画面に見えていない画像が選択されたまま残ると、ユーザーが削除対象を把握できない。

```text
見えていないけど選択済み
```

は、物理削除 UI として危険すぎる。地雷に「地雷」と書いてあるだけ、まだ親切です。

## 8.10 実装例

```js
import { deleteImageFilesWithPhysicalFiles } from '../services/backendApi'

async function deleteSelectedImages({ ids, refresh, clearSelection }) {
  const targetIds = [...(ids ?? [])]

  if (targetIds.length === 0) {
    return
  }

  const confirmed = window.confirm(
    [
      `選択した ${targetIds.length} 件の画像を削除します。`,
      '',
      'この操作では、AZViewer 上の登録情報だけでなく、実際の画像ファイルもディスクから削除されます。',
      'この操作は元に戻せません。',
      '',
      '削除しますか？',
    ].join('\n')
  )

  if (!confirmed) {
    return
  }

  loading.showLoading({
    title: '選択画像を削除中',
    message: '実ファイル、サムネイル、登録情報を削除しています...',
  })

  try {
    const result = await deleteImageFilesWithPhysicalFiles({
      ids: targetIds,
    })

    if (!result?.success) {
      pushToast({
        type: 'danger',
        message: result?.message || '選択画像の削除に失敗しました。',
      })
      return
    }

    const data = result.data ?? {}
    const failedCount = data.failedCount ?? 0

    if (failedCount > 0) {
      pushToast({
        type: 'warning',
        message: `一部画像の削除に失敗しました。対象: ${data.targetCount ?? targetIds.length} 件 / 削除: ${data.deletedRecordCount ?? 0} 件 / 失敗: ${failedCount} 件`,
      })
    } else {
      pushToast({
        type: 'success',
        message: `選択画像の削除が完了しました。対象: ${data.targetCount ?? targetIds.length} 件 / 削除: ${data.deletedRecordCount ?? 0} 件`,
      })
    }

    clearSelection?.()
    await refresh?.()
  } catch (error) {
    pushToast({
      type: 'danger',
      message: '選択画像の削除に失敗しました。',
    })
  } finally {
    loading.hideLoading()
  }
}
```

```python
    deleted_file_count: int
    deleted_thumbnail_count: int
    deleted_record_count: int
    missing_file_count: int
    failed_count: int
```

## 13. 物理削除処理設計

## 13.1 削除順序

削除順序は以下とする。

```text
1. id 配列を正規化する
2. DB から対象レコードを取得する
3. 1件ずつ実ファイル削除を試行する
4. 実ファイル削除に成功、または実ファイルが存在しない ID を DB 削除候補に入れる
5. DB 削除候補のサムネイルを削除する
6. DB 削除候補のレコードを削除する
7. 集計結果を返す
```

## 13.2 ファイル削除と DB 削除の関係

```text
ファイル削除成功
→ サムネイル削除
→ DB レコード削除

ファイルが既に存在しない
→ サムネイル削除
→ DB レコード削除

ファイル削除失敗
→ サムネイル削除しない
→ DB レコード削除しない
→ failedFiles に追加
```

### 理由

ファイル削除に失敗したのに DB レコードを削除すると、実ファイルは残っているが AZViewer 上から見えなくなる。

これは「管理ツール」として中途半端で、ユーザーから見るとファイルが忍者になります。存在するのに見えない。困るタイプの忍者です。

## 13.3 実ファイルが存在しない場合

実ファイルが存在しない場合は、失敗扱いにしない。

```text
missingFileCount += 1
DB レコード削除対象に含める
サムネイル削除対象に含める
```

### 理由

起動時クリーンアップでも、DB 上に存在するが実ファイルが存在しないレコードは削除する設計である。

一括削除時も同じ整合性回復として扱う。

## 13.4 サムネイル削除失敗

サムネイル削除に失敗しても、DB レコード削除は継続する。

### 理由

サムネイルは派生キャッシュであり、元画像や DB レコードより重要度が低い。

```text
サムネイル削除失敗
→ ログ出力
→ 可能であれば failedFiles には含めない
→ DB レコード削除は継続
```

ただし、削除件数には反映する。

```text
deletedThumbnailCount
```

は実際に削除できた件数を入れる。

## 13.5 ファイル削除方法

Python 側では `Path.unlink()` を使用する。

```python
from pathlib import Path

image_path = Path(item.path)

if image_path.exists():
    image_path.unlink()
else:
    missing_file_count += 1
```

```python
if image_path.exists() and not image_path.is_file():
```

## 14. ImageCatalogApi 実装方針

## 14.1 正規化メソッド

```python
def _normalize_delete_ids(self, value) -> list[int]:
    if not isinstance(value, list):
        return []

    ids: list[int] = []

    for item in value:
        try:
            record_id = int(item)
        except (TypeError, ValueError):
            continue

        if record_id <= 0:
            continue

        if record_id in ids:
            continue

        ids.append(record_id)

    return ids
```

## 14.3 delete_image_files_with_physical_files

実装イメージ。

```python
def delete_image_files_with_physical_files(self, payload):
    try:
        data = payload or {}
        record_ids = self._normalize_delete_ids(data.get("ids"))

        if not record_ids:
            return ApiResponse(
                success=False,
                message="削除対象が指定されていません。",
                data=self._empty_physical_delete_data(),
            ).to_dict()

        repository = self._get_repository()
        items = repository.find_by_ids(record_ids)

        result = self._delete_physical_files_and_records(
            repository=repository,
            items=items,
        )

        message = (
            "一部画像の削除に失敗しました。"
            if result.failed_count > 0
            else "選択画像の削除が完了しました。"
        )

        return ApiResponse(
            success=True,
            message=message,
            data=result.to_api_data(),
        ).to_dict()

    except Exception:
        return ApiResponse(
            success=False,
            message="選択画像の削除に失敗しました。",
            data=self._empty_physical_delete_data(),
        ).to_dict()
```

## 14.4 内部削除処理

```python
def _delete_physical_files_and_records(
    self,
    repository: ImageFileRepository,
    items: list[ImageFileListItem],
) -> PhysicalDeleteResult:
    deletable_ids: list[int] = []
    failed_files: list[PhysicalDeleteFailure] = []

    deleted_file_count = 0
    missing_file_count = 0

    for item in items:
        try:
            image_path = Path(item.path)

            if image_path.exists():
                if not image_path.is_file():
                    raise ValueError("画像ファイルではありません。")

                image_path.unlink()
                deleted_file_count += 1
            else:
                missing_file_count += 1

            deletable_ids.append(item.id)

        except Exception as error:
            if len(failed_files) < 20:
                failed_files.append(
                    PhysicalDeleteFailure(
                        id=item.id,
                        path=item.path,
                        reason="ファイルを削除できませんでした。",
                    )
                )

    deleted_thumbnail_count = 0

    if deletable_ids:
        deleted_thumbnail_count = self._thumbnail_service.delete_thumbnails(
            deletable_ids
        )
        deleted_record_count = repository.delete_by_ids(deletable_ids)
    else:
        deleted_record_count = 0

    return PhysicalDeleteResult(
        target_count=len(items),
        deleted_file_count=deleted_file_count,
        deleted_thumbnail_count=deleted_thumbnail_count,
        deleted_record_count=deleted_record_count,
        missing_file_count=missing_file_count,
        failed_count=len(items) - len(deletable_ids),
        failed_files=failed_files,
    )
```

### 注意

既存の `ThumbnailCacheService.delete_thumbnails(record_ids)` が戻り値を返さない実装の場合は、削除件数を返すように変更するか、`deletedThumbnailCount` を省略してもよい。

推奨は、件数集計のために戻り値を返す形へ変更する。

```python
def delete_thumbnails(self, record_ids: list[int]) -> int:
    deleted_count = 0
    for record_id in record_ids:
        if self.delete_thumbnail(record_id):
            deleted_count += 1
    return deleted_count
```

その場合、`delete_thumbnail(record_id)` も bool を返すようにする。

## 21. 注意点

## 21.1 物理削除は戻せない

本機能はファイルをディスクから削除する。

OS のゴミ箱へ移動する仕様ではない。

```text
Path.unlink()
```

を使う場合、通常は即時削除になる。

必要になった場合は将来仕様として「ゴミ箱へ移動」を検討する。

## 21.2 DB とファイル削除は完全な原子性を持たない

DB トランザクションで戻せるのは DB 操作のみである。

```text
ファイル削除成功
DB 削除失敗
```

という状態は理論上発生する。

このリスクを完全には消せないため、処理順序・エラーハンドリング・ログで軽減する。

## 21.3 単体削除の意味を変えない

既存の `delete_image_file` に物理削除を混ぜないこと。

単体削除と一括物理削除の意味が混ざると、UI 上も API 上も事故りやすい。

## 21.4 削除対象は必ず ID から再取得する

フロントエンドから `path` を直接渡して削除しない。

```text
フロント: id のみ送信
バックエンド: id から path を取得
```

とする。

理由は、削除対象を DB 上の登録レコードに限定するため。

## 21.5 確認文言は弱くしない

確認文言では必ず以下を明示する。

```text
- 登録情報だけではない
- 実ファイルも削除される
- 元に戻せない
```

削除確認で遠慮すると、だいたい削除後にユーザーが遠い目になります。

## 5. 基本方針

## 5.1 選択状態は既存仕様を流用する

本機能のために新しい DB カラムは追加しない。

```text
image_file_data に選択状態を持たせない
```

選択状態は既存の `selectedImageIds` / `selectedImageIdSet` をそのまま利用する。

## 5.2 物理ファイル移動と DB 更新は成功分のみ反映する

ファイル移動に失敗した画像については、DB を更新しない。

```text
ファイル移動成功
→ DB path / folder 更新対象

ファイル移動失敗
→ DB 更新しない
→ failedFiles に追加
```

## 5.3 同名ファイルは上書きしない

移動先に同名ファイルが既に存在する場合、その画像は移動失敗扱いにする。

```text
old: C:/old/sample.png
new: D:/new/sample.png

D:/new/sample.png が存在する場合
→ 移動しない
```

自動リネームは行わない。

```text
sample_1.png
sample_2.png
```

のようなリネームは便利だが、今回の仕様には含めない。親切が勝手にファイル名を変え始めると、だいたい後で事情聴取になります。

## 5.4 DB 更新失敗時は補償移動を試みる

実ファイルを移動した後に DB 更新へ失敗した場合、可能であればファイルを元の場所へ戻す。

```text
old_path -> new_path 移動成功
DB 更新失敗
new_path -> old_path へ戻す
```

戻しにも失敗した場合は、失敗詳細へ明示する。

## 5.5 サムネイルキャッシュは削除・再生成しない

サムネイルキャッシュが `id` ベースで `data/thumbs/{id}.png` として管理されている場合、画像ファイルの移動ではサムネイルを変更しない。

理由は、画像内容と `id` が変わらないため。

```js
import {
  moveImageFilesToFolder,
  selectFolderDialog,
} from '../services/backendApi'

function buildMoveConfirmMessage(count, destinationFolder) {
  return [
    `選択した ${count} 件の画像ファイルを移動します。`,
    '',
    '移動先:',
    destinationFolder,
    '',
    '移動後は AZViewer 上の登録パスも更新されます。',
    '実行しますか？',
  ].join('\n')
}

function extractSelectedFolderPath(result) {
  const items = result?.data?.items ?? []
  const folderItem = items.find((item) => item?.type === 'directory')
  return folderItem?.path || ''
}

function pushMoveResultToast(pushToast, result, targetCount) {
  const data = result?.data ?? {}
  const failedCount = data.failedCount ?? 0
  const movedCount = data.movedCount ?? 0
  const skippedCount = data.skippedCount ?? 0

  if (failedCount > 0) {
    pushToast({
      type: 'warning',
      message: `一部画像ファイルの移動に失敗しました。対象: ${data.targetCount ?? targetCount} 件 / 移動: ${movedCount} 件 / スキップ: ${skippedCount} 件 / 失敗: ${failedCount} 件`,
    })
    return
  }

  pushToast({
    type: 'success',
    message: `画像ファイルの移動が完了しました。対象: ${data.targetCount ?? targetCount} 件 / 移動: ${movedCount} 件 / スキップ: ${skippedCount} 件`,
  })
}

async function moveSelectedImages({ ids, refresh, clearSelection }) {
  const targetIds = [...(ids ?? [])]

  if (targetIds.length === 0) {
    return
  }

  const folderResult = await selectFolderDialog()

  if (!folderResult?.success) {
    return
  }

  const destinationFolder = extractSelectedFolderPath(folderResult)

  if (!destinationFolder) {
    return
  }

  const confirmed = window.confirm(
    buildMoveConfirmMessage(targetIds.length, destinationFolder)
  )

  if (!confirmed) {
    return
  }

  loading.showLoading({
    title: '画像ファイルを移動中',
    message: '実ファイルの移動と登録情報の更新を行っています...',
  })

  try {
    const result = await moveImageFilesToFolder({
      ids: targetIds,
      destinationFolder,
    })

    if (!result?.success) {
      pushToast({
        type: 'danger',
        message: result?.message || '画像ファイルの移動に失敗しました。',
      })
      return
    }

    pushMoveResultToast(pushToast, result, targetIds.length)

    clearSelection?.()
    await refresh?.()
  } catch (error) {
    pushToast({
      type: 'danger',
      message: '画像ファイルの移動に失敗しました。',
    })
  } finally {
    loading.hideLoading()
  }
}
```

## 12. バックエンド処理設計

## 12.1 処理順序

```text
1. payload から ids / destinationFolder を取得する
2. ids を正規化する
3. destinationFolder を絶対パス化して検証する
4. DB から対象レコードを取得する
5. 各画像について移動前検証を行う
6. 移動可能な画像だけ実ファイルを移動する
7. 移動成功分の path / folder を DB 更新する
8. DB 更新失敗時は可能な範囲で元パスへ戻す
9. 集計 DTO を返す
```

## 12.2 ID 正規化

一括削除の `_normalize_delete_ids` と同等の処理を追加する。

```python
def _normalize_move_ids(self, value) -> list[int]:
    if not isinstance(value, list):
        return []

    ids: list[int] = []

    for item in value:
        try:
            record_id = int(item)
        except (TypeError, ValueError):
            continue

        if record_id <= 0:
            continue

        if record_id in ids:
            continue

        ids.append(record_id)

    return ids
```

共通化するなら `_normalize_record_ids` としてもよい。

```python
def _normalize_destination_folder(self, value) -> Path:
    folder_text = str(value or "").strip()

    if not folder_text:
        raise ValueError("移動先フォルダが指定されていません。")

    folder_path = Path(folder_text).expanduser().resolve()

    if not folder_path.exists():
        raise ValueError("移動先フォルダが存在しません。")

    if not folder_path.is_dir():
        raise ValueError("移動先がフォルダではありません。")

    return folder_path
```

## 13. ImageCatalogApi 実装方針

## 13.1 move_image_files_to_folder

```python
def move_image_files_to_folder(self, payload):
    try:
        data = payload or {}

        record_ids = self._normalize_move_ids(data.get("ids"))
        if not record_ids:
            return ApiResponse(
                success=False,
                message="移動対象が指定されていません。",
                data=self._empty_file_move_data(),
            ).to_dict()

        try:
            destination_folder = self._normalize_destination_folder(
                data.get("destinationFolder")
            )
        except ValueError as error:
            return ApiResponse(
                success=False,
                message=str(error),
                data=self._empty_file_move_data(),
            ).to_dict()

        repository = self._get_repository()
        items = repository.find_by_ids(record_ids)

        result = self._move_files_and_update_records(
            repository=repository,
            items=items,
            destination_folder=destination_folder,
        )

        message = (
            "一部画像ファイルの移動に失敗しました。"
            if result.failed_count > 0
            else "画像ファイルの移動が完了しました。"
        )

        return ApiResponse(
            success=True,
            message=message,
            data=result.to_api_data(),
        ).to_dict()

    except Exception:
        return ApiResponse(
            success=False,
            message="画像ファイルの移動に失敗しました。",
            data=self._empty_file_move_data(),
        ).to_dict()
```

## 13.2 内部移動処理

```python
def _move_files_and_update_records(
    self,
    repository: ImageFileRepository,
    items: list[ImageFileListItem],
    destination_folder: Path,
) -> FileMoveResult:
    moved_items: list[dict] = []
    skipped_count = 0
    failed_files: list[FileMoveFailure] = []

    for item in items:
        try:
            source_path = Path(item.path)

            if not source_path.exists():
                raise FileNotFoundError("移動元ファイルが存在しません。")

            if not source_path.is_file():
                raise ValueError("画像ファイルではありません。")

            destination_path = destination_folder / source_path.name

            if source_path.resolve() == destination_path.resolve():
                skipped_count += 1
                continue

            if destination_path.exists():
                raise FileExistsError("移動先に同名ファイルが既に存在します。")

            if repository.exists_by_path(str(destination_path)):
                raise ValueError("移動後パスは既に登録済みです。")

            shutil.move(str(source_path), str(destination_path))

            moved_items.append({
                "id": item.id,
                "old_path": str(source_path),
                "new_path": str(destination_path),
                "folder": destination_path.parent.name,
            })

        except Exception as error:
            if len(failed_files) < 20:
                failed_files.append(
                    FileMoveFailure(
                        id=item.id,
                        path=item.path,
                        reason=self._to_file_move_failure_reason(error),
                    )
                )

    if moved_items:
        try:
            updates = [
                {
                    "id": item["id"],
                    "path": item["new_path"],
                    "folder": item["folder"],
                }
                for item in moved_items
            ]
            repository.update_paths(updates)

        except Exception:
            self._restore_moved_files(moved_items, failed_files)
            moved_items = []

    return FileMoveResult(
        target_count=len(items),
        moved_count=len(moved_items),
        skipped_count=skipped_count,
        failed_count=len(failed_files),
        failed_files=failed_files,
    )
```

## 13.3 失敗理由変換

```python
def _to_file_move_failure_reason(self, error: Exception) -> str:
    if isinstance(error, FileNotFoundError):
        return "移動元ファイルが存在しません。"

    if isinstance(error, FileExistsError):
        return "移動先に同名ファイルが既に存在します。"

    if isinstance(error, PermissionError):
        return "ファイルへアクセスできません。別のアプリで使用中の可能性があります。"

    if isinstance(error, ValueError):
        return str(error)

    return "ファイルを移動できませんでした。"
```

## 13.4 DB 更新失敗時の復元

```python
def _restore_moved_files(
    self,
    moved_items: list[dict],
    failed_files: list[FileMoveFailure],
) -> None:
    for item in moved_items:
        try:
            new_path = Path(item["new_path"])
            old_path = Path(item["old_path"])

            if new_path.exists() and not old_path.exists():
                shutil.move(str(new_path), str(old_path))

            if len(failed_files) < 20:
                failed_files.append(
                    FileMoveFailure(
                        id=item["id"],
                        path=item["old_path"],
                        reason="DB 更新に失敗したため、ファイル移動を取り消しました。",
                    )
                )

        except Exception:
            if len(failed_files) < 20:
                failed_files.append(
                    FileMoveFailure(
                        id=item["id"],
                        path=item["new_path"],
                        reason="DB 更新に失敗し、移動済みファイルの復元にも失敗しました。",
                    )
                )
```

このケースはかなり嫌な状態なので、ログ出力も推奨。

```text
移動 API 実行中は LoadingOverlay が表示されること
移動完了時、失敗時のどちらでも LoadingOverlay が解除されること
フォルダ選択ダイアログ表示中は LoadingOverlay を表示しないこと
```

## 17.4 ファイル移動

```text
選択画像の実ファイルが指定フォルダへ移動されること
移動先に同名ファイルがある場合は上書きしないこと
自動リネームしないこと
移動元ファイルが存在しない場合は失敗扱いになること
ファイルロック等で移動できない場合は失敗扱いになること
```

## 17.5 DB 更新

```text
移動成功分のみ image_file_data.path が移動後パスへ更新されること
移動成功分のみ image_file_data.folder が移動後の直接親フォルダ名へ更新されること
移動失敗分の DB は更新されないこと
同一フォルダへの移動は skippedCount として扱われること
```

## 17.6 再検索・選択状態

```text
移動完了後に選択状態がクリアされること
移動完了後に現在検索条件で再検索されること
フォルダ検索条件に一致しなくなった画像は一覧から消えること
```

## 17.7 API 結果

```text
targetCount / movedCount / skippedCount / failedCount が返ること
failedFiles は最大20件まで返ること
一部失敗時も成功分は反映されること
```

## 18. 実装上の注意点

ファイル移動は DB 更新より厄介です。DB はトランザクションで戻せますが、ファイルシステムは「まあ頑張って戻してね」という温度感です。

特に注意すべき点は以下です。

```text
- 同名ファイル上書き禁止
- DB 更新失敗時の補償移動
- ファイルロック時の失敗扱い
- 移動先が現在検索条件から外れる可能性
- サムネイルキャッシュは id ベースなので維持
```

この設計なら、既存の一括物理削除機能の UI・状態管理を流用しつつ、物理ファイル移動と DB 整合性のリスクをある程度抑えられます。
