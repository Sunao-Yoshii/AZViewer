# pywebview + Vue + Bootstrap アプリケーション基盤

# 基本設計書

## SQLite 連携・画像ファイル登録機能

---

## 1. 目的

本設計書は、pywebview + Vue + Bootstrap により構成されるデスクトップアプリケーション基盤に対し、以下の追加機能を実現するための基本設計を定義する。

* SQLite を用いた永続化
* アプリケーション全体で共有する DB 接続管理
* 起動時の DB 初期化および整合性確認
* サイドバー上部へのファイル登録 UI 配置
* Python 側ネイティブダイアログによるファイル／フォルダ選択
* 画像ファイル情報の一括登録
* エラー発生時のロールバックおよびモーダル表示

---

## 3. コンポーネント分割方針

本機能の実装にあたっては、実ファイル配置やディレクトリ構成を本設計書では固定しない。
これは、既存のアプリケーション基盤側で定義されている土台構成と不整合を起こさないようにするためである。

その代わり、本機能は**役割ごとに明確に分離されたコンポーネント／モジュールとして設計する**ことを基本方針とする。

### 3.1 分割方針

以下の責務単位で分離する。

* DB 接続管理
* DB 初期化
* 起動時クリーンアップ
* ファイル走査
* 画像ファイル登録
* ネイティブダイアログ制御
* サイドバー上部 UI
* エラーモーダル UI
* API 呼び出し境界

### 3.2 分離の目的

* 責務の混在を防ぐ
* 土台側構成へ柔軟に組み込みやすくする
* 将来的な機能追加時に影響範囲を局所化する
* UI と業務ロジック、DB アクセスを分離する

### 3.3 設計原則

* Vue 側は表示とユーザー操作を担当する
* Python 側はファイル処理、DB 処理、トランザクション管理を担当する
* DB アクセスは接続管理責務を通じて行う
* 登録処理はファイル選択とフォルダ選択で共通化する
* 各機能は再利用可能な責務単位で分解する
* 実配置は土台側の既存規約に従って決定する

## 4. DB 設計

## 4.1 DB ファイル

* DB ファイルパス: `./data/az_data.sqlite3`
* 基準パス: 実行ディレクトリ
* `./data` ディレクトリが存在しない場合は起動時に生成する

---

## 4.2 テーブル定義

### テーブル名

`image_file_data`

### カラム一覧

| カラム名        | 型       | NOT NULL | 主キー | 備考                            |
| ----------- | ------- | -------: | --: | ----------------------------- |
| id          | INTEGER |        ○ |   ○ | AUTOINCREMENT                 |
| filename    | TEXT    |        ○ |     | ファイル名                         |
| path        | TEXT    |        ○ |     | 絶対パス、一意                       |
| folder      | TEXT    |        ○ |     | 直近フォルダ名                       |
| rating      | TEXT    |        ○ |     | General / R-15 / R-18 / R-18G |
| is_checked  | INTEGER |        ○ |     | 0 / 1                         |
| is_favorite | INTEGER |        ○ |     | 0 / 1                         |
| comment     | TEXT    |          |     | 改行可                           |

### 制約

* `PRIMARY KEY AUTOINCREMENT (id)`
* `UNIQUE (path)`
* `CHECK (rating IN ('General', 'R-15', 'R-18', 'R-18G'))`
* `CHECK (is_checked IN (0, 1))`
* `CHECK (is_favorite IN (0, 1))`

### CREATE TABLE 想定

```sql
CREATE TABLE IF NOT EXISTS image_file_data (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    filename TEXT NOT NULL,
    path TEXT NOT NULL UNIQUE,
    folder TEXT NOT NULL,
    rating TEXT NOT NULL CHECK (rating IN ('General', 'R-15', 'R-18', 'R-18G')),
    is_checked INTEGER NOT NULL CHECK (is_checked IN (0, 1)),
    is_favorite INTEGER NOT NULL CHECK (is_favorite IN (0, 1)),
    comment TEXT NULL
);
```

## 5. 接続管理設計

## 5.1 接続管理方針

* アプリケーション全体で **ConnectionManager** を 1 つ保持する
* ConnectionManager は SQLite 接続インスタンスを内部保持する
* 必要に応じて同一接続を返す
* DB アクセスは Repository または Service から直接 `sqlite3.connect()` せず、必ず ConnectionManager 経由で行う

---

## 5.2 ConnectionManager の責務

* DB ファイルパスの解決
* SQLite 接続生成
* 接続取得
* 接続クローズ
* 二重接続生成防止

---

## 5.3 想定インターフェース

```python
class ConnectionManager:
    def initialize(self) -> None
    def get_connection(self)
    def close(self) -> None
```

---

## 5.4 設計上の注意

* 接続生成は起動時に 1 回のみ行う
* 終了時に必ず close する
* 将来の拡張を考慮し、接続設定は ConnectionManager に閉じ込める
* 他クラスが DB パスや接続生成方法を知らない構造にする

ここを破ると、後で `repository_a.py` と `repository_b.py` がそれぞれ勝手に接続を開き始め、
設計が「共有管理」から「野生の SQLite」になります。

### 2. `import_selected_items(payload)`

#### 概要

選択されたファイル／フォルダ情報を受け取り、登録処理を実行する。

#### 入力

```json
{
  "items": [
    {
      "path": "C:/images/sample.png",
      "type": "file"
    },
    {
      "path": "C:/images/set01",
      "type": "directory"
    }
  ]
}
```

#### 処理内容

* 入力項目検証
* ファイル／フォルダ振り分け
* フォルダ直下ファイル抽出
* 対象拡張子判定
* 重複除外
* DB トランザクション実行
* 成功／失敗結果返却

#### 正常時戻り値例

```json
{
  "success": true,
  "data": null,
  "message": ""
}
```

#### 失敗時戻り値例

```json
{
  "success": false,
  "data": {
    "errorSummary": "データ登録中にエラーが発生しました。",
    "failedFiles": [
      "C:/images/a.png",
      "C:/images/b.png",
      "C:/images/set01/c.png"
    ]
  },
  "message": "UNIQUE constraint failed: image_file_data.path"
}
```

## 11. Repository 設計

## 11.1 `ImageFileRepository`

### 責務

* `image_file_data` への CRUD のうち必要分を提供
* SQL 実行
* DB 結果の DTO 変換

### 想定メソッド

```python
class ImageFileRepository:
    def create_table(self) -> None
    def find_all_paths(self) -> list[str]
    def delete_by_paths(self, paths: list[str]) -> None
```

### 補足

## 13. ファイル登録処理設計

## 13.1 処理フロー

1. Vue がファイル／フォルダ選択操作を受け取る
2. 選択された項目一覧を Python API に渡す
3. Python 側で項目を解析する
4. ファイルはそのまま候補化
5. フォルダは直下ファイルのみ列挙
6. 許可拡張子のみ抽出
7. 絶対パスへ正規化
8. `filename`, `folder` 抽出
9. DB 既存 `path` と重複照合
10. 新規候補のみトランザクションで一括 INSERT
11. エラー時はロールバック
12. 戻り値を Vue に返却
13. 失敗時は ErrorModal 表示

---

## 13.2 登録値固定ルール

* `rating = "General"`
* `is_checked = 0`
* `is_favorite = 0`
* `comment = NULL`

---

## 13.3 エラー時の表示方針

* 1件でもエラーが発生した場合、そのトランザクションは全体失敗とする
* モーダルには **トランザクション対象ファイルを全件「登録失敗」** として表示する
* エラー内容は **代表エラー 1 件** を表示する

### 例

* 代表エラー: `データ登録中にエラーが発生しました`
* 詳細: `UNIQUE constraint failed: image_file_data.path`
* 失敗ファイル一覧: 対象全件

## 14. ネイティブダイアログ設計

## 14.1 利用目的

Python 側ネイティブダイアログでファイル／フォルダ選択を可能とする。

---

## 14.2 ダイアログ種別

### ファイル選択

* 複数選択可
* 対応拡張子のみ対象

### フォルダ選択

* 単一または複数選択は実装仕様に合わせる
* 選択フォルダ直下ファイルのみ対象

---

## 14.3 位置づけ

* ファイル／フォルダ選択の入口
* 登録処理本体は共通化し、ファイル選択とフォルダ選択で分岐させない

ここは大事です。
**選択方法が2つあるだけで、登録ロジックは1つ** にするべきです。
入口ごとに別実装を始めると、半年後に片方だけバグ修正されて、もう片方が地雷原になります。

### DB

* 起動時に DB ファイルが生成される

* 接続マネージャ経由で同一接続を再利用できる

### ダイアログ

* ネイティブダイアログから選択したファイル／フォルダでも同一登録処理が行われる

# 補足: 次に詳細化すべき点

基本設計としては十分ですが、実装前に次を詰めるとスムーズです。

## 1. API 名と戻り値 JSON の正式確定

今の案でほぼよいですが、命名を確定した方が実装がぶれません。

## 2. pywebview のネイティブダイアログ具体API

ここは使うメソッドに合わせて設計を少し寄せる必要があります。

## 3. エラーモーダル文言

ユーザー向け文言と詳細文言を分けるかを決めると UI 実装が楽です。

## 4. ファイル／フォルダ選択時のパス取得方式

Python ダイアログで取得できる絶対パスを前提に、選択導線をどう見せるかを決める必要があります。

必要なら次に、これをそのまま開発単位へ落とした
**詳細設計書** か **実装タスク分解書** にします。

# 補足

後に、ドラッグ＆ドロップすると、ファイル/フォルダの絶対パスが取得できない(ブラウザセキュリティ上の制限と思われる)ことから、ダイアログからの選択のみという形に変更。
